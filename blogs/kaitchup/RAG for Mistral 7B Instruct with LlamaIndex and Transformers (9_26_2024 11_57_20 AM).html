While techniques like LoRA and QLoRA have dramatically reduced the cost of fine-tuning LLMs, regularly fine-tuning again and again adapters to keep the LLM’s knowledge up-to-date is impractical. In the extreme case where the knowledge base is continuously updated, keeping the LLM’s knowledge up-to-date through fine-tuning is even impossible.

We could include all knowledge, unseen during fine-tuning, in the context of the LLM and instruct the LLM to exploit it when appropriate. Even though recent techniques such as LongLoRA, FlashAttention, or LongRoPE improve the efficiency and accuracy of LLMs in dealing with very long contexts, e.g., more than 2 million tokens with LongRoPE, handling long prompts is costly and significantly slows down inference.

Instead, retrieval augmented generation (RAG) retrieves relevant information from a database, for a given query, and only injects this information into the LLM’s context. It keeps the context relatively short and significantly helps the LLM focus on what is relevant.

In this article, I explain the basics of RAG. Then, I show how to set up a simple RAG system using Mistral 7B instruct as a base model. For this purpose, I use LlamaIndex and Hugging Face Transformers. Thanks to GPTQ quantization, you can reproduce my setup on consumer hardware.

How Does RAG Work?

Let’s start by testing the knowledge limit of Mistral 7B. I use the version instruct v0.2 of the model which was released on December 12th, 2023.

I will query the model about Mixtral-8x7B that (I assume) it can’t possibly know.

My prompt:

The answer is quite good but Mistral 7B can only give general advice. It has no information about what is Mixtral-8x7B but at least it could guess that it is a model.

With RAG, we can give the necessary context to Mistral 7B to know what is Mixtra-8x7B and potentially answer the question. By the end of this article, we will have a Mistral 7B knowing what is Mixtral and how to run it.

RAG is very simple. It works by injecting potentially relevant information, given the user’s query, along with additional instructions into the prompt. The main challenge is to retrieve this relevant information. As for any information retrieval tasks, we need:

  • A knowledge base (or database) covering topics/domains for which we want the model to be up-to-date.

  • A retriever able to retrieve relevant information, given a prompt (or query), from the knowledge base.

The retriever works by taking the user's prompt and searching for semantically close pieces, or chunks, of text in the knowledge base. To compute the semantic similarity, the retriever uses text embeddings.

With text embeddings, we can calculate a similarity score between the user's question/query and each chunk of text in the knowledge base. This semantic score is used to rank the chunks. The chunks closest to the query will have a higher rank.

For instance, we can configure the retriever to get the top k most relevant chunks of text from the knowledge base. Then, these k chunks will be injected into the prompt to enrich the LLM’s knowledge. k must be small to avoid injecting too many chunks, that may not be all relevant, which would significantly increase the latency of the LLM.

As for the knowledge base, it must be prepared by following these steps:

  • Collect relevant documents for the topics and domains that you want the RAG system to cover.

  • Chunk the documents. We don’t want RAG to retrieve entire documents and inject them into our prompts. It would be way too many tokens. Instead, we only retrieve chunks of documents. The size of the chunks is one of the most important hyperparameters of RAG. If you set it too small, RAG may only retrieve chunks that are semantically close to the user’s prompt but useless to answering it. For instance, with chunks of the size of a regular sentence, text embeddings will find that the sentence “What is QLoRA?” is very close semantically to “What is LoRA?” which does not answer our question. By setting a chunk size large enough, we make sure that the retriever will retrieve entire paragraphs and not just sentences that are close to the user’s prompt.

  • Encode the chunks with the text embeddings.

  • Index the encoded chunks into the vector database.

We will see how to perform all these steps in the next section.

After retrieving the relevant chunks, RAG automatically decodes the chunk embeddings into tokens. Then, it remains to inject these new tokens into the prompt and instruct the LLM to use them.

Indexing Documents in a Vector Database for RAG

Note: The full code running the RAG system is in the notebook.

Setting up a RAG system requires gathering documents on topics we want the LLM to know more about. For instance, if I want an LLM to cover recent techniques for fine-tuning and quantizing LLMs, then indexing Kaitchup’s articles into the database would be a good start.

But first, we need to set up the database.

I use LlamaIndex (MIT license) for setting up RAG systems. LlamaIndex is a powerful open framework that can do many things for building LLM applications. It is also very convenient to use with Hugging Face Transformers.

You can install it with pip, along with all the other libraries that we will need:

pip install --upgrade llama-index llama-index-embeddings-huggingface transformers auto-gptq accelerate optimum

Note: I installed auto-gptq and optimum since the LLM that I’ll use is the GPTQ quantized version of Mistral 7B instruct. I also always set “upgrade“ on Google Colab as the default transformers version already installed by Colab is often slightly outdated.

Then, import the following:

from llama_index.embeddings.huggingface import HuggingFaceEmbedding
from llama_index.core.retrievers import VectorIndexRetriever
from llama_index.core.query_engine import RetrieverQueryEngine
from llama_index.core.postprocessor import SimilarityPostprocessor
from llama_index.core import Settings, SimpleDirectoryReader, VectorStoreIndex

That’s all we need for setting up the database, indexer, and retriever.

The next step is to decide which text embedding model we should use. There are many available on the Hugging Face hub. We need a good model but also a model that is fast enough and not too big. I use BAAI/bge-large-en-v1.5, which is large (355M parameters) but fits, along with Mistral 7B, on an 8 GB GPU. This model is also available in “base” and “small” versions if you want a faster and more memory-efficient retriever.

Now, we can tell to LlamaIndex that we will use this model:

Settings.embed_model = HuggingFaceEmbedding(model_name="BAAI/bge-large-en-v1.5")

With “HuggingFaceEmbedding”, we can import any embedding model from Hugging Face.

Other important settings for LlamaIndex:

Settings.llm = None #We won't use LlamaIndex's LLM capabilities but Hugging Face Transformers
Settings.chunk_size = 128
Settings.chunk_overlap = 10

As discussed in the previous section, the chunk size is an important hyperparameter. I set it to “128”. Considering that Mistral 7B performs well with a context of up to 8k tokens, we can potentially retrieve and inject in the context up to 64 small chunks of documents. However, we have also to consider the length of the instructions, the chat history with the user, and the size of the LLM’s response. Longer contexts also take more time to be processed resulting in a higher latency. Note: Technically, Mistral 7B can handle up to 32k tokens in its context but its performance is decreasing dramatically for context longer than 8k. I discussed the main reasons for this performance decrease in my article about LongRope.

If we use a good embedding model, we won’t need to retrieve many chunks. We can trust the text embedding model that it will retrieve the most relevant chunks given the user’s query. In this application, my RAG system will retrieve the 6 most relevant chunks:

top_k = 6

The next step is to load the documents that we want to index.

For this tutorial, I have collected 5 articles from The Kaitchup, in PDF format, published after Mistral 7B’s release to be sure that Mistral 7B never saw them during pre-training. You can get them here:

wget https://about.benjaminmarie.com/data/kaitchup_samples/samples.zip
unzip samples.zip

This will create a “samples” directory containing a few PDF files. Then, we can load and parse the text from these PDF files with:

documents = SimpleDirectoryReader("samples").load_data()

“samples” is the path to the directory containing the PDF files. The documents are ready to be indexed:

index = VectorStoreIndex.from_documents(documents)

It should be fast since we only index a few documents.

It remains to set up the retriever and the query engine:

retriever = VectorIndexRetriever(
    index=index,
    similarity_top_k=top_k,
)

query_engine = RetrieverQueryEngine(
    retriever=retriever,
    node_postprocessors=[SimilarityPostprocessor(similarity_cutoff=0.5)],
)

There are many arguments that you can pass to these objects to adjust them for your particular needs. They are all well-documented in LlamaIndex’s documentation.

For the retriever, I use a similarity cut-off of 0.5 which is rather loose. Note: Use a higher cut-off, e.g., 0.75, if you want to make sure that the retriever will only retrieve very relevant chunks. If nothing is retrieved with this higher cut-off, we can indicate it to the LLM, in its instructions, so that in turn it can tell the user that it doesn’t have the knowledge to answer the query.

Our database and retriever are now all set!

We can test it with a query, for instance, the question I tried in the previous section:

How can I run Mixtral-8x7B on my 24 GB GPU?

To retrieve the top k closest chunks from this database with LlamaIndex, we simply call the query_engine and pass it our query:

query = "How can I run Mixtral-8x7B on my 24 GB GPU?"
response = query_engine.query(query)
context = ""
for i in range(top_k):
    context = context + response.source_nodes[i].text + "\n\n"

Context Extension with RAG

Now, we need to give the retrieved chunks to our LLM.

Let’s first load our LLM, Mistral 7B instruct. I use a version quantized with GPTQ to save memory. If you want, you can use a better model such as my Mayonnaise which is a merge of several very good fine-tuned and aligned Mistral 7B models:

  • kaitchup/Mayonnaise-4in1-02-gptq-4bit

from transformers import AutoTokenizer
import transformers
import torch

model = "TheBloke/Mistral-7B-Instruct-v0.2-GPTQ"
tokenizer = AutoTokenizer.from_pretrained(model)
pipeline = transformers.pipeline(
    "text-generation",
    model=model,
    torch_dtype="auto",
    device_map="cuda",
)

As we saw in the previous section, Mistral 7B doesn’t really know what is “Mixtral-8x7B” and was only able to give general advice. To query the model with my question, I passed my prompt to Mistral 7B’s chat template as follows:

messages = [{"role": "user", "content": query}]
prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=False)
outputs = pipeline(prompt, max_new_tokens=512, do_sample=True, temperature=0.7, top_k=50, top_p=0.95)
print(outputs[0]["generated_text"])

Note: I have already shown the entire output in the previous section.

We can give Mistral 7B the context retrieved from our database, i.e., the 6 chunks, by injecting it into the model’s instructions. Since Mistral 7B instruct doesn’t use a “system” role (only “assistant” and “user” are available), I use a custom chat template from this repository to add “system”:

  • chujiezheng/chat_templates

Clone this repository:

git clone https://github.com/chujiezheng/chat_templates.git

And then override the tokenizer’s chat template:

chat_template = open('./chat_templates/chat_templates/mistral-instruct.jinja').read()
chat_template = chat_template.replace('    ', '').replace('\n', '')
tokenizer.chat_template = chat_template

Note: Adding this “system” role is an optional step. I do it because I think it is better and cleaner to separate the system and the assistant. The assistant’s messages are fully generated by the model while the system’s messages are already written or completed by RAG.

We can now add a system “message” including instructions and our context (in bold) retrieved from the database:

messages = [{"role": "system", "content": "You are a helpful, respectful and honest assistant. Always answer as helpfully as possible, while being safe. Your answers should not include any harmful, unethical, racist, sexist, toxic, dangerous, or illegal content. Please ensure that your responses are socially unbiased and positive in nature.\n\nIf a question does not make any sense, or is not factually coherent, explain why instead of answering something not correct. If you don't know the answer to a question, please don't share false information.\n\nContext:\n"+context+"\n\n"},
 {"role": "user", "content": query}]
prompt = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=False)
outputs = pipeline(prompt, max_new_tokens=512, do_sample=True, temperature=0.7, top_k=50, top_p=0.95)

Based on the information provided in the article, it seems that Mixtral-8x7B, with its 46.7B parameters, exceeds the memory capacity of a 24 GB GPU. Even when quantized to 2-bit, the model still requires more memory than what's available.

To run Mixtral-8x7B on your 24 GB GPU, you could consider some optimization techniques mentioned in the article, such as:

1. Decreasing the `per_device_train_batch_size` and increasing `gradient_accumulation_steps` proportionally.
2. Using expert offloading to distribute the workload across multiple GPUs if available.
3. Considering fine-tuning with AQLM instead of QLoRA for better memory efficiency and performance.

However, keep in mind that these optimization techniques might result in slower fine-tuning progress or reduced model performance. You may need to experiment with different configurations to find the best trade-off between memory usage and performance.

If you're still unable to run Mixtral-8x7B on your 24 GB GPU using these methods, you might need to consider upgrading your hardware or looking for alternative models that fit within your GPU's memory constraints.

Mistral 7B clearly exploits the context retrieved with RAG. The answer is more informative but not entirely correct.

It mentions techniques that would work to run Mixtral on consumer hardware such as expert offloading and AQLM. That’s good. Nonetheless, it also suggests doing expert offloading across multiple GPUs while the user mentions in the query only one GPU. More problematic, it says that 2-bit quantization won’t be enough. That’s never written in the context and is wrong. Retrieving larger and cleaner chunks would probably alleviate this kind of issue.

Moreover, it mentions an “article” from which it took this information but the user can’t know what it is referring to since the article was retrieved by the system itself and never shown to the user. This is confusing.

Conclusion

Running a low-cost RAG system with a 7B parameter model is simple with LlamaIndex and a quantized LLM. Even when using a large text embedding model, the entire system never consumed more than 8 GB of GPU RAM. Moreover, despite the size of the context, the latency of the system remains low.

In this article, we have only built a basic RAG system. We can improve it in many ways, for instance:

  • Cleaning the data before indexing them

  • Optimize the chunk size and the other RAG’s hyperparameters

  • Searching for a better text embedding model for our specific domain

  • Fine-tuning the model to make it better at exploiting the context retrieved

We will tackle them in the following articles of this series. If you want me to tackle other specific aspects of RAG, leave a comment and I’ll see what I can do.
