Apple released OpenELM, a family of small open LLMs with sizes ranging from 270M to 3B parameters. With this release, Apple aims at providing LLMs that can run on devices with tiny memory.

The memory footprint of the OpenELM LLMs is small and they have a low computational cost for inference. They can also be fully fine-tuned on consumer hardware. Nonetheless, chat LLMs with less than 1 billion parameters are rarely released as they are usually not good enough to be useful for any applications.

In this article, I review the OpenELM LLMs. I first review the technical report published by Apple describing the models. Then, I show how to fine-tune the smallest OpenELM with ORPO to make a tiny chat model. We will see whether we can turn the smallest OpenELM into a capable chat model.

OpenELM: Why are They So Small?

Apple shared a technical report describing OpenELM here:

OpenELM: An Efficient Language Model Family with Open-source Training and Inference Framework

It describes the model architecture, and its training, and reports on several benchmarks evaluating the efficiency and accuracy of the models.

The Neural Architecture of OpenELM

In the very large majority of transformer LLMs, the allocation of the parameters is uniform. All the layers have the same sizes, with the same number of attention heads and feedforward dimensions.

In OpenELM LLMs, the allocation of the parameters is non-uniform. Different layers have different numbers of KV and Query heads and feedforward dimensions. Let’s have a look at the architecture of OpenELM 270M:

  • OpenELM-270M’s config.json

The model has 16 layers. In the config.json, we can see some original fields such as “num_kv_heads”, “num_query_heads”, and “ffn_multipliers”, adjusting the parameters for each layer. For instance, for num_query_heads, we have:

It means that the first layers will have 12 query heads while the last (deepest) layers have 20 query heads.

According to Apple, this non-uniform allocation of the parameters can train better transformers with less data. This idea is named “layer-wise scaling” and was first proposed by Mehta et al. (2020) but has rarely been applied to LLMs.

OpenELM’s Training

They pre-trained the models on a compilation of publicly available datasets:

They ran 350k training steps with a batch size of 4M tokens, yielding a total of 1.4T tokens used for pre-training. For reference, Llama 2 was trained on 2T tokens, TinyLlama on 3T tokens, Gemma on 6T tokens, and Llama 3 on 15T tokens. For small models such as OpenELM, 1.4T tokens appear to be a lot.

Training a Tiny Instruct LLM, OpenELM 270M, with ORPO

Popular methods to train instruct LLMs, a.k.a., chat models, are RLHF and DPO. However, these methods involve multiple datasets and models. For instance, DPO, a simpler alternative to RLHF, still requires two steps: supervised fine-tuning (SFT) on an instruction dataset followed by preference optimization on another dataset.

ORPO is yet another alternative for preference optimization which doesn’t require the SFT step. I reviewed ORPO and showed how to use it in this article:

ORPO: Preference Optimization without the Supervised Fine-tuning (SFT) Step

Due to its simplicity, I chose this method to train OpenELM 270M as a chat model. OpenELM 270M is a very tiny model. Chat models of this size are known to poorly perform, especially for complex tasks but they can be good enough for simple conversations.

Since the model is small, we can fully train it on consumer hardware without using parameter-efficient fine-tuning (PEFT) methods such as LoRA and QLoRA. Note: In the notebook, you will see that I put the code for LoRA fine-tuning. However, it probably doesn’t work as expected. It consumes as much memory as full fine-tuning. I assume that this is due to the peculiar architecture of OpenELM that the PEFT library might not fully support.

Here is the code for ORPO training:

import torch, multiprocessing
from datasets import load_dataset
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig,
    TrainingArguments,
)
from trl import ORPOTrainer, ORPOConfig

model_name = "apple/OpenELM-270M"
tokenizer_name = "meta-llama/Llama-2-7b-hf"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(tokenizer_name, add_eos_token=True, use_fast=True)
tokenizer.pad_token = tokenizer.eos_token
tokenizer.padding_side = 'left'

dataset = load_dataset("HuggingFaceH4/ultrafeedback_binarized", split=["train_prefs","test_prefs"])

def process(row):
    row["chosen"] = tokenizer.apply_chat_template(row["chosen"], tokenize=False)
    row["rejected"] = tokenizer.apply_chat_template(row["rejected"], tokenize=False)
    return row

dataset[0] = dataset[0].map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

dataset[1] = dataset[1].map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

model = AutoModelForCausalLM.from_pretrained(
          model_name, torch_dtype=torch.bfloat16, device_map={"": 0}, trust_remote_code=True
)
model.gradient_checkpointing_enable()

orpo_config = ORPOConfig(
    output_dir="./results_openelm270M_ORPO/",
    evaluation_strategy="steps",
    do_eval=True,
    optim="adamw_8bit",
    per_device_train_batch_size=8,
    gradient_accumulation_steps=2,
    per_device_eval_batch_size=8,
    log_level="debug",
    logging_steps=100,
    learning_rate=8e-6,
    eval_steps=100,
    save_steps=100,
    save_strategy='epoch',
    num_train_epochs=2,
    warmup_ratio=0.1,
    lr_scheduler_type="linear",
    beta=0.1, #beta is ORPO's lambda
    max_length=1024,
)

trainer = ORPOTrainer(
        model=model,
        train_dataset=dataset[0],
        eval_dataset=dataset[1],
        args=orpo_config,
        tokenizer=tokenizer,
)

trainer.train()

2 training epochs require 18 GB of GPU RAM and 11 hours. Note: I use the L4 instance of Google Colab.

Let’s look at the learning curves:

The training loss and the validation loss are decreasing. The model is learning but reaches a minimum around 3,000 training steps. However, we also want the model to learn to distinguish good (“chosen”) from bad (“rejected”) answers. If we compare their log probabilities, we see that they are both increasing, reaching a maximum around 3,000 training steps, while the difference between both remains the same:

At least, we observe that the “rejected” answers have a lower probability than the “chosen” answer. In other words, the model could be a good classifier to determine in a pair of answers which one is more acceptable to humans.

I applied the learning rate and Beta used by the authors of ORPO. I recommend searching for better ones since the values I used don’t seem to yield good results.

Note: Inference with this model only requires 0.8 GB of GPU RAM.

import torch
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer
)

model_name = "kaitchup/OpenELM-270M-oasstguanaco-2e-ORPO"
tokenizer_name = "meta-llama/Llama-2-7b-hf"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(tokenizer_name, use_fast=True)

model = AutoModelForCausalLM.from_pretrained(
          model_name, torch_dtype=torch.bfloat16, device_map={"": 0}, trust_remote_code=True
)

chat = [[{"role": "user", "content": "How is vanilla cultivated?"}],
  [{"role": "user", "content": "How much money do I have if I have one dollar?"}],
  [{"role": "user", "content": "Where is Berlin?"}],
  [{"role": "user", "content": "Give me a list of 5 European countries."}],
  [{"role": "user", "content": "What is AI?"}],
  [{"role": "user", "content": "What can you do right? Exactly?"}]
]

for c in chat:
  p = tokenizer.apply_chat_template(c, tokenize=False)
  inputs = tokenizer(p, return_tensors="pt").to("cuda")
  outputs = model.generate(**inputs, do_sample=True, pad_token_id=tokenizer.eos_token_id, top_p=0.9, max_new_tokens=150)
  result = tokenizer.decode(outputs[0], skip_special_tokens=True)

  print(result)

Conclusion

OpenELM is a family of open LLMs by Apple with a particular architecture exploiting layer-wise scaling. According to Apple, it makes the LLMs capable of learning more from less data.

270 million is a very small number of parameters for a chat model, too small to be useful in most situations. To get better results, I suggest to try with larger models. The next size is also probably too small but the 1.1B model should yield somewhat useful results. I expect it to be at least as good as TinyLlama which also has 1.1B parameters.

For tiny models, finding good hyperparameters is even more critical. And since they are small models, searching for these hyperparameters can be fast and cheap. Trying several values for the ORPO’s beta, learning rate, and warmup may yield significantly better chat models.
