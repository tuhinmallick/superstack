Fine-tuning large language models requires a huge amount of GPU memory. We need a GPU with enough RAM to load the model and store the optimizer states. If we take a 7 billion parameter model, e.g., a Mistral 7B, we need 14 GB of GPU memory to load the model, assuming that the model is loaded with 16-bit parameters. Moreover, for each parameter, the standard AdamW optimizer creates and stores 2 new 32-bit parameters, i.e., we need an additional 56 GB of memory. Thats already a total of 70 GB!

This is without counting the additional copies of the model made at various stages of fine-tuning and the models activations whose memory consumption is also significant but difficult to estimate as it depends on hyperparameters. One A100/H100 80 GB GPU wouldnt be enough to fully fine-tune a 7B model.

Storing the optimizer states is by far the most expensive part. Even if we quantize the optimizers parameters to 8-bit, they would still require 14 GB of memory.

GaLore can significantly reduce this memory consumption to a point where it becomes possible to perform full fine-tuning of a 7B model on a 24 GB consumer GPU. Moreover, while parameter-efficient fine-tuning (PEFT) methods such as LoRA often dont perform as well as full fine-tuning, GaLore performs comparably to full fine-tuning.

Memory-efficient Pre-training and Full Fine-tuning with Gradient Low-rank Projection

GaLore is presented in this paper:

GaLore: Memory-Efficient LLM Training by Gradient Low-Rank Projection

The paper first demonstrates that, when training deep learning models such as LLMs, the gradients have an inherent low-rank structure.

Gradient

A gradient is like an arrow pointing in the direction where a model's error or mistake increases the fastest. When training a model, we want to make its errors as small as possible. To do this, we look at the gradient to see which way not to goand then we move our model's adjustments, i.e., the weight updates, in the opposite direction to reduce its mistakes.

Using projection matrices, GaLore leverages this low-rank structure to project the full-precision (float32) gradients into a low-rank subspace. This step significantly reduces the dimensionality of the gradients. In practice, it can reduce the memory consumption of the optimizer by more than 80%.

GaLore can be combined with other techniques to further reduce memory consumption, such as 8-bit quantization.

The optimizer, for instance, AdamW, usually updates the weights for all layers simultaneously after completing backpropagation, by keeping all the weight gradients in memory. To minimize memory consumption during training, GaLore can also update weights layer by layer as backpropagation occurs. It increases the computational cost as we have to split the updates but for deep models with many layers, the reduction of memory consumption is substantial.

GaLore introduces additional hyperparameters:

  • Rank r: As for LoRA, a higher rank consumes more memory but may also improve the models performance. As shown in the figure above, a rank of 1,024 yields significantly better results than 512. Note also that the GaLores rank would typically be higher than what you would use for LoRA. A rank of 128 would be a minimum it seems.

  • Scale factor : Another one that we also find in LoRA adjusting the intensity of the low-rank update. According to the authors, GaLore is very robust to change of . Values between 0.2 and 2.0 seem to work well.

  • Frequency of subspace change T: Although the ideal frequency for T varies based on the training length and the task's complexity, setting T within a range of 50 to 1000 has minimal impact on results.

GaLore Full Fine-tuning for Mistral 7B with a 24 GB GPU

GaLore is already available and implemented as an optimizer in Hugging Face Transformers.

To fine-tune Mistral 7B with GaLore, I installed:

pip install -q -U bitsandbytes
pip install git+https://github.com/jiaweizzhao/GaLore
pip install git+https://github.com/huggingface/transformers
pip install -q -U accelerate
pip install -q -U datasets
pip install -q -U trl

I installed transformers from source since GaLore is very recent and still regularly updated. We also need bitsandbytes for the 8-bit quantization of the optimizer states. FlashAttention (flash_attn) is not mandatory but I recommend using it for faster fine-tuning if your GPU is compatible.

Then, import all the necessary for fine-tuning:

import torch
from datasets import load_dataset
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    TrainingArguments,
)
from trl import SFTTrainer

Nothing special here. Actually, there isnt any difference with the code that we would write for a standard full fine-tuning, except for the training arguments:

major_version, minor_version = torch.cuda.get_device_capability()
if major_version >= 8:
  torch_dtype = torch.bfloat16
  attn_implementation='flash_attention_2'
  print("Your GPU is compatible with FlashAttention and bfloat16.")
else:
  torch_dtype = torch.float16
  attn_implementation='eager'
  print("Your GPU is not compatible with FlashAttention and bfloat16.")


model_name = "mistralai/Mistral-7B-v0.1"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_name, add_eos_token=True, use_fast=True)
tokenizer.pad_token = tokenizer.eos_token
tokenizer.padding_side = 'left' #Necessary for FlashAttention compatibility

ds = load_dataset("timdettmers/openassistant-guanaco")

model = AutoModelForCausalLM.from_pretrained(
          model_name, device_map={"": 0}, attn_implementation=attn_implementation, torch_dtype=torch_dtype
)
model.gradient_checkpointing_enable()

training_arguments = TrainingArguments(
        output_dir="./galore_adamw_8bit_layerwise_r512_1e-5_wl/",
        evaluation_strategy="steps",
        do_eval=True,
        per_device_train_batch_size=8,
        per_device_eval_batch_size=8,
        log_level="debug",
        optim="galore_adamw_8bit_layerwise",
        optim_args="rank=512, update_proj_gap=200, scale=1.8",
        optim_target_modules=[r".*attn.*", r".*mlp.*"],
        save_strategy = 'epoch',
        logging_steps=50,
        learning_rate=1e-5,
        eval_steps=50,
        fp16= not torch.cuda.is_bf16_supported(),
        bf16= torch.cuda.is_bf16_supported(),
        num_train_epochs=3,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
)

For the optimizer optim, we have several galore options for AdamW:

  • galore_adamw: This is standard GaLore with float32 parameters, still consuming too much memory for consumer hardware.

  • galore_adamw_8bit: This version quantizes the optimizers parameters to 8-bit. For fine-tuning Mistral 7B, this configuration requires 35 GB of GPU RAM, or around 30 GB if you use a lower rank of 128.

  • galore_adamw_8bit_layerwise: Same as galore_adamw_8bit but with layerwise updates. This configuration consumes 22.5 GB. It runs on a 24 GB consumer GPU.

As for the hyperparameters optim_args, I use a rank of 512. 1,024 would yield better results according to the authors of GaLore for a 7B models but it wouldnt fit on 24 GB of GPU RAM without reducing the batch size. Note: The current implementation of GaLore doesnt support gradient accumulation steps.

I set update_proj_gap to 200. I didnt try any other values; the authors mentioned that any values between 50 and 1,000 would work fine. I set alpha to 1.8, but other values between 0.25 and 2 seem to work as well, confirming that GaLore is quite robust to hyperparameter changes.

Finally, similarly to LoRA, we must target modules where we will apply GaLore. This is set with optim_target_modules. We want to target most modules to reduce memory consumption as much as possible. For a model like Mistral 7B, it means targeting all the self-attention and MLP modules. However, GaLore is only able to target PyTorchs nn.Linear layers. In the case of Mistral 7B, we can match the compatible layers with regex by setting [r".*attn.*", r".*mlp.*"]. For all models using the Llama 2 architecture, you can set the same.

We can start training with GaLore by running:

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

The training will take several minutes to start. This is the time required to project the gradients. Training with GaLore stalls regularly to update this projection. If you see that nothing is happening, dont stop the training, GaLore is doing some updates.

However, it seems to only affect the layerwise implementation. If we use the same configuration but replace galore_adamw_8bit_layerwise with galore_adamw_8bit, the training is much better:

The downside is that now GaLore consumes 35 GB of GPU memory. It is still a significant reduction of the memory consumption but we are far from being able to run it on consumer hardware.

Ill update this article once the layerwise implementation is fixed.

Conclusion

GaLore is a significant advance in parameter-efficient fine-tuning. Thanks to a projection of the gradients into low-rank subspaces, it can reduce memory consumption by 80% while preserving the models performance. Fully fine-tuning 7B models is now possible on consumer hardware.

GaLore can also be used to pre-train LLMs. Pre-training LLMs on consumer hardware is getting easier but it would still require many weeks to obtain a 7B model trained from scratch.
