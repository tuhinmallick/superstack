Llama 3 70B is currently one of the best LLMs. According to public leaderboards such as Chatbot Arena, Llama 3 70B is better than GPT-3.5 and some versions of GPT-4. However, with its 70 billion parameters, this is a very large model. Inference with Llama 3 70B consumes a lot of GPU RAM. For fast inference on GPUs, we would need 2x80 GB GPUs. This is far from an affordable configuration but we can significantly reduce these requirements.

In this article, I show how to quantize Llama 3 70B with mixed precision using ExLlamaV2 and a 24 GB GPU. ExLlamaV2’s quantization method preserves the important weights from quantization while aggressively quantizing the remaining weights. I quantized Llama 3 70B with 4, 3.5, 3, 2.5, and 2.18 bits per weight, on average, and benchmarked the resulting models. We will see that quantization below 2.5 bits per weight makes the model small enough to run on a 24 GB GPU.

Llama 3 70B Requirements

To estimate Llama 3 70B GPU requirements, we have to get its number of parameters. You can get this information from the model card of the model. Llama 3 70B has 70.6 billion parameters. The parameters are bfloat16, i.e., each parameter occupies 2 bytes of memory. It means that Llama 3 70B requires a GPU with 70.6 billion * 2 bytes: 141.2 GB of VRAM. It’s much more than the 24 GB of a consumer GPU.

With quantization, we can reduce the size of the model so that it can fit on a GPU. Using 4-bit quantization, we divide the size of the model by nearly 4. It would still require a costly 40 GB GPU. With 2-bit quantization, Llama 3 70B could fit on a 24 GB consumer GPU but with such a low-precision quantization, the accuracy of the model could drop.

Since I wrote this article, ExLlamaV2 has been improved. It is now possible to quantize a 70B model, such as Llama 3 70B, with a consumer GPU. For instance, this is the memory consumption reported by Google Colab (L4 instance) when I ran the “measurement” step (see next section) for Llama 3 70B:

During the quantization of the model, the memory consumption is slightly lower. However, you will need a lot of hard drive space as ExLlamaV2 will make a copy of the model and create the quantized version. I recommend having at least 2.5 times the size of Llama 3 70B, i.e., around 325 GB, available on your hard drive.

Measuring the Quantization Error for Llama 3 70B with ExLlamaV2

The measurement step of Llama 3 70B consists of trying different quantization configurations for each module of each layer of the model. Llama 3 70B has 80 layers which contain attention and MLP modules. The attention modules are k_proj, q_proj, v_proj, and o_proj. The MLP modules are up_proj, gate_proj, and down_proj.

In total, that’s 7 modules per layer, i.e., 7*80=560 modules to quantize for the entire model. Technically, there are 561 modules since ExLlamaV2 also quantizes the language modeling head. Note: The quantization logs also mention a quantization of the RMSNorm module but no quantization is effectively applied there.

In the measurement file, we can find the quantization errors for each module. Then, when we quantize the model with a target average “bits per weight” (bpw), ExLlamaV2 will read the measurement file and pick the quantization precision (i.e., “bits per weight“) for each module which minimizes the quantization error while achieving the target bpw.

These measurements are also interesting to read to know which layers of the model are the most difficult to quantize. For these layers, if a low bpw is applied, the perplexity of the model would significantly worsen.

At the beginning of the quantization, ExLlamaV2 reports in the logs the bpw applied for each layer. I used the data to observe which layers are the most difficult to quantize:

The deepest layers are the ones that are the most preserved from quantization with a higher bpw.

I found that ExLlamaV2 can’t achieve a target bpw of 2.0 but quantize all the layers with the same bpw of 2.18 which seems to be the minimum for Llama 3 70B:

I also found that ExLlamaV2 always quantizes Llama 3 70B with a higher bpw for the attention modules than for the MLP.

Quantizing Llama 3 70B with ExLlamaV2

Note: The quantization of a 70B model by ExLlamaV2 only takes a few hours and is feasible with a 24 GB GPU. What you will need the most is space on your hard drive. To be comfortable, I recommend 400 GB. If you want to do this with Google Colab, the L4 instance connected to a Google Drive with 400 GB available will work. Still, you may have to resume quantization several times as Google applies data transfer limits on Google Drive that are reinitialized daily.

Quantizing with ExLlamaV2 is simpler than the first time I wrote about it. We don’t need to download a parquet data file for calibration. ExLlamaV2 now uses a default dataset for the calibration step.

For quantization and evaluation of the quantized model, I installed the following:

pip install exllamav2 datasets
git clone https://github.com/turboderp/exllamav2.git

Next, we need to download the model to quantize since ExLlamaV2 can only quantize a locally available model.

from huggingface_hub import snapshot_download
snapshot_download(repo_id="meta-llama/Meta-Llama-3-70B", ignore_patterns=["*.bin", "*.pth"], local_dir="./Llama-3-70B/", local_dir_use_symlinks=False)

Then, we run the measurement step which tries different quantization configurations for each layer, and logs the quantization error. This step takes several hours.

mkdir ./Llama-3-70B/temp/
python exllamav2/convert.py \
    -i ./Llama-3-70B/ \
    -o ./Llama-3-70B/temp/ \
    -nr \
    -om ./Llama-3-70B/measurement.json

/!\ “./Llama-3-70B/temp/” will contain large temporary files. The directory's content is entirely deleted at the beginning of the process. If you want to resume an interrupted measurement, don’t set “-nr”. The content of the temp directory will be kept.

The quantization errors are stored in the file “./Llama-3-70B/measurement.json”

I released it here:

  • Llama 3 70B’s measurement.json

You can use this file to quantize Llama 3 70B with the bpw of your choice, as follows:

!python exllamav2/convert.py \
    -i ./Llama-3-70B/ \
    -o ./Llama-3-70B/temp/ \
    -nr
    -m  ./Llama-3-70B/measurement.json\
    -cf ./Llama-3-70B/4.0bpw/ \
    -b 4.0

The argument “-m” expects the measurement file produced in the previous step. “-cf” is the directory that will contain the quantized model. “-b” is our target bpw.

In this example, the model will be quantized with an average precision of 4 bits (4 bpw). I also quantized Llama 3 70B with 3.5, 3.0, 2.5, and 2.0 bpw. I released all the models here:

  • Hugging Face collection: Llama 3 70B ExLlamaV2 (bpw: 4.0, 3.5, 3.0, 2.5, and 2.18)

I benchmarked the perplexity on openassistant-guanaco’s test split, inference speed, and memory consumption of each model with the following command:

wget -O oasstguanaco-test.parquet https://huggingface.co/datasets/timdettmers/openassistant-guanaco/resolve/refs%2Fconvert%2Fparquet/default/test/0000.parquet?download=true

!python exllamav2/test_inference.py -m ./Llama-3-70B/2.5bpw/ --eval_dataset oasstguanaco-test.parquet --eval_rows 518 --eval_length 256 -p "Once upon a time," --tokens 512

The results:

For now, let’s ignore the results for 2.0 bpw. A decrease of 0.5 bpw approximately equals a reduction of the model size of 4 GB. The perplexity on the test split of openassistant-guanaco remains acceptable even for 2.5 bpw. In theory, it should be possible to fit this version on a 24 GB GPU without batch decoding and other optimizations. The decoding speed, 18.6 tokens per second, is also good enough for many applications.

However, with ExLlamaV2’s inference script, I could only run the 2.0 bpw version on 24 GB of GPU RAM. As we saw in the previous section, this is not really 2.0 bpw but more like 2.18 bpw, hence the size of the model didn’t decrease that much compared to the 2.5 bpw.

With a perplexity above 80, this 2.0 bpw version is mostly useless. The minimal bpw for reasonably good results should be between 2.3 and 2.5.

Conclusion

Llama 3 70B can yield reasonably good results with quantization at 2.5 bpw. While 2.5-bit quantization makes the model small enough to be loaded on a consumer GPU (24 GB), inference itself will require slightly more memory. With a decoding speed nearing 19 tokens/second, ExLlamaV2 also makes the quantized model very fast for inference. However, if you target a higher precision of 4.0 bpw, I’d rather recommend using GPTQ and Marlin which would make a model of similar quality and faster.
