Proximal Policy Optimization (PPO) for RLHF

Our goal is to improve the model fine-tuned at step 1 so that the generated responses are more aligned with humans: less toxic, less biased, etc.

We evaluate the responses generated by the model fine-tuned in Step 1 with a reward model trained on rankings of the responses. But we also don’t want to trust the reward model blindly otherwise the model trained with RLHF would find a way to maximize the reward by generating gibberish. So, when we update the model parameters, we want to make sure that we stay within the neighborhood of the original model (trained in Step 1) for the current batch of training examples. This is the role of the Proximal Policy Optimization.

In practice, a penalty is applied using the KL divergence between the probability distributions of the responses generated by the RLHF model and the original model trained at step 1. If the difference between the probability distributions is larger, the penalty applied by PPO will be stronger.

The parameters of the model are updated given this penalty and the scalar reward of the response generated.

There is also a value model, often initialized by the reward model, whose role is to evaluate the difference between the predicted states, generated by the RLHF model, and the expected states. This model is updated during training and returns another scalar used to calibrate the updates of the RLHF model.

Note that we won’t update the model fine-tuned at step 1, since we need it fixed to ensure that we don’t go too far from it. We will optimize a copy of this model, which is the RLHF model, as follows:

  1. Make a copy (A’) of the model (A) trained in Step 1. A is frozen.

  2. Make a copy (V) of the reward model (W) trained in Step 2. V is the value model.

  3. Given a prompt (p), A’ generates a response (r’) (we don’t generate text but only the probabilities).

  4. Feed r’ to A to compute its relative probabilities.

  5. W (the reward model) only scores the response r’ (i.e., the response of the model that we are fine-tuning). We want to maximize this score. W is frozen.

  6. But at the same time, we don’t want to diverge too much from the original model A. We apply PPO. PPO’s role is to keep us within the neighborhood of A, for the current training batch.

  7. V only scores the states of the generated responses.

  8. Given the penalties (computed with KL divergence), and the scores given by the reward model and the value model, the parameters of A’ are updated.

  9. Iterate from 3 to 8.

Here is the process illustrated:

To get an even better model, we could stop iterating at some point and decide to replace A with A’ for the next iterations. It would require retraining a reward model on responses generated by A’ and ranked by humans. Then, we would optimize a new model A’’, a copy of A’, with this new reward model. With enough time and budget, we could iterate like this several times to continue improving the model.

In practice, given the cost of training instruct LLMs, this is rarely done for more than one iteration.

From 3 to 8, we use 4 different models:

  • The original model fine-tuned with SFT (A)

  • The reward model (W)

  • The value model (V)

  • The current model that we are training with RLHF (A’)

Loading all these 4 models in memory would be challenging on consumer hardware. DeepSpeed Chat implements several functionalities to avoid it.

About the RLHF terminology

The models loaded and fine-tuned during RLHF have different names in the litterature. The value model is often called critic model while the RLHF model is often called actor model. DeepSpeed Chat used this actor-critic terminology.

Requirements and DeepSpeed Chat optimizations for RLHF

The model fine-tuned in Step 1 and the model trained with RLHF have the same size. It can be a very large model. OpenAI used a model with 175 billion parameters. In contrast, I used OPT-1.3B which has only 1.3 billion parameters. But still, that’s a big model. We need 2.6 GB of memory to load it, twice, since we also make a copy of it.

We also need to load the reward model and make a copy of it to initialize the value model. I used OPT-350M which only requires around 750 MB of memory.

In total, we would need around 7 GB (A’ + A + V + W) of memory just to load all the models. Or 14 GB, if you don’t use safetensors.

On consumer hardware, using an RTX 4060 16 GB or an RTX 3090/4090 24 GB only leaves a little space in memory for fine-tuning with large batches.

DeepSpeed Chat proposes to offload the original model (A) to the CPU to save some VRAM. The CPU is slower than the GPU but since we only use this model to compute the response’s probabilities, the DeepSpeed Chat team figured out that using the CPU for this is not significantly slower.

DeepSpeed Chat also proposes LoRA fine-tuning. I used it for SFT and will use it again for RLHF since it can reach similar performance to full fine-tuning while saving a significant amount of memory.

To save even more memory, I recommend activating gradient checkpointing for all the models (without it, the VRAM consumption quickly becomes unreasonable).

To sum up, we can save memory with:

  • Gradient checkpointing

  • LoRA

  • Offloading the original model to the CPU

With all these optimizations applied, DeepSpeed Chat documentation claims that training the RLHF model won’t consume more memory than SFT (Step 1).

I verified it and can say that it’s almost true. In practice, you need a little bit more VRAM than what you used for SFT.

If you use small batches (size 2 or 4), you can train OPT-1.3B with RLHF on consumer hardware. It wouldn’t take more than 13 GB of VRAM while the CPU RAM consumption would stay below 10 GB. It’s even possible to train it for free on Google Colab.

However, with such small training batches and a consumer GPU (or the T4 of Google Colab), it would take more than 1 day of training. For this article, I used the A100 of Google Colab Pro. The GPU VRAM consumption peaked at 26.7 GB with a batch size of 24, but the training only took 2.5 hours (cost of around $3 on Colab).

Training the RLHF model with DeepSpeed Chat

The notebook for training the RLHF model is available here.

Keep in mind that RLHF is a very difficult training. You may struggle to find hyperparameters that make the training converge.

The important hyperparameters for performance and efficient training are the following:

  • data_path Dahoas/rm-static, and data_split 2,4,4: I use only one dataset for this step. As we saw in Step 1, “2,4,4” means that we use 20%, 40%, and 40% of the dataset for Step1, 2, and 3, respectively.

  • actor_model_name_or_path kaitchup/OPT-1.3B-SFT-DSChatLoRA: The model we trained in Step 1.

  • critic_model_name_or_path kaitchup/OPT-350M-RM-DSChat: The model we trained in Step 2.

  • per_device_generation_batch_size 16, and per_device_training_batch_size 16: Reduce the batch sizes if you don’t have enough VRAM. If you are training on consumer hardware with 24 GB of VRAM, I don’t recommend setting it greater than 8.

  • gradient_accumulation_steps 8: Increase the gradient_accumulation_steps if you decrease the batch size, in the same proportion. For instance, if you set the batch size to 8, set gradient_accumulation_steps to 16. It’s just a rule of thumb. You may try different values and see what works best.

  • ppo_epochs 1: The number of training epochs. 1 should be enough.

  • actor_learning_rate 5e-4, and critic_learning_rate 5e-6: I use the default values proposed by DeepSpeed Chat. They are not optimal but good enough for the models I use.

  • actor_zero_stage 0, and critic_zero_stage 0: I set it to 0 since I used only 1 GPU. If you use multiple GPUs, set it to 3. You can find more information on the different DeepSpeed zero stages here.

  • offload_reference_model: Offload the SFT model to the CPU to save some VRAM. It has only a little impact on the training speed so I recommend doing it.

  • actor_lora_dim 128: The dimensions of the LoRA adapters. I used the default value of DeepSpeed Chat. Note that this value is quite high. I don’t expect a loss in performance if you reduce it to 32 for instance, while training would be faster.

  • actor_gradient_checkpointing, and critic_gradient_checkpointing: We need gradient checkpointing for training unless you have high-end GPUs with more than 40 GB of VRAM.

  • disable_actor_dropout: Disabling dropout improves training stability. Enable it if you train for several epochs to avoid overfitting.

  • only_optimize_lora: Don’t forget this one! If you do, you would train LoRA and also update all the parameters of the model.

  • enable_hybrid_engine: The hybrid engine is another reason to use DeepSpeed Chat for RLHF. In my experiments, it made training 2x faster.

I trained the RLHF model with this command:

cd training/step3_rlhf_finetuning/
deepspeed --num_gpus 1 main.py \
   --data_path Dahoas/rm-static \
   --data_split 2,4,4 \
   --actor_model_name_or_path kaitchup/OPT-1.3B-SFT-DSChatLoRA \
   --critic_model_name_or_path kaitchup/OPT-350M-RM-DSChat \
   --num_padding_at_beginning 1 \
   --per_device_generation_batch_size 16 \
   --per_device_training_batch_size 16 \
   --generation_batches 1 \
   --ppo_epochs 1 \
   --max_answer_seq_len 256 \
   --max_prompt_seq_len 256 \
   --actor_learning_rate 5e-4 \
   --critic_learning_rate 5e-6 \
   --num_train_epochs 1 \
   --lr_scheduler_type cosine \
   --gradient_accumulation_steps 8 \
   --num_warmup_steps 100 \
   --deepspeed --seed 1234 \
   --actor_zero_stage 0 \
   --critic_zero_stage 0 \
   --offload_reference_model \
   --actor_lora_dim 128 \
   --actor_gradient_checkpointing \
   --critic_gradient_checkpointing \
   --disable_actor_dropout \
   --only_optimize_lora \
   --output_dir ./rlhf

Note: To speed up training, I used a batch size of 24 with gradient_accumulation_steps to 6. One training epoch took around 2.5 hours on the A100 40 GB of Google Colab PRO.

As we saw, two models are trained:

  • The RLHF model

  • The value model

DeepSpeed Chat saves both. Saving the value model is only useful if you want to resume training later.

Conclusion

This was the last step to train an instruct LLM with DeepSpeed Chat.

My main motivation for starting this series was to check how affordable it could be to train an RLHF model with DeepSpeed Chat. Affordable RLHF was one of the main claims of the authors of DeepSpeed Chat, but in their paper and documentation, they used multiple A100 GPUs which are far from affordable.

I confirm that it is definitely possible to run all 3 steps on consumer hardware to train a good instruct LLM.

However, DeepSpeed Chat tends to consume a huge amount of CPU RAM, before training, when processing the training datasets. I don’t know why. If you lack CPU RAM, consider running all three steps using only one dataset (instead of the 3 datasets I used).

DeepSpeed Chat is very simple to use. Each step can be trained with one command line. It’s also very fast. The total training time, combining all three steps, wasn’t more than 15 hours.

A next step to further explore DeepSpeed Chat and make RLHF even more affordable would be to apply QLoRA. It would enable larger training batches or the possibility to train larger models on consumer hardware.
