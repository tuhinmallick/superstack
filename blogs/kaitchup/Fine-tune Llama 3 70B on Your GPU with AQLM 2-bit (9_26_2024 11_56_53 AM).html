Large language models (LLMs) can be quantized to significantly reduce their size. However, once quantized, the model can’t be fine-tuned anymore. Updating weights in low precision, such as 4-bit, is yet something that we don’t know how to do well. Instead of directly updating the weights of the model, we can fine-tune an adapter on top of it. It’s what we do with QLoRA:

Most implementations of QLoRA only support 4-bit quantization. We can’t apply QLoRA to models quantized to a lower precision which might be necessary to further reduce the size of the model. An alternative is to fine-tune an adapter on top of a model quantized with another algorithm such as GPTQ which supports 2-bit and 3-bit quantizations. I’ve tried it but often couldn’t get good results. We need a better quantization algorithm.

With AQLM, we can accurately quantize very large models to a low precision, such as 2-bit. Using Mixtral-8x7b, I showed that we can fine-tune a model quantized with AQLM in 2-bit with only 24 GB of GPU RAM:

The main downside of AQLM is the cost of the quantization itself. For a 70B model such as Llama 3 70B, it can take weeks. Fortunately, the creators of AQLM released a 2-bit AQLM version of Llama 3 70B.

AQLM Quantization for Llama 3 70B

AQLM is a state-of-the-art quantization algorithm that performs especially well for quantization to low precisions such as 2-bit and 3-bit. I explained how AQLM works here:

Run a 7.7x Smaller Mixtral-8x7B on Your GPU with AQLM 2-bit Quantization

Read full story

The AQLM version of Llama 3 70B released by ISTA-DASLab is available here:

  • ISTA-DASLab/Meta-Llama-3-70B-Instruct-AQLM-2Bit-1x16

For this quantization, they use 1 codebook of 16 bits, i.e., the same hyperparameters used for their 2-bit quantization of Mixtral-8x7B. It yields a model 6.4x smaller than the original Llama 3 70B, reducing its size from 141 GB to 22 GB.

According to their own evaluation, this 2-bit version of Llama 3 70B didn’t lose much of its accuracy:

On the MMLU benchmark, which is often regarded as one of the most accurate benchmark to evaluate the performance of an LLM, 2-bit quantization only decreases the accuracy of the model from 79.8 to 75.9. This is still 10 points of accuracy more than Llama 3 8B while Llama 3 70B 2-bit is only 5 GB larger than Llama 3 8B.

5 GB for 10 points of accuracy on MMLU is a good trade-off in my opinion.

Nonetheless, while Llama 3 70B 2-bit is 6.4x smaller than the original version, 21.9 GB might still be a bit too much to make fine-tuning possible on a single consumer GPU.

Let’s verify whether we can fine-tune this model with 24 GB of GPU RAM.

Fine-tuning Llama 3 70B Quantized with AQLM 2-bit

To fine-tune Llama 3 70B quantized with AQLM, we can follow the same steps that I presented in the article fine-tuning Mixtral-8x7B quantized with AQLM:

  • Get the model quantized (we don’t quantize by ourselves because it’s too costly)

  • Load the model and its tokenizer

  • Create and mount the adapter on top of the model

  • Load (and format) the training data

  • Start training

The only important difference compared to the Mixtral-8x7B’s settings is that Llama 3 70B is almost 1.5x larger. As we saw in the previous section, the AQLM model has a size of 21.9 GB. If we want to use a consumer GPU (24 GB of GPU RAM), it remains only 2.1 GB for fine-tuning, i.e., to store the optimizer states, the gradients, and the activations. That’s challenging but not impossible.

With the right configuration of the LoRA adapter and training hyperparameters, we can fine-tune Llama 3 70B using only 23.5 GB of GPU RAM.

I used this LoRA configuration:

peft_config = LoraConfig(
        lora_alpha=8,
        lora_dropout=0.05,
        r=8,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ["gate_proj", "up_proj", "down_proj"]
)

This configuration creates 70.8M trainable parameters. That’s already a lot for an adapter. Increasing the rank would yield better results but then the fine-tuning would consume more than 24 GB of GPU RAM.

The target modules also have a significant impact on memory consumption. Here I only target the MLP modules. I chose these modules as targeting them instead of the self-attention modules (or a subset of them) usually leads to better results with the Llama architecture.

Nonetheless, to obtain the best results we should target all the linear layers (self-attention + MLP modules), and use a higher rank (e.g., 32 or 64). You can try it if you have access to a 32 GB GPU.

As for the training arguments, I used:

training_arguments = TrainingArguments(
        output_dir="./Llama-3-8B-aqlm-2bit-lora",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=1,
        gradient_accumulation_steps=16,
        per_device_eval_batch_size=1,
        log_level="debug",
        logging_steps=25,
        learning_rate=1e-4,
        eval_steps=25,
        save_steps=50,
        bf16=True,
        save_strategy='steps',
        max_steps=200,
        warmup_steps=25,
        lr_scheduler_type="linear",
)

If you are not familiar with the meaning of these arguments, have a look at my guide:

The arguments with the most impact on memory consumption are in bold.

  • optim: paged_adamw_8bit is the most memory-efficient implementation of AdamW. It quantizes the optimizer states to 8-bit and exploits CPU paging. AdaFactor is an even more memory-efficient alternative but it can be unstable. Use it if you need the fine-tuning to consume less memory.

  • per_device_train_batch_size and per_device_eval_batch_size: I set them to 1. Since Llama 3 70B is a large model, the activations, i.e., all the tensors created during the forward pass, consume a lot of memory. We can’t process more than one example at a time, hence a batch size of one.

  • gradient_accumulation_steps: I set it to 16. It means that we have a total training batch size of 16. This value is arbitrary. A higher value could yield better results.

I ran training for only 200 steps. It’s not enough and the validation was still significantly decreasing when the fine-tuning stopped. For this dataset, one epoch is usually long enough.

To use the fine-tuned adapter, we simply have to load it on top of the AQLM model for inference:

from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer
)
from peft import PeftModel
import torch

adapter_id = "./Llama-3-8B-aqlm-2bit-lora/checkpoint-200/"
model_id = "ISTA-DASLab/Meta-Llama-3-70B-Instruct-AQLM-2Bit-1x16"

model = AutoModelForCausalLM.from_pretrained(
    model_id,
    trust_remote_code=True, torch_dtype=torch.bfloat16, device_map="cuda", low_cpu_mem_usage=True, attn_implementation="flash_attention_2"

)
tokenizer = AutoTokenizer.from_pretrained(adapter_id)
model = PeftModel.from_pretrained(model, adapter_id)


prompt = "### Human: Hello! Tell me what can I cook for diner tonight.### Assistant:"
inputs = tokenizer(prompt, return_tensors="pt").to("cuda")
outputs = model.generate(**inputs, do_sample=True, max_new_tokens=150)
result = tokenizer.decode(outputs[0], skip_special_tokens=True)

print(result)

Conclusion

Fine-tuning Llama 3 70B on consumer hardware is possible.

Of course, fine-tuning a 2-bit version of the model can’t be as good as fine-tuning the original version but even the 2-bit version of Llama 3 70B performs better than most models of similar sizes. As we saw, it remains significantly better than Llama 3 8B.

Nonetheless, if you have more than 24 GB of GPU RAM available, I recommend increasing the LoRA rank and targeting all the linear modules to obtain much better results. A rank of 8 for such a large model is usually not suitable and limit the model’s ability to learn from the data.
