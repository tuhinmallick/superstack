Alibaba released Qwen2.5. The models are available in many sizes: 0.5B, 1.5B, 3B, 7B, 14B, 32B, and 72B, as base and instruct models:

  • Hugging Face collection: Qwen2.5

With all these sizes, Qwen2.5 offers flexibility across various hardware configurations. The 7B and 14B models are ideal for 24 GB GPUs, and they support efficient fine-tuning with LoRA and QLoRA. For full fine-tuning, smaller models are a better fit.

In this article, I explain how to fine-tune Qwen2.5, step by step. We will compare QLoRA, LoRA, and full fine-tuning speed and memory consumption. Since many model sizes are available, even if you only have a small 8 GB GPU, you will find a Qwen2.5 model that can be fine-tuned on your hardware.

Qwen2.5: Qwen2 but Trained on 18T Tokens!

The Qwen2.5 language models were pre-trained on a dataset of 18 trillion tokens. This massive scale has enabled Qwen2.5 to acquire substantially more knowledge than Qwen2 which was trained on 7T tokens. To give you another point of comparison, Llama 2, which was released a bit more than one year ago, was trained on 2T tokens!

LLM makers are scaling training very quickly and I guess we will have LLMs trained on more than 20T tokens before 2025.

Other than the training data's increase in size, Qwen2.5 seems very similar to Qwen2. We don’t know much more yet. They published a blog post but it mainly focuses on telling us how the models are better.

Qwen2.5: A Party of Foundation Models!

Qwen2.5 Full Fine-tuning

Full fine-tuning (FFT) consumes much more memory than PEFT methods like LoRA and QLoRA. This is mainly because FFT updates all the model’s weights. The optimizer, AdamW, has to create and store 2 new parameters for each parameter of the model. For Qwen2.5 7B, we have to store 14B optimizer parameters in memory. Keeping them in full precision (fp32) would cost 14B*4 bytes = 56 GB of GPU RAM.

To make FFT more cost-effective, we can quantize the optimizer states to 8-bit, reducing their memory consumption by a factor of four, bringing it down to 14 GB. Additionally, using a paged optimizer like paged_adamw allows these states to occupy GPU RAM only when available; otherwise, they are offloaded to the CPU RAM. While this makes FFT slower, the unified memory of NVIDIA GPUs ensures that the performance remains sufficiently fast.

In addition to the optimizer, the activations generated and stored for updating the model’s gradients during backpropagation also consume a significant amount of memory. The exact memory usage depends on various training hyperparameters, such as the length of training examples and batch size. To prevent activations from exceeding 100 GB of GPU RAM, for a 7B model, we need to enable gradient checkpointing (also known as 'activation checkpointing').

With these optimizations, and if we load the model with bfloat16 instead of full precision, it is possible to fully fine-tune Qwen2.5 on a 40 GB GPU.

First, we need to install the following packages:

pip install --upgrade transformers bitsandbytes accelerate datasets trl

  • We need transformers to load the model, the tokenizer, and because it’s a dependency of the other packages.

  • bitsandbytes is used to quantize the optimizer states.

  • accelerate allows us to specify where to load the model, e.g., on the first GPU of your machine.

  • datasets is necessary to load and process the training dataset.

  • trl implements very simple but powerful methods for fine-tuning LLMs.

Full Fine-tuning Code for Qwen2.5 7B

We will need the following methods:

import torch, os, multiprocessing
from datasets import load_dataset
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    set_seed
)
from trl import SFTTrainer, SFTConfig

First, I set a "seed". This makes the training reproducible if we want to do the same again later. Otherwise, the training loop will make training batches that will be different since they are randomly created given the seed.

set_seed(1234)

Then, before we load the model, we have to decide with which precision we want to load the weights. If you have a recent GPU, I recommend the bfloat16 data type. Otherwise, you will have to opt for float32 but will need much more memory.

If you have a GPU that supports bfloat16, it also supports FlashAttention which further reduces memory consumption. The alternative for older GPUs is Pytorch’s SDPA.

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float32
  attn_implementation = 'sdpa'

Next, we load the tokenizer and configure it. Qwen2.5’s tokenizer doesn’t support add_eos_token so we will have to add the EOS token to each training example by ourselves, as explained in this article:

We also need to set a pad token. We need, preferably, a special token from the model’s vocabulary not used by Qwen2.5. We have plenty. Most of the special tokens listed in the tokenizer configuration file are not used. We can use <|im_end|> which is a special token used by the instruct version but not by the base version of Qwen2.5.

As for the padding side, I didn’t see any difference in terms of performance between choosing right or left.

model_name = "Qwen/Qwen2.5-7B"
tokenizer = AutoTokenizer.from_pretrained(model_name)
tokenizer.pad_token = "<|im_end|>"
tokenizer.pad_token_id = 151645
tokenizer.padding_side = 'left'

Now that the tokenizer is ready, we can load the dataset that we will use for training. Here, my goal is to make a cheap multilingual chat model based on Qwen2.5 with this chat format:

### Human: {my prompt}### Assistant: {model’s response}

We can use timdettmers/openassistant-guanaco for this. It is small enough so that we can fine-tune Qwen2.5 7B for one epoch in less than one hour.

ds = load_dataset("timdettmers/openassistant-guanaco")

Remember that datasets don’t have an EOS token. This token is specific to the model. If we use the dataset as it is for training, Qwen2.5 won’t learn when to stop generating its answers. We need to add an EOS token to each training example. The model will learn when to generate it, and then the inference framework will stop the generation once it reads that the model generates the EOS token.

#Add the EOS token
def process(row):
    row["text"] = row["text"]+tokenizer.eos_token
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

The next step is to load the model:

model = AutoModelForCausalLM.from_pretrained(
          model_name, device_map={"": 0}, attn_implementation=attn_implementation, torch_dtype=compute_dtype
)

AutoModelForCausalLM automatically finds the corresponding model architecture given the "model_name". With device_map, I specify that I want to load the model on my first device, i.e., my GPU. Then, attn_implementation and torch_dtype are the attention implementation (e.g., FlashAttention) and the data type of the parameters (e.g., bfloat16), respectively. If you don’t set torch_dtype, Transformers will load the model with a float32 dtype, meaning that the model will consume twice more memory.

The next step is very important to save memory: gradient checkpointing.

model.gradient_checkpointing_enable(gradient_checkpointing_kwargs={'use_reentrant':True})

It takes as argument "gradient_checkpointing_kwargs" with "use_reentrant". Set it to True. Note: We will see next week in detail, in another article, the impact of gradient checkpointing and what is "use_reentrant".

Next, we set the training arguments and hyperparameters. Since we will use TRL’s SFFTrainer, we have to define them in "SFTConfig":

training_arguments = SFTConfig(
        output_dir="./Qwen2.5_7B_FFT",
        eval_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=2,
        gradient_accumulation_steps=16,
        per_device_eval_batch_size=2,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=25,
        learning_rate=1e-5,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=25,
        num_train_epochs=1,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
        dataset_text_field="text",
        max_seq_length=512,
)

I explain all these hyperparameters in detail in this article:

I set the training batch size to 2. Even if you have more memory available, I don’t recommend using a batch size of 4. After many experiments with various GPUs, I found that a batch size of 4 is usually poorly optimized, especially when gradient checkpointing is activated for some reason. Possible optimal values for the batch size are 1, 2, 8, 16, 32, 64, etc.

Then, we can start training:

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        tokenizer=tokenizer,
        args=training_arguments,
)
trainer.train()

Qwen2.5 Full Fine-tuning Memory Requirements

I ran training on Google Colab’s A100 which has 40 GB of memory.

  • 50.47 minutes used for training.

  • Peak reserved memory = 37.771 GB.

  • Peak reserved memory for training (excluding the memory used by the model itself) = 21.667 GB.

  • Peak reserved memory % of max memory (40 GB) = 95.468 %.

  • Peak reserved memory for training % of max memory = 54.764 %.

Thanks to paged AdamW, we could use almost 100% of the GPU memory.

Full Fine-tuning with Minivoc

I made Minivoc versions of Qwen2.5 1.5B and Qwen2.5 7B:

  • kaitchup/Qwen2.5-1.5B-Minivoc-32k-v0.1a (Apache 2.0 license)

  • kaitchup/Qwen2.5-7B-Minivoc-32k-v0.1a (For The Kaitchup Pro subscribers Only; get your access token)

With Minivoc, we reduce the vocabulary size of Qwen2.5 from 151k to 32k tokens. The main advantages are that it makes the model much more memory-efficient and slightly faster, for both inference and fine-tuning. In other words, Minivoc models are significantly cheaper to run.

Qwen2.5 7B Minivoc 32k consumes slightly less memory with a batch size of 16 than Qwen2.5 7B with a batch size of 2. It makes fine-tuning 8x cheaper!

And, on average, they have an accuracy on public benchmarks that is slightly better (Qwen2.5 7B) or worse (Qwen2.5 1.5B) than the original model.

I found that the accuracy largely depends on what dataset I use to train Minivoc. Currently, I use 0.2B tokens randomly sampled from the HuggingFaceFW/fineweb-edu. This training seems to favor better accuracy on Arc Challenge, GPQA, and MUSR while degrading it on the BBH and MMLU tasks.

I couldn’t spend more time improving them but if you are interested in getting better checkpoints, or if you want Minivoc versions of other Qwen2.5 models (14B, 32B, 72B, etc), please let me know.

Qwen2.5 LoRA Fine-tuning

LoRA creates adapters (small tensors) and mounts them on top of the model’s linear modules such as self-attention’s Q, K, and V, and MLP while freezing all the original parameters of the model. Since we only have to update the weights of the adapter, the optimizer states are significantly smaller than for full fine-tuning. Their memory footprint is almost negligible.

With LoRA, it is possible to fine-tune Qwen2.5 7B on a consumer GPU with 24 GB of RAM.

Using LoRA only requires adding a few more lines to the full fine-tuning code.

We will need to install the same packages in addition to peft.

pip install --upgrade transformers bitsandbytes accelerate datasets trl peft

  • peft contains all the methods we need to create a LoRA configuration.

LoRA Fine-tuning Code for Qwen2.5 7B

It is the same code as for full fine-tuning, except that we have to add a LoraConfig:

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",  
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

lora_alpha, r, and lora_dropout are hyperparameters specific to LoRA. Changing them has little impact but if you really care about achieving the lowest loss (for instance, if you are writing a scientific paper and trying to beat a baseline), then increasing r should help but it would cost more memory.

As for the target modules, here I target all the "linear" modules of the model, except the language modeling head. To find the linear module of a model, simply print it "print(model)" and you will see all the modules that are "linear".

Then, we have to pass this LoraConfig to the SFTTtrainer:

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        tokenizer=tokenizer,
        args=training_arguments,
)

and as float32:

Qwen2.5 QLoRA Fine-tuning

QLoRA, it is LoRA but this time the model is quantized, usually to 4-bit with bitsandbytes even though we saw that other quantization methods can be faster and as accurate:

4-bit quantization reduces the size of Qwen2.5 7B from 16 GB (16-bit) to 5.5 GB.

We need to install the same packages as for LoRA:

pip install --upgrade transformers bitsandbytes peft accelerate datasets trl

QLoRA Fine-tuning Code for Qwen2.5 7B

We must define a bitsandbytes configuration:

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)

  • bnb_4bit_quant_type is the data type used for quantization. nf4 stands for NormalFloat4.

  • bnb_4bit_compute_dtype is the data type used for the forward pass, usually float16 or bfloat16.

  • bnb_4bit_use_double_quant indicates that we will quantize the quantization constants, i.e., nested quantization.

For more details about QLoRA, I wrote about it here:

Then, we load the model with this quantization configuration:

model = AutoModelForCausalLM.from_pretrained(
          model_name, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
)

These could be the only changes. However, I usually also replace model.gradient_checkpointing_enable with the following line:

model = prepare_model_for_kbit_training(model, gradient_checkpointing_kwargs={'use_reentrant':True})

prepare_model_for_kbit_training does several things:

  • Cast the layernorm and the language modeling head to fp32

  • Activate gradient checkpointing

It’s really helpful in making QLoRA fine-tuning more accurate.

The remaining code is the same as for LoRA fine-tuning.

Qwen2.5 QLoRA Fine-tuning Memory Requirements

For QLoRA, Google Colab’s L4 had enough memory (24 GB of RAM).

  • 209.31 minutes used for training.

  • Peak reserved memory = 18.529 GB.

  • Peak reserved memory for training = 7.406 GB.

  • Peak reserved memory % of max memory = 83.584 %.

  • Peak reserved memory for training % of max memory = 33.409 %.

QLoRA is much slower than LoRA. Even if we had used the A100, QLoRA is usually twice as slow but also much more memory-efficient.

Conclusion.

We explored three fine-tuning methods for Qwen2.5 7B that you can straightforwardly adapt to other model sizes:

  1. Full Fine-Tuning (FFT): FFT is the best performing but memory-intensive, making it suitable for smaller models unless you are GPU-rich.

  2. LoRA Fine-Tuning: This method introduces adapters to specific layers, freezing the original model parameters and significantly reducing memory consumption. LoRA enables fine-tuning larger models like the 7B and 14B variants on GPUs with 24 GB of RAM.

  3. QLoRA Fine-Tuning: By quantizing the model to 4-bit precision, QLoRA further reduces memory requirements, allowing for efficient training on consumer-grade GPUs. However, this comes at the cost of slower training speeds compared to standard LoRA.

The Qwen2.5 models are probably the best LLMs that you can find now. If you were using Gemma 2 or Llama 3.1, I suggest that you take a look at these new models.
