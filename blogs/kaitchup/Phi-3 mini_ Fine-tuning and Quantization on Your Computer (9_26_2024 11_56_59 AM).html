Phi-3 “mini” is the new Phi model released by Microsoft. It has 3.8 billion parameters, 1.1 billion more parameters than Phi-2. Even though the number of parameters is growing, a 3.8B model can still be fine-tuned with a very affordable GPU.

Despite their small size, the Phi models have always performed very well thanks to their pre-training on synthetic data generated by larger and better models. Microsoft’s own evaluation shows that Phi-3 mini is as good as Llama 3 8B on some benchmarks.

Phi-3: Phi-2 but Larger, Trained Longer, and Adopting the Llama Architecture

Microsoft published a technical report presenting Phi-3:

Phi-3 Technical Report: A Highly Capable Language Model Locally on Your Phone

This is a short report that doesn’t give a lot of information. We learn that Microsoft exploits the Llama architecture and Llama 2’s tokenizer for Phi-3. Phi-3 mini has:

  • Hidden dimensions: 3072

  • Layers: 32

  • Attention heads: 32

Two versions supporting different context lengths are available:

  • microsoft/Phi-3-mini-4k-instruct: Can support a context of up to 4k tokens

  • microsoft/Phi-3-mini-128k-instruct: Can support a context of up to 128k tokens

The context of the 128k version was extended with Microsoft’s LongRoPE. Surprisingly, it can only handle 128k tokens while the LongRoPE paper claimed to extend context length to more than 2 million tokens without degrading the model's performance.

In addition to the architecture change, Phi-3 was trained on a new dataset better than the one used for Phi-2 according to the authors. It’s still a mix of web data and synthetic data generated by other LLMs but we don’t know in what proportions. They mention that Phi-3 was first trained on web data only and then, in a second phase, synthetic data was introduced in the training data.

Phi-3 is also an instruct model. They seem to have used standard DPO for instruct fine-tuning.

The remainder of the paper is about the benchmark scores. Phi-3 performs extremely well, given its size, on most of them. However, there is no information on how they have decontaminated the training data, i.e., removing the benchmark data from the training data. As usual, my recommendation is to ignore all these scores and run your own evaluation for your own tasks.

Fine-tuning Phi-3 with LoRA and QLoRA

Fine-tuning Phi-3 is easier than fine-tuning Phi-2 thanks to its Llama architecture. The only particularity of Phi-3 fine-tuning is that we have to set the UNK token as the pad token. Note: Microsoft also proposes a fine-tuning script in the model repository.

QLoRA fine-tuning is possible on an 8 GB GPU while LoRA fine-tuning, which yields better results, is possible on a 16 GB GPU and maybe on a 12 GB GPU with a smaller batch size.

We need to install the following packages:

pip install -qqq --upgrade bitsandbytes transformers peft accelerate datasets trl flash_attn

For example, I fine-tuned Phi-3 mini on the instruction dataset “timdettmers/openassistant-guanaco” for three epochs.

Here is the full code for QLoRA fine-tuning:

import torch
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig,
    TrainingArguments,
)
from trl import SFTTrainer

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

model_name = "microsoft/Phi-3-mini-4k-instruct"

tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True, add_eos_token=True, use_fast=True)
tokenizer.pad_token = tokenizer.unk_token
tokenizer.pad_token_id = tokenizer.convert_tokens_to_ids(tokenizer.pad_token)
tokenizer.padding_side = 'left'

ds = load_dataset("timdettmers/openassistant-guanaco")

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, torch_dtype=compute_dtype, trust_remote_code=True, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
)

model = prepare_model_for_kbit_training(model)

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

training_arguments = TrainingArguments(
        output_dir="./Phi-3_QLoRA",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=4,
        gradient_accumulation_steps=8,
        per_device_eval_batch_size=4,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=100,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=100,
        num_train_epochs=3,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

If we don’t write the lines:

tokenizer.pad_token = tokenizer.unk_token
tokenizer.pad_token_id = tokenizer.convert_tokens_to_ids(tokenizer.pad_token)

and set the EOS token as the pad token by default, Phi-3 will generate new and irrelevant tokens until it reaches the max_seq_len.

It takes 4 hours to complete this fine-tuning. Note: I used the L4 instance of Google Colab.

The training logs:

For LoRA fine-tuning, we only need to remove “quantization_config=bnb_config” when loading the model, and replace:

model = prepare_model_for_kbit_training(model)

with:

model.gradient_checkpointing_enable()

LoRA fine-tuning is faster thanks to the absence of quantization.

It took 3 hours and 13 minutes.

The training logs:

As expected, LoRA yields a lower (better) perplexity than QLoRA since we didn’t quantize the model.

Quantization with GPTQ

GPTQ is a quantization method that produces fast and accurate quantized models.

Feb 1
Read full story

I use AutoGPTQ which supports 2, 3, 4, and 8-bit quantizations. Since Phi-3 is small, I don’t recommend choosing a lower precision than 4-bit. The quantization errors below 4-bit precision are significantly worse.

I use this code to quantize Phi-3 mini:

from transformers import AutoModelForCausalLM, AutoTokenizer
from optimum.gptq import GPTQQuantizer
import torch
model_path = 'microsoft/Phi-3-mini-4k-instruct'
w = 4 #quantization to 4-bit. Change to 2, 3, or 8 to quantize with another precision

quant_path = 'Phi-3-mini-4k-instruct-gptq-'+str(w)+'bit'

# Load model and tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_path, use_fast=True, trust_remote_code=True)
model = AutoModelForCausalLM.from_pretrained(model_path, torch_dtype=torch.float16, device_map="auto", trust_remote_code=True)
quantizer = GPTQQuantizer(bits=w, dataset="c4", model_seqlen = 2048)
quantized_model = quantizer.quantize_model(model, tokenizer)

quantized_model.save_pretrained("./"+quant_path, safetensors=True)
tokenizer.save_pretrained("./"+quant_path)

The original Phi-3 mini model occupies 7.6 GB on the hard drive. After quantization, it only occupies 2.3 GB.

I pushed the quantized version to the Hugging Face Hub:

  • kaitchup/Phi-3-mini-4k-instruct-gptq-4bit

Conclusion

Phi-3 mini is a small model that is easy to use on consumer hardware. For inference on the GPU, a 4 GB GPU is enough if the model is quantized to 4-bit. For QLoRA fine-tuning, an 8 GB GPU is enough.
