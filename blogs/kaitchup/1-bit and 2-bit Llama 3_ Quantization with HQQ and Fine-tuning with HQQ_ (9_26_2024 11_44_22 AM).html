1-bit quantization significantly reduces the size of large language models (LLMs) by replacing their weights with 0s and 1s. This quantization method is very aggressive compared to 4-bit quantization. Naive 1-bit quantization would turn any LLM into a gibberish generator.

Several approaches have been proposed to improve 1-bit quantization for LLMs. For instance, HQQ 1-bit quantization has been applied to Llama 2 7B. While it damages the model, it has been shown that fine-tuning an adapter on top of a 1-bit LLM quantized with HQQ can recover a significant part of the lost accuracy.

An Overview of Half-Quadratic Quantization (HQQ)

There are two main types of weight-only quantization approaches: data-free methods like bitsandbytes, which use only the model weights, and calibration-based methods like GPTQ and AWQ, which rely on external datasets in addition to the model weights.

While calibration-based methods often provide better quantization quality, they face challenges such as calibration data bias and significant computational requirements, making it difficult to test and deploy multiple models.

Other methods such as AQLM are even more accurate but with a higher cost as the quantization method must be trained.

The goal of Half-Quadratic Quantization (HQQ) is to achieve the quality of calibration-based methods with the speed of calibration-free methods.

Basic quantization often reduces model accuracy due to the wide range of weight values, especially outliers. Algorithms like GPTQ and AWQ use calibration data to minimize errors in layer outputs. HQQ, on the other hand, focuses on minimizing errors in the weights. By using a sparsity-promoting loss, HQQ effectively models outliers.

HQQ proposes a robust optimization approach to find the quantization parameters by minimizing the sparsity-promoting loss function between the original and dequantized weights.

This quantization method runs calculations in inference mode with half-precision and converges in just a few iterations. This results in a significant speed-up allowing even the largest models to be processed in minutes.

Regarding quantization accuracy, HQQ yields quantized models that outperform GPTQ, especially for 2-bit quantization.

With HQQ, we can even quantize to 1-bit, i.e., replacing the model’s parameters with binary parameters (0s and 1s).

1-bit quantization is of course even more challenging. Since we can only use 0 or 1 for each parameter, we have to expect a significant loss of accuracy. To try to recover part of this lost accuracy, the authors of HQQ demonstrated that fine-tuning an adapter on top of 1-bit/2-bit LLMs is very effective. They called it HQQ+. Here are their results for Llama 2 7B:

It works but 8.53 is still a very high perplexity. Models of the size of Llama 2 7B, and smaller, are notoriously very difficult to quantize to low precisions. We can expect large models such as Llama 3 70B to be less negatively impacted by 1-bit quantization.

How to Quantize Llama 3 to 1-bit with HQQ and Hugging Face Transformers

HQQ is supported by Hugging Face Transformers. Using HQQ is as simple as using bitsandbytes and significantly faster than GPTQ.

For all the following experiments, I used the L4 instance of Google Colab which has 22.5 GB of GPU RAM available.

1-bit quantization with HQQ took 1.1 minutes for Llama 3 8B and almost 22 minutes for Llama 3 70B. This is extremely fast compared to GPTQ which would take hours while consuming much more than 22 GB of GPU RAM. As for quantization with AQLM, it would have taken days.

For this quantization, we first need to install the necessary libraries:

pip install hqq
pip install --upgrade bitsandbytes transformers peft accelerate datasets trl

Note: I also installed at the same time all the libraries we will need to fine-tune the model.

Then, the quantization is done while the model is loading, using a quantization configuration defined in an HqqConfig object:

model_id = "meta-llama/Meta-Llama-3-8B"

quant_config = HqqConfig(nbits=1, group_size=32, quant_zero=False, quant_scale=False, axis=0)

tokenizer = AutoTokenizer.from_pretrained(model_id, use_fast=True)
tokenizer.pad_token = tokenizer.eos_token

model = AutoModelForCausalLM.from_pretrained(
    model_id,
    torch_dtype=torch.bfloat16,
    device_map="cuda",
    quantization_config=quant_config
)

This is a very standard code except for the HqqConfig. Three of its arguments are particularly important:

  • nbits: Precision of the quantization. In this article, I only experimented with 1 and 2.

  • group_size: For higher nbits, a group size of 128 is fine. This value is also the default for other algorithms such as GPTQ. Lower values make the quantization more accurate but also yield larger models. For very low-precision quantization, e.g., 1-bit and 2-bit, 64 or 32 for the group size will yield significantly better models than with a group size of 128.

  • axis: The axis on which the quantization grouping is done. I got slightly better results when setting it to 0.

Then, for the fine-tuning, I used the same code that I usually use to fine-tune Llama 3, except that I replaced the bitsandbytes quantization with the HQQ quantization:

from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    HqqConfig
)
from trl import SFTTrainer, SFTConfig
import torch, multiprocessing

model_id = "meta-llama/Meta-Llama-3-8B"
# All linear layers will use the same quantization config
quant_config = HqqConfig(nbits=1, group_size=32, quant_zero=False, quant_scale=False, axis=0)

#Load the tokenizer to save it along with the model
tokenizer = AutoTokenizer.from_pretrained(model_id, use_fast=True)
tokenizer.pad_token = tokenizer.eos_token

# Load and quantize
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    torch_dtype=torch.bfloat16,
    device_map="cuda",
    quantization_config=quant_config
)

model = prepare_model_for_kbit_training(model)

dataset = load_dataset("timdettmers/openassistant-guanaco")

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

#Add the EOS token
def process(row):
    row["text"] = row["text"]+"<|end_of_text|>"
    return row

ds = dataset.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)
training_arguments = SFTConfig(
        output_dir="./Llama3_8b_HQQ-1bitgs32a0-adapter",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=8,
        gradient_accumulation_steps=4,
        per_device_eval_batch_size=8,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=100,
        learning_rate=1e-4,
        eval_steps=100,
        num_train_epochs=3,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

For Llama 3 8B, this configuration consumes:

  • With 1-bit quantization: 13.6 GB of GPU RAM

  • With 2-bit quantization: 15.3 GB of GPU RAM

With the L4 GPU of Google Colab, the fine-tuning requires 8 hours.

For Llama 3 70B, I only tried this configuration:

quant_config = HqqConfig(nbits=1, group_size=64, quant_zero=False, quant_scale=False, axis=0)

It consumes around 20.5 GB of GPU RAM. Using 2-bit quantization or a lower group size would require more than 24 GB of GPU RAM. I only ran the fine-tuning for 1 epoch. 3 epochs would have required 55 hours.

Note: It’s not clear on the plots but the loss for the 2-bit configurations is also decreasing and the adapter is learning. Check the notebook to see the loss values.

All the 1-bit models yield a high loss. Fine-tuning an adapter as proposed by HQQ+ indeed helps to improve the model but doesn’t sufficiently reduce the loss of the model for language generation.

On the other hand, the 2-bit configurations are much better. While Llama 3 8B 2-bit with a group size of 64 (red curve) is probably useless (at step 0), fine-tuning improves the model by a large margin.

The 1-bit models are completely unable to generate language except for the configuration Llama 3 8B 1-bit gs=32 axis=0 which outputs more probable sequences of tokens (“I like”, “I'm sorry for you”, etc.).

The 2-bit versions on the other hand don’t show signs of quantization. The answers generated are fluent, relevant to the prompt, and look like they could have been generated by the original model.

I don’t think we can say that the 1-bit models are broken. For instance, low-precision quantization with GPTQ can yield a model with a perplexity well above 1,000. In that case, I would say that the model is truly broken, i.e., it can’t model anymore the probability of a sequence of tokens. Even fine-tuning an adapter on top of it wouldn’t be able to fix it. In contrast, with HQQ, the perplexity can easily decrease below 10 thanks to fine-tuning. This is far from being good enough but the model still can learn what are frequent and probable tokens given the context. This is not enough for language generation but it might be enough for simpler tasks such as classification.

Conclusion

Fine-tuning an adapter on top of a 1-bit HQQ model, as proposed by HQQ+, indeed significantly improves the performance but not enough for language generation. All the 1-bit versions of Llama 3 that I have tried can’t generate understandable output.

However, the 2-bit versions of Llama 3 8B are surprisingly good. They enable inference on a 6 GB GPU while still being able to generate answers relevant to the user prompt.

Since HQQ is very fast at quantizing models, 2-bit quantization with HQQ followed by the fine-tuning of an adapter seems to be an interesting option for small GPUs.
