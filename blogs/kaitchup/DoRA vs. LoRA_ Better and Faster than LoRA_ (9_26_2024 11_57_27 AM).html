LoRA is the most used technique for parameter-efficient fine-tuning (PEFT) of large language models (LLMs). This PEFT method adds a few trainable parameters, i.e., an adapter, on top of the LLM whose parameters are frozen. Since only the adapters parameters are trained during fine-tuning, LoRA is significantly more memory-efficient and faster to converge.

While LoRA can be nearly as good as a standard full fine-tuning, it often remains some performance gap. Moreover, finding optimal LoRA hyperparameters, especially the rank (r), is a tedious task.

Many alternatives have been proposed to extend and improve LoRA with various goals such as fine-tuning quantization-aware adapters (e.g., QA-LoRA, LQ-LoRA), reducing further the number of trainable parameters (e.g., VeRA), and using LoRA to pre-train LLMs from scratch (e.g., ReLoRA, LoRA-the-Explorer).

DoRA has been proposed to improve LoRA with a better theoretical grounding. DoRA is more robust to hyperparameter changes, learns faster (i.e., requires fewer training examples), is more parameter-efficient, and further closes the performance gap with full fine-tuning.

DoRA vs. LoRA? Which one is better?

In this article, I explain DoRA. Then, I show how to use DoRA to fine-tune Mistral 7B on consumer hardware and compare it with the standard LoRA.

DoRA: LoRA but with a New Magnitude Vector

DoRA, short for Weight-Decomposed Low-Rank Adaptation, is presented in this paper by NVIDIA and HKUST:

DoRA: Weight-Decomposed Low-Rank Adaptation

DoRA decomposes the pre-trained model that we want to fine-tune into two components:

  • a magnitude vector

  • a directional matrix

DoRA's strategy to decouple magnitude from direction aims to refine the update mechanism, applying LoRA specifically to the directional aspect and permitting separate adjustments of the magnitude.

The motivation behind this decomposition is to make the parameter updates look like the updates seen during full fine-tuning, as shown in this figure:

The authors illustrate DoRA with the figure below:

The typical LoRA (represented by modules A and B in the figure) is still applied to the pre-trained model and trained during fine-tuning. DoRA introduces new trainable parameters with the magnitude vector.

It yields a tiny increase in the number of trainable parameters (around +0.01%) compared with LoRA. This increase is compensated by DoRA achieving better performance than LoRA with a lower rank (the hyperparameter r of LoRA).

The authors have evaluated DoRA in multiple language and visual tasks and found that DoRA outperforms previous PEFT methods, as shown in the table below.

Unfortunately, they didnt compare DoRA with full fine-tuning on these tasks. We dont know how much of the performance gap remains to be closed between full fine-tuning and DoRA. Note: For vision tasks, they show that DoRA may outperform full fine-tuning. Many reasons could explain these surprising results. The authors assume that full fine-tuning overfits, i.e., the training hyperparameters and training data needed further adjustments.

Fine-tuning Mistral 7B with DoRA

DoRA is already supported by the PEFT library. It is as simple as setting use_dora=True in the LoraConfig, for instance:

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        use_dora=True,
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

This DoRA configuration yields 43.319M trainable parameters.

The full code to train a DoRA adapter on top of Mistral 7B is in the notebook (#51). I used timdettmers/openassistant-guanaco for this fine-tuning which is small but good enough to turn a base model into an instruct/chat model.

I ran the training for 3 epochs with a training batch size of 4 and a gradient accumulation step of 6 (total batch size = 4*6 = 24).

The training consumes up to 21.4 GB of GPU RAM. It can run on consumer hardware using a 24 GB GPU. If you only have a 16 GB GPU, you can apply the following optimizations:

  • Reduce the maximum sequence length to 512 (instead of 1024, see below).

  • Reduce the training batch size to 1 while setting the gradient accumulation steps to 24 to keep a total training batch of 24.

  • Reduce the rank r in the LoraConfig, e.g., to 8 or 4.

  • Remove some target modules in the LoraConfig, for instance, you may only keep the MLP modules "gate_proj", "down_proj", and "up_proj".

The current implementation of DoRA doesnt support bitsandbytes quantization for now. We cant perform QLoRA, or QDoRA, to reduce the memory consumption further with quantization. The support for quantized layers should be added shortly, once this PR is merged into the PEFT library. Note: If the PR is merged when you read this article, then training with QDoRA only requires passing a quantization configuration when loading Mistral 7B, similar to what I did in this article.

The validation loss doesnt decrease but the training loss does. This discrepancy is rather due to the task itself than DoRA. For confirmation, I ran LoRA with the same configuration and hyperparameters.

My fine-tuning logs using LoRA:

LoRA yields a similar learning curve but is almost twice as fast.

While the addition of the magnitude vector in DoRA only creates 1.3M parameters more than LoRA (+3.3%), it seems to significantly slow down fine-tuning, especially during the validation runs. In other words, the current implementation is not optimized yet. To avoid slow inference, Hugging Face currently recommends merging the DoRA adapter into the base model rather than loading it.

DoRA introduces a bigger overhead than pure LoRA, so it is recommended to merge weights for inference, see LoraModel.merge_and_unload().

Since merging adapters trained on top of a quantized model is far from optimal, this overhead will be problematic for fast inference with adapters trained with QDoRA.

Conclusion

DoRA improves upon LoRA by adding a new trainable magnitude vector. We can train a DoRA adapter with PEFT but the implementation is currently slower than the standard LoRA.

Moreover, in my experiments, I didnt observe a significant difference between LoRA and DoRA in accuracy in downstream tasks. According to the DoRA paper, this difference should be more obvious with different ranks, e.g., a LoRA adapter with a rank of 64 may perform as well as a DoRA adapter with a rank of 32, while the latter has twice as few trainable parameters as the former. If you want to save memory, then using DoRA might be a better choice.

Nonetheless, overall, I would always use DoRA since it is better theoretically than LoRA. However, if you dont want to merge the adapter once trained, I would wait for Hugging Face to better optimize their DoRA implementation in PEFT (or use another library).
