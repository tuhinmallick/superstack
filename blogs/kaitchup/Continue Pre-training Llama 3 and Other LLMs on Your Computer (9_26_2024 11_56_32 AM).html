Recent base large language models (LLM) are pre-trained on trillions of tokens. The pre-training data are text usually extracted from the Web without targeting any specific domains or tasks. In contrast, fine-tuning a base LLM requires much less data and exploits data for specific tasks or domains.

“Continued” pre-training is yet another step that can be executed between pre-training and fine-tuning. Continuing pre-training is especially helpful when we want to teach a pre-trained LLM a new language or very specific domains for which we have millions of tokens. You can see it as a fine-tuning but without any particular tasks in mind.

In this article, I show how to continue pre-training LLMs. We will review the main differences between fine-tuning and continued pre-training. I use Llama 3 8B and a recipe proposed by Unsloth to make it possible on consumer hardware.

Continued Pre-training vs. Fine-tuning

Technically, continued pre-training is a fine-tuning. We want to adapt the model to a particular domain. The difference between continued pre-training and fine-tuning is in the nature of the dataset used and the objective.

For fine-tuning, we need training examples illustrating our target tasks. For instance, for a binary classification task, we would need a sequence of tokens paired with labels 0 or 1. We want the model to learn the label given the sequence of tokens. For instruct fine-tuning, we need pairs of prompts and answers. We want the model to learn the answer given the prompt.

On the other hand, for continued pre-training, our objective is simply to adapt the model’s weights to a new domain without any particular task in mind. The dataset we need can be any text in the targeted domain.

For instance, if we want to turn a base LLM into a Japanese law assistant, continued pre-training on millions of tokens from law documents written in Japanese will make the LLM much better in the law domain and Japanese language. Then, once continued pre-training is over, we can fine-tune our model with an instruction dataset in the law domain in Japanese to make it a chatbot specialized in this domain. This would be a small dataset. If we directly fine-tune the base LLM on this dataset, without continued pre-training, the LLM might struggle to generate Japanese language and to understand law terms.

Continued pre-training also requires slightly different hyperparameter settings than fine-tuning, especially if we use LoRA or QLoRA.

Since we want to adapt the model to a new domain/language, we must update the token embeddings and language modeling head to make sure that the model will better model the in-domain tokens/terms that might have been very rare in the pre-training data of the base LLM.

If we use LoRA or QLoRA, using a higher rank than usual, i.e., adding more trainable parameters, is also recommended to make sure the adapter will have enough capacity to learn the new domain features.

Continue Pre-training Llama 3 on High-quality Educational Content

I first got the idea for this article when I read this blog post by Unsloth:

Continued Pretraining with Unsloth

Then, following the release by Hugging Face of FineWeb-Edu, I thought about using this dataset to continue pre-training a state-of-the-art LLM such as Llama 3. If you don’t know FineWeb-Edu, it’s a large dataset (1.3T tokens) of high-educational content. According to experiments by Hugging Face, LLMs trained on this dataset are better on public benchmarks. So let’s try to improve the performance of Llama 3 8B (base version) by continuing its pre-training on FineWeb-Edu.

I use Unsloth for this task also because of its better memory efficiency. Note: Unsloth claims that it is 2x faster and 60% more memory-efficient than continued pre-training with Hugging Face Transformers.

My reference code was this notebook by Unsloth.

We need the following packages:

pip install "unsloth[colab-new] @ git+https://github.com/unslothai/unsloth.git"
pip install --no-deps xformers "trl<0.9.0" peft accelerate bitsandbytes

Note: Unsloth doesn’t support yet the last update of TRL. Installing an older version is necessary and can be done with "trl<0.9.0".

The fine-tuning code is as follows:

from transformers import TrainingArguments
from unsloth import is_bfloat16_supported, UnslothTrainer, UnslothTrainingArguments, FastLanguageModel
from datasets import load_dataset
import torch
max_seq_length = 2048 # Choose any. Unsloth support RoPE Scaling internally
dtype = None # None for auto detection. Float16 for Tesla T4, V100, Bfloat16 for Ampere+
load_in_4bit = True # Use 4bit quantization to reduce memory usage. Can be False.

#I load a pre-quantized version of Llama 3, llama-3-8b-bnb-4bit. You can also load the standard version. The only difference is that llama-3-8b-bnb-4bit is faster to download
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/llama-3-8b-bnb-4bit",
    max_seq_length = max_seq_length,
    dtype = dtype,
    load_in_4bit = load_in_4bit,
)


model = FastLanguageModel.get_peft_model(
    model,
    r = 128,
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj",
                      "embed_tokens", "lm_head",], # Add for continual pretraining
    lora_alpha = 32,
    lora_dropout = 0, # Supports any, but = 0 is optimized
    bias = "none",    # Supports any, but = "none" is optimized
    use_gradient_checkpointing = "unsloth", # True or "unsloth" for very long context
    random_state = 3407,
    use_rslora = True,
)


dataset = load_dataset("HuggingFaceFW/fineweb-edu", "sample-10BT", split = "train[:10%]",)

trainer = UnslothTrainer(
    model = model,
    tokenizer = tokenizer,
    train_dataset = dataset,
    dataset_text_field = "text",
    max_seq_length = max_seq_length,
    dataset_num_proc = 8,

    args = UnslothTrainingArguments(
        per_device_train_batch_size = 2,
        gradient_accumulation_steps = 8,
        warmup_ratio = 0.1,
        max_steps = 2000,

        # Select a 2 to 10x smaller learning rate for the embedding matrices!
        learning_rate = 5e-6,
        embedding_learning_rate = 1e-6,

        fp16 = not is_bfloat16_supported(),
        bf16 = is_bfloat16_supported(),
        logging_steps = 10,
        save_steps = 100,
        save_total_limit = 10,
        optim = "adamw_8bit",
        weight_decay = 0.00,
        lr_scheduler_type = "cosine",
        seed = 3407,
        output_dir = "./Llama-3-8B-fineweb-edu-r128a32wd0lstcosinelr5e06-10BT",
    ),
)

trainer_stats = trainer.train()

The differences with a standard QLoRA fine-tuning are in bold in the code above. This is only one set of hyperparameters that I tried. We will see below the learning curves for different hyperparameters.

Some comments:

  • r=128: We want to give enough capacity to the model to learn new knowledge. More trainable parameters are better. That’s why for continued pre-training a high rank, i.e., higher than a rank that we would normally use for LoRA fine-tuning, is preferable.

  • "embed_tokens", "lm_head": Since we are continuing pre-training, we certainly want the model to learn new token embeddings to better model the specific in-domain terms, or words, in training data. Fully fine-tuning the token embeddings and the language modeling head should yield better results than freezing them.

  • use_rslora = True: This activates the use of ranked stabilized LoRA. I put it in bold but in my opinion, this should always be activated, even for a standard fine-tuning.

  • dataset = load_dataset("HuggingFaceFW/fineweb-edu", "sample-10BT", split = "train[:10%]"): We continue pre-training on FineWeb-Edu. I used the smallest sample of it released by Hugging Face which is still very large (10B tokens).

  • embedding_learning_rate = 1e-6: Unsloth recommends using a lower learning rate for the embeddings. I didn’t challenge this recommendation and followed their suggestion. I think it’s intuitive since the embeddings have already been pre-trained on billions of tokens, we probably don’t want to update them too aggressively.

Even though I set max_steps to 2,000, I stopped each training after 1,000 steps since the training loss did not change much.

The learning curves for different learning rates:

Note: For all the configurations, I used an embedding learning rate at least twice as low as the main learning rate. For the configuration with LR=2e-04, I stopped training at 600 epochs since the results were not good enough.

As you can see, we can’t see much difference between each configuration. The training loss remains quite stable for all the configurations. It seems that Llama 3 8B has nothing more to learn from this dataset. This is not too surprising since Llama 3 has probably already seen this dataset several times during its pre-training.

Continue Pre-training Llama 3 on Japanese

Since continued pre-training on FineWeb-Edu didn’t have much effect on Llama 3, what if we continue pre-training on a domain that we know for sure was not well represented in the pre-training data of Llama 3?

For instance, we know that Llama 3 was mainly pre-trained on English data. If we want to fine-tune it for specific tasks in a specific language, it is preferable to first continue pre-training it using a large dataset in this specific language, and then fine-tune it with the limited data that we have for this specific task in this language.

Let’s say that we want to tackle a task in Japanese.

The vocabulary of Llama 3 has all the tokens we need to generate the Japanese language. However, the weights of the model are mainly fine-tuned to generate English rather than Japanese. We can assume that most Japanese tokens were rarely seen during pre-training and that their embeddings are poorly estimated. Consequently, we can expect continued pre-training on Japanese to have much more impact.

To verify it, I used the code in the previous section, simply replacing the training dataset with Japanese Wikipedia.

The learning curve:

The training loss slowly decreases. Llama 3 is learning Japanese.

This adapter should be much easier to fine-tune for Japanese tasks than the original Llama 3.

Conclusion

Continued pre-training is important to prepare an LLM for tasks in a domain or language rarely seen during pre-training.

On the other hand, continued pre-training won’t have much impact if performed on data in a domain already largely mastered by the LLM.

Moreover, since it requires a large amount of data, this continued pre-training can be quite costly. Thanks to QLoRA and Unsloth, we can speed up pre-training while making it more memory-efficient.
