LoRA fine-tunes large language models (LLMs) by adding an adapter on top of the pre-trained LLM, with only this adapter being trainable while the LLM’s original parameters remain frozen. This approach significantly reduces the number of parameters that need to be trained, resulting in much smaller optimizer states. Consequently, LoRA fine-tuning consumes considerably less memory compared to standard full fine-tuning.

Nonetheless, depending on LoRA’s hyperparameters, such as the rank and the number of targeted modules, LoRA may still create very large adapters with hundreds of millions of parameters that are too large to be fine-tuned on consumer hardware.

Many alternatives have been proposed to reduce the size of adapters.

VeRA: Fine-tuning Vectors on top of Random Matrices

VeRA is presented in this paper:

VeRA: Vector-based Random Matrix Adaptation

Note: This is one of my favorite papers presenting an alternative to LoRA. It’s very well-written and motivated. It’s also easy to read even if you don’t have a strong background in machine learning. I recommend reading it.

The method is well-illustrated by Figure 1:

LoRA creates and trains two matrices A and B for each layer of the model. VeRA also has these two matrices but they are frozen, random, and shared across layers. The trainable parameters are in two vectors d and b that are placed after A and B, respectively. d and b are not shared across layers.

The use of random matrices to support the trainable vectors is particularly interesting. The effectiveness of random matrices in machine learning has already been demonstrated before, e.g., by Peng et al. (2021), but using random matrices in the context of parameter-efficient fine-tuning is new.

Since VeRA only trains 2 vectors, VeRA has significantly fewer trainable parameters than LoRA. The difference is even more significant for deeper models as shown by Table 1.

For a model of the size of GPT-3, with a rank of 16, LoRA has 75.5M trainable parameters while VeRA only has 2.8M, i.e., 27x fewer. On the hard drive, this VeRA adapter only occupies 10.5MB. That’s almost nothing.

In Table 4, the authors show that despite training much fewer parameters, VeRA performs close to LoRA.

In most of their experiments, they use a rank of 1,024 for VeRA and empirically show that a higher rank doesn’t yield better results.

Let’s verify how well VeRA performs for Llama 3 8B.

VeRA Fine-tuning for Llama 3 8B

VeRA is implemented in Hugging Face PEFT.

To check which linear modules we can target in the model, we can print the model. Here is an example with Llama 3 8B:

LlamaForCausalLM(

    (embed_tokens): Embedding(128256, 4096)
    (layers): ModuleList(
      (0-31): 32 x LlamaDecoderLayer(
        (self_attn): LlamaFlashAttention2(
          (q_proj): Linear(in_features=4096, out_features=4096, bias=False)
          (k_proj): Linear(in_features=4096, out_features=1024, bias=False)
          (v_proj): Linear(in_features=4096, out_features=1024, bias=False)

        )

          (gate_proj): Linear(in_features=4096, out_features=14336, bias=False)
          (up_proj): Linear(in_features=4096, out_features=14336, bias=False)

  )
  (lm_head): Linear(in_features=4096, out_features=128256, bias=False)
)

All the “Linear“ modules can be targeted by VeRA.

However, while VeRA is supposed to save tiny adapters, this implementation will actually produce a larger adapter than LoRA. This is because PEFT also saves the random matrices in addition to the fine-tuned vectors. Saving these matrices is not mandatory but guarantees the portability of the fine-tuned adapter to other hardware/software configurations. Indeed, since these matrices are random, depending on the configuration (PyTorch version, OS, CUDA, etc.), the same random seed might not yield the same matrices, hence saving the matrices is necessary to make sure the fine-tuned vectors are loaded on top of the same matrices seen during fine-tuning. If you are sure that your adapter will always run on the same machine with a stable configuration, you can disable saving the matrices by setting “save_projection=False” in the VeraConfig (see below).

For VeRA fine-tuning, as a starting point, you can use the code I proposed to fine-tune Llama 3 with QLoRA.

Fine-tune Llama 3 on Your Computer

The main difference is that we can’t quantize the model and we will use a VeraConfig object instead of LoraConfig.

import torch, os, multiprocessing
from datasets import load_dataset
from peft import VeraConfig
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer
)
from trl import SFTTrainer, SFTConfig

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float32
  attn_implementation = 'sdpa'

model_name = "meta-llama/Meta-Llama-3-8B"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_name, use_fast=True)
tokenizer.pad_token = "<|eot_id|>"
tokenizer.pad_token_id = 128009
tokenizer.padding_side = 'left'

ds = load_dataset("timdettmers/openassistant-guanaco")

#Add the EOS token
def process(row):
    row["text"] = row["text"]+"<|end_of_text|>"
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

model = AutoModelForCausalLM.from_pretrained(
          model_name, device_map={"": 0}, torch_dtype=compute_dtype, attn_implementation=attn_implementation
)


print(model)

model.gradient_checkpointing_enable()


peft_config = VeraConfig(
        vera_dropout=0.05,
        r=1024,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ["gate_proj", "up_proj"]
)


training_arguments = SFTConfig(
        output_dir="./drive/MyDrive/Llama3_8b_up-gate_VeRA/",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=2,
        gradient_accumulation_steps=16,
        per_device_eval_batch_size=2,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=100,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=100,
        num_train_epochs=3,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

The important modification is this one:

peft_config = VeraConfig(
        vera_dropout=0.05,
        r=1024,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ["gate_proj", "up_proj"]

)

I arbitrarily chose a rank of 1,024 and a dropout of 0.05. You may obtain a similar performance with a rank of 512. This configuration targets gate_proj and up_proj.

It only yields 983k trainable parameters. The same configuration but targeting k_proj and v_proj yields 131k trainable parameters. If you target all the MLP and self-attention modules, the VeRA adapter will have 1.6M parameters. These are very tiny adapters. To give you a point of comparison, LoRA fine-tuning targetting all the self-attention and MLP modules yields 41M parameters, i.e., 25.6x times larger than VeRA.

Here are the learning curves:

The first obvious observation is that VeRA for Llama 3 8B isn’t nearly as good as LoRA. The learning curves of VeRA don’t go down as fast as for LoRA which means that the VeRA adapters saturate very quickly, probably due to their very small number of training parameters.

Nonetheless, VeRA is learning from the fine-tuning data. It also converges much faster which means that we can stop the training earlier to make it cheaper.

Conclusion

VeRA is a PEFT method with a very small amount of trainable parameters. In my opinion, the main originality of this work is the exploitation of frozen random matrices which works surprisingly well.

Despite its very small number of trainable parameters, VeRA can still learn features from the fine-tuning data. However, it can’t easily match the performance of a LoRA.
