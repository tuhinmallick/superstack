Instruct large language models (LLM), a.k.a., chat models, are base LLMs fine-tuned on instruction datasets. In an instruction dataset, each training example is an instruction paired with a possible answer to this instruction. In other words, each example has two columns, an instruction, and an answer.

Nonetheless, chat models are causal LLMs. They expect a sequence of tokens as input to predict the next token of this input. It means that we have to format each training example so that the columns instruction and answer become a unique sequence of tokens.

The best way to do this is to create and apply a chat template. With a chat template, we can systematically apply the same format to all the training examples and make sure that the same format is used at inference time. Templates are also very handy to make sure that we dont forget any of the special tokens (BOS, EOS, etc.), always include the same system instructions, and dont introduce any unwanted space that would significantly reduce the quality of the output.

In this article, I explain how to create and modify a chat template. We will see how Llama 3s chat template works and apply a new one through fine-tuning. For demonstration, I fine-tune Llama 3 8B with a chat template and a dataset of pirate speak to turn it into a pirate chat model.

I also made a notebook showing this fine-tuning and how to create/modify chat templates for Llama 3. The same method can be applied to any other LLMs:

Using the Chat Template of Llama 3 Instruct

Llama 3 models are available in two versions:

  • Base: meta-llama/Meta-Llama-3-8B

  • Instruct: meta-llama/Meta-Llama-3-8B-Instruct

The base model is not a chat model. It has only been pre-trained on 15T tokens. The instruct version is fine-tuned to answer users queries. Its the base model fine-tuned on an instruction dataset.

This fine-tuning has been performed using a specific format for the training examples, i.e., a chat template. At inference time, we must use the exact same format for the input to get optimal results. If we dont, the model might still work but it may also largely fail to answer specific queries.

Meta included the chat template in the tokenizer configuration. We can see it here:

  • Llama 3 8B instructs tokenizer_config.json

In Hugging Face models, the chat templates are written using the Jinja syntax. The best way to understand it is to apply the template and then print the result.

The chat template expects a JSON object with two fields:

  • role: Who is writing the current message? The assistant (i.e., the model) or the user? Sometimes a third role, system, is also defined to include specific instructions for the LLMs that will not be displayed to the user. This is the case for Llama 3.

  • content: A sequence of tokens.

For instance, Meta provides in the model card this example to begin a conversation with Llama 3:

messages = [
    {"role": "system", "content": "You are a pirate chatbot who always responds in pirate speak!"},
    {"role": "user", "content": "Who are you?"},
]

If we apply the chat template to messages and print the result:

seq = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)

We get:

<|begin_of_text|><|start_header_id|>system<|end_header_id|>

You are a pirate chatbot who always responds in pirate speak!<|eot_id|><|start_header_id|>user<|end_header_id|>

Who are you?<|eot_id|><|start_header_id|>assistant<|end_header_id|>

As we can see, there are many special tokens that we have to carefully place for the model to understand that this is a chat. For instance, <|eot_id|> indicates the end of a dialogue turn. Roles (assistant, user, and system) are marked with the special tokens <|start_header_id|> and <|end_header_id|>.

We must pass add_generation_prompt=True to apply_chat_template, otherwise the tokenizer wont prompt the model to generate the next dialogue turn. Here is an example without it, i.e., using add_generation_prompt=False:

<|begin_of_text|><|start_header_id|>system<|end_header_id|>

You are a pirate chatbot who always responds in pirate speak!<|eot_id|><|start_header_id|>user<|end_header_id|>

Who are you?<|eot_id|>

<|start_header_id|>assistant<|end_header_id|> is missing.

Now, lets add to messages one more turn, including a user query, the answer by the model, and then a new user query:

messages = [

    {"role": "user", "content": "Who are you?"},

    {"role": "user", "content": "Why are you a pirate?"},
]

We get:

<|begin_of_text|><|start_header_id|>system<|end_header_id|>

You are a pirate chatbot who always responds in pirate speak!<|eot_id|><|start_header_id|>user<|end_header_id|>

Who are you?<|eot_id|><|start_header_id|>assistant<|end_header_id|>

Why are you a pirate?<|eot_id|><|start_header_id|>assistant<|end_header_id|>

The model will answer not only given the previous user query but given the entire chat history provided in messages.

With Llama 3 8B instruct and some other instruct LLMs, we have to specify the system prompt to initialize the style and personality of the LLM. For many chat applications, we want the LLM to always behave the same, e.g., following a specific style, having a unique name, etc. In this situation, we dont want the user to set the system prompt.

In the next section, we will see how to fine-tune the LLMs with a specific template, including a system prompt, to ensure that the LLM will use a consistent style across different dialogues with users.

Fine-tuning Llama 3 with a New Chat Template

To fine-tune Llama 3 for chat applications, I recommend using the base model rather than the instruct model. Instruct LLMs are already fine-tuned with a specific template. Fine-tuning them again simply tries to override this first fine-tuning and the resulting model might be confused between the two different fine-tuning it has seen.

For this tutorial, we will turn the base model into a model chatting like a pirate. The code I used for this fine-tuning is similar to the one I used in this article:

The main difference is that we will format the data with a custom chat template.

For this fine-tuning, we want the model to have the personality of a pirate. I named the assistant Barbe Noire and defined a system prompt that will always be included in the chat context:

You are the pirate Barbe Noire. You answer only as Barbe Noire and using pirate speak.

As for the chat template, I use a template similar to Llama 3 8B instruct notably using the same special tokens which, as we will see later in this article, is far from optimal.

tokenizer = AutoTokenizer.from_pretrained(model_name)
assistant_name = "Barbe Noire"
system_prompt = "You are the pirate Barbe Noire. You answer only as Barbe Noire and using pirate speak."
format_system_prompt = "<|start_header_id|>system<|end_header_id|>\n\n"+system_prompt+"<|eot_id|>"
tokenizer.chat_template = "<|begin_of_text|>"+format_system_prompt+"{% set loop_messages = messages %}{% for message in loop_messages %}{% set content = '<|start_header_id|>' + message['role'] + '<|end_header_id|>\n\n'+ message['content'] | trim + '<|eot_id|>' %}{% if loop.index0 == 0 %}{% set content = bos_token + content %}{% endif %}{{ content }}{% endfor %}{% if add_generation_prompt %}{{ '<|start_header_id|>"+assistant_name+"<|end_header_id|>\n\n' }}{% endif %}"

With this template, we remove the need for a system role since the system prompt is always included.

To give a pirate style to our model, we need to fine-tune it on text using this style. I used a synthetic instruction dataset in which the responses were automatically generated by an LLM using a pirate style:

  • Peyton3995/dolly-15k-mistral-pirate (CC-BY)

Then, we apply the chat template to each row of the dataset:

def process(row):
    messages = [
      {"role": "user", "content": row['instruction']},
      {"role": assistant_name, "content": row['response']},
    ]
    row["text"] = tokenizer.apply_chat_template(messages, tokenize=False)+"<|end_of_text|>"
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

I append <|end_of_text|> to each entry for the model to learn when to stop to generate. We will see that this is actually ineffective. The remainder of the fine-tuning code is in the notebook and is standard.

It took almost 12 hours to fine-tune it using the L4 of Google Colab.

I tested the model with this code:

terminators = [

The answer is quite good but the generation should have stopped at the first <|reserved_special_token_104|> which should have been <|eot_id|>.

It seems that the model completely ignores and never generates all the special tokens used by the chat template (<|eot_id|>, <|start_header_id|>, etc.). Instead, the model seems to randomly generate reserved_special_token which are special tokens reserved in Llama 3s vocabulary.

Whats happening?

The problem here is that the embeddings of all these special tokens have never been trained. For instance, for the model, <|reserved_special_token_123|>, <|reserved_special_token_103|> or <|eot_id|> have all the same embeddings. The model cant know which one to use. Note: This is not a bug and is intended by Meta. These tokens have been conveniently introduced by Meta to make it easier for users to add and train their own special tokens. You can see them as placeholder tokens.

Moreover, since I use LoRA, the token embeddings are also not fine-tuned by my code. There are several solutions to correct this problem and make the model generate special tokens when expected:

  • Full fine-tuning, i.e., not using LoRA

  • Use LoRA but retrain the token embeddings and the language modeling head. For this, we only need to add the following line in the LoRAConfig: modules_to_save=["lm_head","embed_tokens"],. However, since Llama 3s vocabulary is large, it adds many trainable parameters and makes fine-tuning much more costly.

  • Replace the special tokens with tokens that are not defined as special in Llama 3s vocabulary. For instance, you can replace <|eot_id|> with <|eot|>. The token <|eot|> will be tokenized into several subtokens from Llama 3s vocabulary. Its not ideal since we wont be able to use it as a stop token for inference but it will at least properly format the output with the expected template. This option is the cheapest since it doesnt change the number of trainable parameters.

Conclusion

Chat templates are very useful to ensure a chat model consistently generates answers in the same format. We can also use templates to inject additional instructions for the model.

I recommend always defining the chat template by yourself instead of using the default one of the model which could be updated and then become incompatible with your models already fine-tuned.
