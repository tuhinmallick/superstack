Gemma 2 are two LLMs of 9B and 27B parameters released by Google. A 2.6B model will be released later. These LLMs have been shown to perform very well in many language generation tasks. For specialized domains and tasks, this performance can be further improved with fine-tuning.

9B and 27B models are particularly suitable for high-end consumer GPUs (24 GB of RAM). The 9B model can be fine-tuned with LoRA, i.e., without quantization while the 27B model requires quantization but still leaves some memory available for batch training and longer sequences.

Gemma 2: A Capped Student Model

Google describes Gemma 2 in this paper:

Gemma 2: Improving Open Language Models at a Practical Size

The models are distributed on the Hugging Face Hub:

  • Gemma 2 models (Gemma license, commercial use allowed)

Gemma 2 Neural Architecture

Gemma 2 uses a neural architecture close to Gemma v1 which is derived from Llama 2’s architecture.

All the models use GeGLU which is, with SwiGLU, one of the best-performing activation functions for LLMs.

They all use grouped-query attention (GQA) to reduce memory consumption and speed up the attention computation. Note that the 27B model has more attention heads but they are smaller than the heads of the other two models (128 against 256 dimensions).

Gemma has an attention-sliding window of 4096 tokens to extend the context length it can process. It’s similar to the attention window of Mistral 7B, except that it is used alternatively, in every other layer, with a global standard attention span of 8192 tokens (the same length as Llama 3).

As for the vocabulary size, it is huge. With 256k vocabulary entries, Gemma 2 has one of the largest vocabularies. This is the same size as for Gemma v1 and the same tokenizer, and it’s 100k entries more than for Llama 3.

It means that Gemma 2 has large embedding parameters. For production, this is important information to keep in mind. These embedding parameters are not quantized by most quantization methods. For instance, for the 9B model, 11% of the parameters are not compressed by quantization methods.

Gemma 2 also uses logit soft-capping which prevents the logits from growing excessively. It limits the logits in each attention layer and the final layer so that their values remain within the range of -soft_cap to +soft_cap where soft_cap is a hyperparameter. Google doesn’t provide information on why they apply this soft-capping and its impact on the model’s performance. Note: When they released the models, the support of FlashAttention was deactivated in Hugging Face Transformers because of this soft-capping. This is not the case anymore and Gemma 2 now fully supports FlashAttention.

A Longer Training

Gemma 2 27B and 9B have been trained on 13T and 9T tokens, respectively, of web documents, code, and science articles. This is significantly more than the 6T tokens used for pre-training Gemma v1.

Gemma 2 2.6B and 9B are the products of knowledge distillation (KD). A larger and better model has been used to train them. It is well-known, notably thanks to the Phi models, that KD performs very well for pre-training.

As for the hardware used for training, Google used its TPUs. I’m not familiar at all with TPUs so I won’t comment much on their hardware configuration. It is detailed in Section 3.3 of the report.

Google also released instruct versions of Gemma 2. They were obtained by post-training the based version with what seems to be a standard RLHF pipeline with a few modifications.

They first ran supervised fine-tuning (SFT) involving behavioral cloning on both synthetic and real prompts, with responses generated by a larger teacher model. As for the RLHF step, it used a similar algorithm to Gemma v1 but employed a reward model significantly larger than the policy, focused on enhancing conversational capabilities, especially in multi-turn interactions.

Finally, model merging was achieved by averaging models from experiments conducted with varying hyperparameters.

Evaluation of Gemma 2

I won’t comment much on the evaluation. As usual, in my opinion, we have too little information on the evaluation settings (data contamination, prompt format, example selections for few-shot learning, …) to interpret the results.

I would just conclude that Gemma 2 9B performs as well as, or slightly better than, Llama 3 8B. While Gemma 2 27B's performance is between Gemma 2 9B and Llama 3 70B.

Fine-tuning Gemma 2 on Consumer Hardware

Gemma 2 are gated models. You can’t access them without:

  • A Hugging Face account

  • A Hugging Face access token

  • Agreeing with Google’s terms for Gemma 2 (registration possible on the model card)

Then, to access the model, you simply need to add these lines in your code:

from huggingface_hub import login
login(token="...")

where you replace “…” with your access token.

Fine-tuning with Hugging Face or Unsloth?

Unsloth is much faster and more memory-efficient than fine-tuning directly with Hugging Face libraries. This is because Unsloth is highly optimized for specific models and environments mainly depending on PyTorch and CUDA versions. It makes Unsloth less flexible and more difficult to install. I don’t recommend it if you are fine-tuning your first LLMs. You would have more bugs than usual. For instance, at the time I wrote this article, LoRA fine-tuning didn’t seem to work with Gemma 2 9B but QLoRA does.

On the other hand, Hugging Face libraries are by default consuming more memory and slower. The code is easier to debug and their installation is straightforward.

Using Hugging Face Transformers

We will need the following packages:

pip install --upgrade bitsandbytes transformers peft accelerate datasets trl

Then, we import what we will need:

import torch, os, multiprocessing
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig
)
from trl import SFTTrainer, SFTConfig

if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

Next, we configure the tokenizer:

tokenizer = AutoTokenizer.from_pretrained(model_name, add_eos_token=True, use_fast=True)
tokenizer.padding_side = 'right'

Gemma 2 has a pad token! This is an extremely rare feature. With most other recent LLMs, such as Llama 3 or Qwen2, we have to sacrifice an existing token to make it a pad token, i.e., we need to find one useless token in the vocabulary which can be difficult, or create a new token from scratch to use as a pad token which requires resizing the embeddings.

With Gemma 2, we don’t need to think about the pad token. However, I recommend setting up the padding side to “right”. When I used “left”, training was unstable.

For the fine-tuning dataset, I use timdettmers/openassistant-guanaco (Apache 2.0 license), an instruction dataset that has the particularity to be multilingual and relatively small. This is ideal for testing/debugging LLMs as one training epoch is relatively cheap.

The remainder of the code is standard QLoRA fine-tuning code. I explained it in detail in this article:

        lora_dropout=0.00,

        eval_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=4,

        log_level="debug",
        save_strategy="epoch",
        logging_steps=20,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=20,
        num_train_epochs=1,

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,

)

trainer.train()

Due to the large vocabulary of Gemma 2, whose token embeddings are not quantized by bitsandbytes, this configuration consumes much more memory than Llama 3 8B: 21.9 GB against 16.6 GB.

The training logs:

QLoRA fine-tuning took 3 hours and 44 minutes to complete one epoch using the L4 GPU of Google Colab.

LoRA is faster than QLoRA since there is no dequantization during training. The main difference is that for LoRA we don’t have to pass a “quantization_configuration” when loading the model. However, since I had already reached the limit of the L4 memory (22.5 GB on Colab), I reduced the batch size to 1 and increased the gradient accumulation steps to 32 to get a training configuration equivalent to the one I had for QLoRA.

The training logs:

It’s 1.6x faster than QLoRA. However, I strangely got a higher training loss after one epoch. I would need to further investigate these results but I assume this has something to do with the token embeddings and language modeling head probably not being cast to float32 when I don’t use QLoRA. Embeddings and language modeling head might have been bfloat16 which is less precise.

Using Unsloth

Unsloth is always a bit tricky to install. This is because it’s highly optimized for specific environments (given PyTorch and CUDA versions). I recommend following the installation instructions provided in the Unsloth repository.

For Google Colab, what works currently is the following:

pip install "unsloth[colab-new] @ git+https://github.com/unslothai/unsloth.git"
pip install --no-deps "xformers<0.0.27" trl peft accelerate bitsandbytes

The fine-tuning code using Unsloth, equivalent (approximately) to the one I used with Hugging Face libraries, is the following:

from unsloth import FastLanguageModel, is_bfloat16_supported
from datasets import load_dataset
from trl import SFTTrainer, SFTConfig
import multiprocessing
import torch
max_seq_length = 512
dtype = None
load_in_4bit = True

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "google/gemma-2-9b",
    max_seq_length = max_seq_length,
    dtype = dtype,
    load_in_4bit = load_in_4bit,
)

model = FastLanguageModel.get_peft_model(
    model,
    r = 16,
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj",],
    lora_alpha = 16,
    lora_dropout = 0, # Supports any, but = 0 is optimized
    bias = "none",    # Supports any, but = "none" is optimized
    use_gradient_checkpointing = "unsloth", # True or "unsloth" for very long context
    random_state = 3407,
)

ds = load_dataset("timdettmers/openassistant-guanaco")

#Add the EOS token
def process(row):
    row["text"] = row["text"]+tokenizer.eos_token
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

training_arguments = SFTConfig(
        output_dir="./gemma-2-9b/r16a16_QLoRA_Unsloth",
        eval_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=4,
        gradient_accumulation_steps=8,
        per_device_eval_batch_size=4,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=20,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=20,
        num_train_epochs=1,
        warmup_ratio=0.1,
        dataset_text_field="text",
        max_seq_length=512,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

It only consumed 12.7 GB of GPU memory and took 2.5 hours to complete one training epoch, i.e., it required 42% less memory and was 1.5x faster than with Hugging Face libraries.

The training and validation losses are almost the same as the ones obtained with Hugging Face Transformers.

Unsloth is faster and more memory-efficient for the same result.

What about LoRA fine-tuning?

When I tried it, using the same configuration as QLoRA but setting load_in_4bit to False, I got this error:

Unsupported: accumulated_cache_size_limit reached

It seems related to torch.compile. I didn’t further investigate.

Conclusion

Gemma 2 is a significant improvement over the first version. We are still waiting for the 2.6B but the 9B and 27B are already small enough to fit on consumer GPUs once quantized. Gemma 2 models are also easy and cheap to fine-tune.

Once fine-tuned, you can merge the adapter into the base model. However, remember that after merging, bitsandbytes quantization poorly performs.
