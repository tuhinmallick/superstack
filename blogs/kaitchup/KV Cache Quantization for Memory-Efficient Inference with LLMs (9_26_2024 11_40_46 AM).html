Quantization reduces the size of a large language model (LLM) by lowering the precision of its parameters, e.g., from 16-bit to 4-bit.

However, quantizing the model only reduces the memory consumption of the model’s parameters. Although loading an LLM on a smaller GPU is possible, the inference process still demands additional memory to store primarily:

  • The activations, i.e., tensors created during the forward pass

  • The KV cache

These two components can both consume much more memory than the model itself depending on the inference hyperparameters and the model’s architecture. Notably, the size of the KV cache rapidly grows with longer contexts and deeper models. This is especially an issue for applications processing long contexts like RAG systems.

What is the KV Cache of a Transformer Model?

In transformer models, text generation is performed token by token, where each new prediction depends on the context of all previously generated tokens.

This sequential dependency can make the process slow, as predicting the next token requires reprocessing the entire sequence generated so far. For instance, to predict the 100th token, the model needs to incorporate information from the preceding 99 tokens, involving complex matrix multiplications across their representations. Predicting the 101st token would require again the same computations performed for the first 99 tokens, plus the additional calculations performed for the 100th token.

The key-value (KV) cache optimizes this process by storing the results of these calculations, allowing the model to reuse them for subsequent tokens without recomputation. It means that to generate the 101st token, instead of recomputing the information for the first 99 tokens, we can retrieve them from the KV cache, and then only compute the missing information for the 100th token.

Specifically, the KV cache stores key-value pairs derived from self-attention layers of previously processed tokens. In the transformer architecture, self-attention layers generate attention scores by multiplying queries with keys and producing weighted sums of value vectors as outputs. By storing these keys and values in a cache, the model can efficiently retrieve them to speed up the generation process.

Exploiting a KV cache is now standard during inference. The speed-up is even more significant as the sequence generated gets longer. However, it also means that the KV cache is growing. We must store all these tensors on the GPU to quickly retrieve and exploit them. For deep models, long sequences, and large batches, the KV can easily occupy dozens of GB of GPU RAM.

Estimating the Memory Consumption of the KV cache

The KV cache usually stores tensors in 16 bits, using the float16 or bfloat16 data types.

For one token, the KV cache stores a pair of tensors, key-value, for each layer and attention head. The size of these tensors is governed by the dimension of the attention head. The total memory consumption of this pair of tensors, in bytes, is:

number of layers * number of KV attention heads * dimension of the attention head * (bit width / 8) * 2

The last “2” is there because we have two tensors, the key and the value. The bit width is 16 most of the time. Since 8 bits are 1 byte, we divide the bit width by 8 which yields 2 bytes for each 16-bit parameter in the KV cache. If we take Llama 3 8B, this equation becomes:

32 * 8 * 128 * 2 * 2 = 131,072

Note: Llama 3 8B has 32 attention heads. However, thanks to grouped-query attention (GQA), only 8 attention heads are used for the keys and values.

The KV cache for one token occupies 131,072 bytes, i.e., 0.1 MB. It seems small but for many different types of applications, LLMs are required to generate thousands of tokens. For instance, if we want to exploit the full context size of Llama 3 8B, which is 8192, the KV cache will store the KV tensors for 8191 tokens. That’s 1.1 GB. For a consumer GPU with 24 GB of RAM, it’s 4.5% of its total memory that will be occupied by the KV cache.

For larger models, the KV cache grows faster. For instance, for Llama 3 70B which has 80 layers, our equation becomes:

80 * 8 * 128 * 2 * 2 = 327,680

For 8191 tokens, the KV cache of Llama 3 70B would occupy 2.7 GB.

Note also that this is the memory consumption for one sequence. If we do batch decoding, we have to multiply this value by the batch size.

For instance, using a batch size of 32 with Llama 3 8B requires 35.2 GB of GPU RAM. This can’t be done with one consumer GPU.

Fortunately, we can now quantize the KV cache.

Quantization of the KV cache for Llama 3

Quantization reduces the precision of the parameters, for instance, from 16 bits to 4 bits. It means that the total size of a 16-bit tensor could be divided by 4, or by 8 with 2-bit quantization.

If we take the last example of memory consumption from the previous section, in theory, 4-bit quantization could reduce the size of the KV cache from 35.2 GB to 8.8 GB, or 4.4 GB with 2-bit quantization. However, in practice, quantization algorithms are not so effective. The reduction of memory consumption depends on the quantization method and its hyperparameters, especially the group size.

With HQQ quantization, you can expect a reduction of memory consumption between 3x and 2x for 4-bit quantization. With a simpler method like block-wise quantization, you can expect a reduction closer to 3x.

Hugging Face Transformers supports KV cache quantization. If you are interested in knowing how Hugging Face has implemented it, have a look at their blog post:

Unlocking Longer Generation with Key-Value Cache Quantization

At the moment, the version that supports it is not available directly with pip. We have to install Transformers from source:

pip install git+https://github.com/huggingface/transformers

It supports HQQ quantization and Quanto (block-wise). HQQ is much more accurate but not as effective as Quanto. HQQ is also much slower than Quanto as reported by Hugging Face:

If decoding speed is not much of a concern for your applications, I recommend using HQQ. Otherwise, use Quanto.

If you want to use HQQ, run:

pip install HQQ

If you prefer Quanto, run:

pip install quanto==0.1.0

Note: Hugging Face recommends using the version 0.1.0 for now.

Here is the code I use to quantize the KV cache of Llama 3 8B Instruct:

import torch
from transformers import AutoTokenizer, AutoModelForCausalLM, set_seed, BitsAndBytesConfig

set_seed(1234)  # For reproducibility

prompt = "The best tomato sauce is"

model_id = "meta-llama/Meta-Llama-3-8B-Instruct"

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=torch.float16,
        bnb_4bit_use_double_quant=True,
)

tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(model_id, quantization_config=bnb_config, attn_implementation="flash_attention_2", torch_dtype=torch.float16, device_map="cuda:0")

inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
outputs = model.generate(**inputs, max_new_tokens=150, cache_implementation="quantized", cache_config={"backend": "HQQ", "nbits": 4, "q_group_size": 128, "residual_length": 64, "device":model.device})
result = tokenizer.decode(outputs[0], skip_special_tokens=True)

print(result)

The line in bold is the one introducing the KV cache quantization. cache_implementation indicates that the KV cache will be quantized during inference. cache_config indicates all the hyperparameters used for the quantization:

  • backend: HQQ or Quanto depending on whether you are more interested in preserving accuracy or inference throughput.

  • nbits: The precision of the quantization. You can set 4 for 4-bits. 2 for 2-bits is also possible but it might severely degrade the accuracy of the LLMs for a slight reduction of memory consumption compared to 4-bit quantization.

  • q_group_size: I would leave it to 128. Reduce it for a more accurate quantization but be aware that the reduction in memory consumption will be less significant.

  • residual_length: Indicates the number of tokens for which the KV cache shouldn’t be quantized. This is critical to preserve the accuracy. Increasing it will improve the accuracy while consuming more memory. 64 is a good value.

  • device: This is not mandatory but I recommend setting it. It should be the same device on which the model is loaded. If you let Transformers manage it automatically, it might not work (e.g., HQQ by default seems to operate on the CPU even when you load the model on the GPU).

If all these hyperparameters are set properly, you shouldn’t see a big difference in the outputs generated with and without KV cache quantization.

As for the difference in memory consumption, remember that you will only notice a significant difference when dealing with large batches and very long sequences.

Conclusion

KV cache quantization is another way to further reduce the memory consumption of LLMs for inference. It can reduce the size of the KV cache by 2x or 3x, depending on the backend and quantization hyperparameters, potentially saving dozens of gigabytes.

However, this quantization is not without drawbacks. It reduces the decoding speed and might significantly reduce the accuracy of the LLMs. Speed and accuracy can be improved by tweaking the quantization hyperparameters.
