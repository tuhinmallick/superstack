With quantization, we can reduce the size of large language models to run them on consumer hardware.

However, quantization is not accurate as it loses information in the process. Typically, large LLMs can be aggressively quantized to lower precision with only little loss of accuracy while smaller LLMs are much more difficult to accurately quantize.

When is it better to use a small LLM rather than quantizing a larger one?

In this article, we will answer this question by experimenting with 8-bit, 4-bit, 3-bit, and 2-bit quantizations, using GPTQ, applied to Mistral 7B, Llama 2 7B, and Llama 13B. We will compare their memory consumption using optimum-benchmark and their accuracy using the LLM Evaluation Harness.

GPTQ Quantization

I have presented GPTQ in previous articles. If you want to know about GPTQ, I recommend reading this section.

Why GPTQ and not AWQ?

AWQ is known to be more accurate and to yield faster models for inference. However, all of the implementations that I have tried fully support only 4-bit quantization.

On the other hand, AutoGPTQ supports 2, 3, 4, and 8-bit quantization with GPTQ and is much more popular, and consequently supported by a greater number of frameworks.

I use AutoGPTQ in Transformers for quantizing LLMs with GPTQ. You can quantize Llama 2 7B, Mistral 7B, and Llama 2 13B with a 16 GB GPU using this code:

from transformers import AutoModelForCausalLM, AutoTokenizer
from optimum.gptq import GPTQQuantizer
import torch
model_path = 'meta-llama/Llama-2-7b-hf'

for w in [2,3,4,8]:
  quant_path = 'Llama-2-7b-hf-gptq-'+str(w)+'bit'

  # Load model and tokenizer
  tokenizer = AutoTokenizer.from_pretrained(model_path, use_fast=True)
  model = AutoModelForCausalLM.from_pretrained(model_path, torch_dtype=torch.float16)
  quantizer = GPTQQuantizer(bits=w, dataset="c4", model_seqlen = 4096)
  quantized_model = quantizer.quantize_model(model, tokenizer)

Note: The code used in my previous article using GPTQ in Transformers doesn’t work well anymore: It’s very slow and no longer recognizes the C4 dataset for GPTQ’s calibration. This new version above, calling “GPTQQuantizer” from Optimum, works as expected.

This code will create 2, 3, 4, and 8-bit GPTQ versions of Llama 2 7B.

I have uploaded all the quantized models to the Hugging Face Hub. You can get them from my GPTQ collection.

Benchmarking Memory-Efficiency and Accuracy

As we will go from 16-bit to 2-bit LLMs, we want to know the following:

  1. How much memory do we save?

  2. How much of the accuracy in downstream tasks do we lose?

To answer the first question, I will use Optimum-benchmark. I presented it in this article:

The Evaluation Harness runs many benchmarks to assess the capabilities of LLMs in downstream tasks. It notably powers the Open LLM Leaderboard.

For benchmarking with Optimum-benchmark, I used this configuration file. 

defaults:
  - backend: pytorch # default backend
  - benchmark: inference # we will monitor the inference
  - launcher: process
  - experiment # inheriting from experiment config
  - _self_ # for hydra 1.1 compatibility
  - override hydra/job_logging: colorlog # colorful logging
  - override hydra/hydra_logging: colorlog # colorful logging

hydra:
  run:
    dir: experiments_ob/${experiment_name} #The results will be reported in this directory. Note that "experiment_name" refers to the configuration field name "experiment_name" below
  sweep:
    dir: experiments_ob/${experiment_name}
  job:
    chdir: true
    env_set: #These are environment variable that you may want to set before running the benchmark
      CUDA_VISIBLE_DEVICES: 0
      CUDA_DEVICE_ORDER: PCI_BUS_ID

experiment_name: Llama-2-13b-hf-16bit
model: meta-llama/Llama-2-13b-hf #The model that we want to evaluate. It can be from the Hugging Face Hub or local directory
device: cuda #Which device to use for the benchmark. We will use CUDA, i.e., the GPU

backend:
  torch_dtype: float16

benchmark:
  memory: true #We will monitor the memory usage
  warmup_runs: 10 #Before the monitoring starts, the inference will be run 10 times for warming up

  new_tokens: 1000 #Inference will generate 1000 tokens
  input_shapes:
    sequence_length: 512 #Prompt will have 512 tokens
    batch_size: 2

For the Evaluation Harness, I ran this command:

lm_eval --model hf --model_args pretrained=meta-llama/Llama-2-13b-hf --tasks winogrande,hellaswag,arc_challenge --device cuda:0 --num_fewshot 5 --batch_size 2 --output_path ./eval_harness/Llama-2-13b-hf-16b

These examples are for the benchmarking of meta-llama/Llama-2-13b-hf. Change the model name at “pretrained=” and “model:” in the command and configuration file above, respectively, to benchmark another model.

The Trade-Off Memory/Accuracy

Let’s have a look at the peak memory usage, i.e., the maximum memory needed by the model to run on the GPU, for Llama 2 7B, 13B, and Mistral 7B using 2, 3, 4, 8, and 16-bit precisions.

Observations:

  • Dividing by 2 the precision doesn’t divide by 2 the peak memory consumption. This is because not all the parameters are quantized and because memory is also consumed by the inference batches (not only for loading the model). For instance, for Llama 2 7B, 16-bit consumes 15.5 GB, 8-bit consumes 9.0 GB, and 4-bit consumes 5.9 GB.

  • Mistral 7B slightly consumes more VRAM for 8-bit (9.5 GB) and 16-bit (16.5 GB) precisions than Llama 2 7B.

What is the impact on downstream tasks?

I looked at the accuracy of the models on 3 tasks with the Evaluation Harness:

  • Winogrande: Benchmark evaluating commonsense reasoning.

  • Arc Challenge: A set of grade-school science questions.

  • HellaSwag: Benchmark evaluating commonsense inference.

These three benchmarks are also used by the Open LLM Leaderboard.

Here are the results:

We can observe similar trends for the three tasks.

For the three models, the accuracy in these tasks doesn’t decrease from 16-bit to 8-bit, and only slightly decreases from 8-bit to 4-bit. It is a good illustration of how good are the 4-bit quantization algorithms today. Remember, GPTQ is not state-of-the-art, AWQ would yield results even closer to 8-bit quantization.

Choosing 4-bit quantization instead of 8-bit and 16-bit for inference is a relatively safe choice. However, I wouldn’t choose 3-bit quantization. As we can see, there is a significant decrease in accuracy for only a little decrease in peak memory consumption. As for 2-bit quantization, for these sizes of LLMs, it makes the models useless.

For now, let’s put aside the results of Mistral 7B and focus on Llama 2 7B vs. 13B. Llama 2 13B quantized to 3-bit appears slightly better than Llama 2 7B quantized to 4-bit.

Should we prefer Llama 2 13B 3-bit to Llama 2 7B 4-bit?

Llama 2 13B 3-bit consumes 8.4 GB of memory against 5.9 GB for Llama 2 7B 4-bit. Using Llama 2 13B 3-bit means that we will use 2.5 GB more than Llama 2 7B 4-bit for only small improvements in downstream tasks. Llama 2 7B 4-bit seems to be a better choice.

If you have 3 more GB of VRAM available, then Llama 2 13B 4-bit would seem like a better option. It consumes 9.5 GB for a much better accuracy.

Nonetheless, for 3, 4, 8, and 16-bit precision, Mistral 7B remains the best choice overall. A Mistral 13B would be great but I don’t think Mistral AI will ever release one as they chose instead the path of the mixture of experts:

Conclusion

After running these experiments, I cannot see any good reasons to use a model quantized to a lower precision than 4-bit. 3-bit and 2-bit quantization are just not good enough yet. Note: With much larger models, e.g., 70B+ parameters, my conclusion might have been different.

As for 8-bit quantization, it needs twice as much memory for an accuracy similar to 4-bit models. It also seems slower (see Appendices below) to generate and with a higher latency. For 7B and 13B models, I can’t see any good use cases for 8-bit quantization.

Also, remember that I’ve only evaluated GPTQ in this article. AWQ is more accurate and faster.
