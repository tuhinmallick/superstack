GPU Benchmarking: What Is the Best GPU for LoRA, QLoRA, and Inference?

You probably don't need $10k+ GPUs

Fine-tuning and running large language models (LLMs) can be quite costly, with GPUs being the primary component driving these expenses. However, finding the most cost-effective GPU for specific tasks is challenging due to the lack of extensive and up-to-date benchmarking.

For instance, we usually dont know which is the most cost-effective GPU for LoRA/QLoRA fine-tuning. The most expensive GPUs are not always the fastest.

Choosing the right GPU is critical to saving both time and money.

In this article, I benchmark 18 NVIDIA GPUs. Results show the training time and cost associated with each GPU for LoRA and QLoRA fine-tuning. Additionally, I have benchmarked inference throughput with and without bitsandbytes 4-bit quantization.

To compute these results, I used Llama 3 8B and mainly relied on the GPUs proposed by RunPod. Note: Here is my referral link if you want to try RunPod while supporting The Kaitchup.

I plan to regularly update these results by adding new LLMs, new settings (e.g., different batch sizes), and new GPUs. Ill notify you in The Weekly Kaitchup when this happens.

Benchmark Settings

Platforms and GPUs

I benchmarked GPUs provided by two different platforms: Google Colab and RunPod. I chose Google Colab because most of my notebooks were developed on Colab and RunPod for its large choice of GPUs. Both platforms have costs among the lowest. Moreover, as we will see, the choice of platform doesnt significantly impact performance. Inference and fine-tuning speed mainly depend on the GPU. CPU, hard drive, network, etc. have almost no impact (unless they are poorly chosen or set up, which is not the case for RunPod and Colab).

As for the GPUs, I only benchmarked NVIDIA GPUs from the Ampere generation or newer. Note: I might add later scores for Mac and AMD hardware.

Here is the list of the 18 GPUs that I benchmarked: H100 PCI-E, A100 SXM, A100 PCI-E, RTX 4090, L40S, RTX 4090 community cloud, RTX 6000 Ada, L40, RTX A6000, A40, RTX A5000, RTX 3090, RTX 4000 Ada, RTX A4500, L4 RunPod, L4 Colab, RTX A4000, RTX 2000 Ada, RTX 4070 Ti, and RTX 4080.

RTX 4090 community cloud, RTX 4080, and RTX 4070 Ti are GPUs from the RunPods community cloud. They are cheaper to run than the other RunPod GPUs that I benchmarked through their secure cloud. L4 Colab is the L4 GPU benchmarked through Google Colab.

QLoRA and LoRA Settings

For each GPU, I measured QLoRA and LoRA training times without changing any hyperparameters and training arguments. This choice has pros and cons. While it allows direct comparisons for the same settings, it ignores the fact that the GPUs can be more or less optimized for particular batch sizes. Moreover, since I fixed the batch size, the GPUs with smaller memory ran out of memory for LoRA fine-tuning (RTX 4070 Ti and RTX 4080) while others with much more memory (e.g., A100 and H100) would have benefited from a much larger batch size. Ill try to add scores for other batch sizes when Ill find the time. For now, consider it as a benchmark for small batch sizes, which is a common setting for LoRA/QLoRA fine-tuning.

Here is the code I used for fine-tuning:

import torch, os, multiprocessing
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig
)
from trl import SFTTrainer, SFTConfig

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'
  
def LoRA(model_id, q=False): #q is whether we do QLoRA (True) or LoRA (False)
  model_name =  model_id.split('/')[1]
  #Tokenizer
  tokenizer = AutoTokenizer.from_pretrained(model_id, use_fast=True)
  tokenizer.pad_token = tokenizer.eos_token
  tokenizer.padding_side = 'left'

  ds = load_dataset("timdettmers/openassistant-guanaco")
  #Add the EOS token
  def process(row):
      row["text"] = row["text"]+"<|end_of_text|>"
      return row

  ds = ds.map(
      process,
      num_proc= multiprocessing.cpu_count(),
      load_from_cache_file=False,
  )
  if q:
    bnb_config = BitsAndBytesConfig(
            load_in_4bit=True,
            bnb_4bit_quant_type="nf4",
            bnb_4bit_compute_dtype=compute_dtype,
            bnb_4bit_use_double_quant=True,
    )
    model = AutoModelForCausalLM.from_pretrained(
              model_id, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
    )
    model = prepare_model_for_kbit_training(model, gradient_checkpointing_kwargs={'use_reentrant':True})
    output_dir = model_name+"_QLoRA"
  else:
    model = AutoModelForCausalLM.from_pretrained(
            model_id, device_map={"": 0}, attn_implementation=attn_implementation, torch_dtype=compute_dtype
    )
    model.gradient_checkpointing_enable(gradient_checkpointing_kwargs={'use_reentrant':True})
    output_dir = model_name+"_LoRA"

  peft_config = LoraConfig(
          lora_alpha=16,
          lora_dropout=0.05,
          r=16,
          bias="none",
          task_type="CAUSAL_LM",
          target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
  )


  training_arguments = SFTConfig(
          output_dir="./"+output_dir,
          optim="adamw_8bit",
          per_device_train_batch_size=2,
          gradient_accumulation_steps=8,
          log_level="debug",
          save_strategy="epoch",
          logging_steps=10,
          learning_rate=1e-4,
          fp16 = not torch.cuda.is_bf16_supported(),
          bf16 = torch.cuda.is_bf16_supported(),
          num_train_epochs=1,
          warmup_ratio=0.1,
          lr_scheduler_type="linear",
          dataset_text_field="text",
          max_seq_length=512,
  )

  trainer = SFTTrainer(
          model=model,
          train_dataset=ds['train'],
          peft_config=peft_config,
          tokenizer=tokenizer,
          args=training_arguments,
  )

  trainer.train()

This is a very standard code for fine-tuning adapters for Llama 3 8B.

The goal here is to find which are the most cost-effective GPUs for this task, i.e., get the training time returned and combine it with the GPU cost to get a cost-effectiveness score. The choice of the dataset used for fine-tuning is arbitrary and shouldnt have an impact on the results. What would have an impact and might change the conclusion of this benchmark are, mainly:

  • max_seq_length: Longer sequences take more time to be processed and consume more memory. High-end professional GPUs (H100, A100, etc.) should be better at handling long sequences than consumer GPUs.

  • per_device_train_batch_size: High-end professional GPUs have much more memory and could be much faster if allowed to process larger batches.

  • optim: I used a quantized version of Adamw (adamw-8bit) to save memory. GPUs might not be all optimized the same for processing 8-bit parameters. H100 GPUs could be faster (?) with the float32 version of Adamw.

Inference Settings

Similarly to fine-tuning, I fixed the settings for inference to be able to compare the GPUs running the same settings.

I measured the inference throughput using optimum-benchmark (Apache 2.0 license). Note: I planned to also use optimum-benchmark for benchmarking QLoRA/LoRA fine-tuning but I postponed it due to the lack of documentation on how to do it.

def inference_bench(model_id, quant=False):
    model_name = model_id.split('/')[1]
    launcher_config = TorchrunConfig(nproc_per_node=1)
    input_shapes = {"batch_size": 1, "num_choices": 1, "sequence_length": 200}

    scenario_config = InferenceConfig(latency=True, memory=True, input_shapes=input_shapes)
    if quant:
      name = "benchmark_inference_report_quant"+model_name+".csv"
      quantization_scheme = 'bnb'
      quantization_config = {
                              "bnb_4bit_compute_dtype": "float16",
                              "bnb_4bit_quant_type": "nf4",
                              "bnb_4bit_use_double_quant": True,
                              "llm_int8_enable_fp32_cpu_offload": False,
                              "llm_int8_has_fp16_weight": False,
                              "llm_int8_threshold": 6.0,
                              "load_in_4bit": True,
                              "load_in_8bit": False,
                            }
      backend_config = PyTorchConfig(model=model_id, quantization_scheme=quantization_scheme, torch_dtype="bfloat16", quantization_config=quantization_config, device="cuda", device_ids="0", no_weights=False)
    else:
      name = "benchmark_inference_report_"+model_name+".csv"
      backend_config = PyTorchConfig(model=model_id, device="cuda", torch_dtype="bfloat16", device_ids="0", no_weights=False)
    benchmark_config = BenchmarkConfig(
        name="pytorch_"+model_name,
        scenario=scenario_config,
        launcher=launcher_config,
        backend=backend_config,
    )
    benchmark_report = Benchmark.launch(benchmark_config)


    benchmark_report.log() 
    benchmark_config.to_dict() 
    benchmark_report.save_csv(name)

batch_size and sequence_length are also important hyperparameters that have an impact on inference performance. We can increase both with more GPU memory. For this benchmark, I used a batch size of 1 and a sequence length of 200 tokens.

The Benchmark Results

LoRA Fine-tuning

For LoRA fine-tuning, the fastest GPUs are, as expected, the H100 and A100. Remember that they could probably be much faster than that with larger training batches as their memory is much larger. However, note that training LoRA with large batches is rather unusual and might lead to overfitting.

The RTX 4090, a consumer GPU with 24 GB of VRAM, appears as a very good alternative to professional GPUs. For instance, it is faster than the L40S which is currently (July 15th, 2024) $5000 more expensive than the RTX 4090. Having a more expensive GPU doesnt guarantee that you will get faster results.

At the bottom of the chart, we can see the L4 GPUs which are the slowest for this fine-tuning. Despite using a very different hardware configuration (network, CPUs, RAM, etc.) L4 Colab and L4 RunPod perform almost the same. The RTX 4090 from RunPods community cloud performs also similarly to the RTX 4090 from RunPods secure cloud despite using very different hardware configurations. These results demonstrate that, with a reasonable hardware configuration, the GPU is the most critical factor influencing performance.

With this second chart, I wanted to answer which GPU achieved this fine-tuning for the lowest cost.

All the GPUs are priced per hour. To compute their cost-effectiveness, I divided the training time, in seconds, by 3600 and then multiplied the result by their cost ($/hour).

The RTX 4090 is the most cost-effective, by far. If you want to fine-tune an LLM with LoRA and if you dont need more than 24 GB of GPU RAM, choose the RTX 4090 to reduce cost. Since the RTX 4090 is also among the fastest that I have tested, you will also get your results faster.

The A100 and H100 are not cost-effective for LoRA fine-tuning with small batch sizes.

QLoRA Fine-tuning

For QLoRA fine-tuning, the main difference is that the model is quantized to 4-bit with the NormalFloat4 data type.

4-bit quantization with bitsandbytes is poorly supported by GPUs. All the GPUs tested were at least 50% slower than for LoRA fine-tuning. If you have enough memory to avoid quantization, dont use QLoRA fine-tuning. Your fine-tuning would be faster and better.

In the bar chart above, we have a ranking very similar to the one we had for LoRA. This time, thanks to quantization, the GPUs with less VRAM could complete this fine-tuning. Interestingly, the RTX 4070 Ti, which costs less than $1000, performs better than the L4 which is more experience.

Moreover, the RTX 4070 Ti is more cost-effective than the RTX 4090 from the community cloud. It has also less memory (12 GB vs. 24 GB). If your fine-tuning doesnt need much memory, choosing a middle-end GPU such as the RTX 4070 Ti will reduce the fine-tuning cost while still being able to complete the fine-tuning within a reasonable time.

Inference Throughput without Quantization

The inference throughput has been benchmarked for a batch size of one. The results show which is the fastest GPU for processing a single sequence at a time.

I benchmarked two stages of inference: prefill and decode. The prefill stage encodes the prompt and fills the KV cache while the decode stage computes the probabilities, samples, and then generates the next token. Prefill is much faster than decode.

The throughput is given in tokens/second, i.e., how many tokens are processed by the prefill and decode stages during one second.

The RTX 6000 Ada is the fastest for prefill. The RTX 4090 and A100 follow. Interestingly, the H100 is almost 20% slower at this task than the RTX 6000 Ada.

For the decode stage, the RTX 4090 is significantly faster than all the other GPUs by generating 45 tokens/seconds while the H100 only generates 27 tokens/seconds.

Again, we can see that the other hardware components (CPUs, disks, etc.) dont have much impact on the inference throughput since the RTX 4090 community cloud performs similarly to the RTX 4090 from RunPods secure cloud.

Inference Throughput with 4-bit Quantization (bitsandbytes)

When 4-bit quantization is involved, the dequantization of the parameters is necessary to compute the activations (i.e., all the tensors created during inference) and preserve the models accuracy. This dequantization requires additional operation which slows down inference. Both stages, prefill and decode, are slower.

The RTX 4090 is again the fastest GPU.

High-end professional GPUs such as the A100 and H100 process almost 1000 tokens less per second compared to the RTX 4090.

For the decode stage with a quantized model, the difference is even larger:

The RTX 4090 is 2.4x faster than the A100. All the other GPUs generate between 15 and 25 tokens/second while the RTX 4090 generates 45 tokens/second.

In other words, quantized or not quantized, the RTX 4090 is the best choice if your model can fit on 24 GB of VRAM and if you dont need batch inference.

Conclusion

Fine-tuning and running LLMs is a cost-intensive process, primarily driven by GPU usage. This benchmarking study shows the importance of choosing the right GPU for specific tasks to optimize both time and cost. While the most expensive GPUs like the A100 and H100 are not necessarily the fastest, consumer GPUs such as the RTX 4090 offer a highly cost-effective alternative for both fine-tuning and inference.

Why?

If we compare the specifications of the RTX 4090 against, for instance, the H100, we can see that the RTX 4090 has a higher clock speed, more CUDA cores, and slightly more tensor cores.

However, the expensive HBM memory of the H100 has a much higher bandwidth than the RTX 4090s GDDR6x memory. If you increase the batch size, for inference or fine-tuning, the effectiveness of the more expensive GPUs, with better memory, will significantly increase. Moreover, for LLMs with more than 30B parameters, the RTX 4090 may not have enough memory for fine-tuning and inference, with standard hyperparameters, even if the model is quantized. There would be no other alternative than using slower GPUs equipped with more memory.
