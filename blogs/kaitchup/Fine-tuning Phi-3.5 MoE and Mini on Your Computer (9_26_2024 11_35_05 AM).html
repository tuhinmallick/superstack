Microsoft released Phi-3.5. For now, it includes a new Mini version, a mixture of experts (MoE), and a vision language model (VLM):

  • microsoft/Phi-3.5-mini-instruct

  • microsoft/Phi-3.5-MoE-instruct

  • microsoft/Phi-3.5-vision-instruct

They are all available with an MIT license.

We don’t know much about the model yet. Phi-3.5 Mini seems to outperform the previous version, especially in multilingual tasks. The architecture of the model remains the same.

Phi-3.5 MoE is a mixture of 16 Phi-3.5 Mini activating 2 of them during inference. The model has 41.9B parameters. 6.6B parameters are active during inference. According to the public benchmarks, it is better than Gemma 2 9B and Llama 3.1 8B.

QLoRA and LoRA with Phi-3.5 MoE are not possible on consumer hardware. I provide the fine-tuning code but you will need at least a 32 GB GPU.

Phi-3.5 Mini and Phi-3.5 MoE GPU Requirements

Phi-3.5 Mini Requirements

Phi-3.5 Mini requires a 12 GB GPU, or an 8 GB GPU once quantized.

It’s a 3.8B parameter model. Each parameter is a bfloat16 which consumes 2 bytes of memory. Loading Phi-3.5 Mini requires a GPU with 3.8B * 2 bytes of memory, i.e., 7.6 GB. An 8 GB GPU is enough to load the model, but it’s not enough to run it or fine-tune it. For this, to store the optimizer states and gradients (fine-tuning) and the activations (inference and fine-tuning), you will need at least a 12 GB GPU.

With bitsandbytes and AutoRound, we will see that we can reduce the size of the model to 2.28 GB thanks to 4-bit quantization. Once quantized, fine-tuning and inference become possible on an 8 GB GPU.

Phi-3.5 MoE Requirements

Unfortunately, the MoE version is too large to fit on a consumer GPU. While the model is an MoE with a small number of parameters active during inference, we still need to load the entire model. That’s 41.9B parameters, i.e., 83.8 GB.

You will need 100 GB of GPU RAM to fine-tune the model with LoRA, e.g., 3 GPUs with 48 GB of VRAM such as an L40.

Even quantized to 4-bit with bitsandbytes, the model still consumes 27.8 GB during fine-tuning with a minimal batch size. A 32 GB GPU would work.

Quantization for Phi-3.5 Mini and MoE

Quantize Phi-3.5 MoE and Mini with bistandbytes

bitsandbytes quantizes the model at loading time. Both Phi-3.5 MoE and Mini are supported by bitsandbytes. You can do it as follows:

import torch
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig,
)

#use bf16 if supported
if torch.cuda.is_bf16_supported():
  compute_dtype = torch.bfloat16
else:
  compute_dtype = torch.float16

#replace by "microsoft/Phi-3.5-MoE-instruct" for Phi-3.5 MoE
model_name = "microsoft/Phi-3.5-Mini-instruct" 

tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True, use_fast=True)

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, trust_remote_code=True, quantization_config=bnb_config, device_map={"": 0}
)

Quantize Phi-3.5 MoE and Mini with AutoRound

AutoRound is more accurate than bitsandbytes. It also makes the model faster for fine-tuning and inference.

Quantizing Phi-3.5 Mini with AutoRound using Google Colab’s L4 GPU takes less than 2 hours. It should be much faster on an RTX 3090/4090.

We need to install:

pip install --upgrade transformers auto-round flash_attn optimum auto-gptq

Then, to run the quantization:

from transformers import AutoModelForCausalLM, AutoTokenizer
import torch
model_name = "microsoft/Phi-3.5-Mini-instruct"
model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype=torch.float16, trust_remote_code=True)
tokenizer = AutoTokenizer.from_pretrained(model_name)

from auto_round import AutoRound

bits, group_size, sym = 4, 128, True
autoround = AutoRound(model, tokenizer, bits=bits, group_size=group_size, batch_size=2, seqlen=512, sym=sym, gradient_accumulate_steps=4, device='cuda')
autoround.quantize()
output_dir = "./AutoRound/"
autoround.save_quantized(output_dir)

For Phi-3.5 MoE, quantization with AutoRound (and GPTQ) doesn’t seem to work yet. Note: I tried. For the hardware, it requires enough CPU RAM to load the model, i.e., more than 90 GB. Then, it requires a 40 GB GPU for the optimization. The quantization itself took 10 hours on RunPod’s L40 GPU (for almost $1/hour) (referral link). It seems to work. The issue is that the serialization fails. The GPTQ format doesn’t seem to support the model. I couldn’t try other formats (AWQ is also supported by AutoRound) since the model is discarded from the memory after the (failed) serialization step.

Evaluation of Phi-3.5 Mini Quantized

Since Phi-3.5 Mini is a small model, we can’t assume that 4-bit quantization will be accurate. We must evaluate the models to check whether they are significantly degraded.

I used the Evaluation Harness to evaluate Phi-3.5 Mini quantized with AutoRound and Bitsandbytes and compared the results with the original model, on these three benchmarks: MMLU, MMLU-Pro, and Arc Challenge.

The accuracy of the quantized models is good. Phi-3.5 Mini quantized with AutoRound only loses 2 points of accuracy on MMLU and MMLU-PRO. The AutoRound version yields better results than bitsandbytes.

Phi-3.5 MoE QLoRA Fine-tuning

As we saw in the previous section, we need at least a 32 GB GPU to fine-tune Phi-3.5 MoE quantized. I used RunPod’s L40. One epoch on openassistant-guanaco requires almost 11 hours. This is 10x times slower than the LoRA fine-tuning of Phi-3.5 Mini.

The fine-tuning code itself is standard but since the architecture of the model is different, we have to modify the LoraConfig. To know what are the modules that we can target with LoRA, we can print the model:

PhiMoEForCausalLM(
  (model): PhiMoEModel(
    (embed_tokens): Embedding(32064, 4096)
    (layers): ModuleList(
      (0-31): 32 x PhiMoEDecoderLayer(
        (self_attn): PhiMoEFlashAttention2(
          (q_proj): Linear4bit(in_features=4096, out_features=4096, bias=True)
          (k_proj): Linear4bit(in_features=4096, out_features=1024, bias=True)
          (v_proj): Linear4bit(in_features=4096, out_features=1024, bias=True)
          (o_proj): Linear4bit(in_features=4096, out_features=4096, bias=True)
          (rotary_emb): Phi3LongRoPEScaledRotaryEmbedding()
        )
        (block_sparse_moe): PhiMoESparseMoeBlock(
          (gate): Linear4bit(in_features=4096, out_features=16, bias=False)
          (experts): ModuleList(
            (0-15): 16 x PhiMoEBlockSparseTop2MLP(
              (w1): Linear4bit(in_features=4096, out_features=6400, bias=False)
              (w2): Linear4bit(in_features=6400, out_features=4096, bias=False)
              (w3): Linear4bit(in_features=4096, out_features=6400, bias=False)
              (act_fn): SiLU()
            )
          )
        )
        (input_layernorm): LayerNorm((4096,), eps=1e-05, elementwise_affine=True)
        (post_attention_layernorm): LayerNorm((4096,), eps=1e-05, elementwise_affine=True)
      )
    )
    (norm): LayerNorm((4096,), eps=1e-05, elementwise_affine=True)
  )
  (lm_head): Linear(in_features=4096, out_features=32064, bias=True)
)

All the “Linear” layers are good candidates. Here is my LoraConfig:

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj","gate","w1","w2","w3"]
)

I targeted all the experts but this is quite costly in memory as it creates 273.6M trainable parameters. If you want to reduce the memory consumption, you may remove w1, w2, and w3 from the target modules but keep the “gate” which is responsible for choosing the expert during inference.

Conclusion

The Phi-3.5 Mini demonstrates notable improvements in multilingual tasks compared to its predecessor, making it a good choice for use with mid-range GPUs. On the other hand, the MoE version presents challenges for consumer hardware deployment due to its substantial computational demands, even after quantization.

We saw that AutoRound is a good alternative to bitsandbytes for QLoRA fine-tuning, even with smaller models like the Phi-3.5 Mini.

I will update this article once I have confirmed a successful quantization of the MoE version using AutoRound.
