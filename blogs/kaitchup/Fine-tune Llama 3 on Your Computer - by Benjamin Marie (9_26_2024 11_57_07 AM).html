Llama 3 is currently available in two versions: 8B and 70B. The 8B version, which has 8.03 billion parameters, is small enough to run locally on consumer hardware.

With parameter-efficient fine-tuning (PEFT) methods such as LoRA, we don’t need to fully fine-tune the model but instead can fine-tune an adapter on top of it. To further decrease memory consumption, we can even apply this method on top of a quantized Llama 3 with QLoRA.

An Overview of Llama 3’s Architecture and Performance

Llama 3 is available on the Hugging Face Hub:

  • meta-llama/Meta-Llama-3-8B

  • meta-llama/Meta-Llama-3-70B

Note: A third version with more than 400B parameters is still under training (as of the 21st of April, 2024).

I will mainly write about the 8B version in this article since only this one can be fine-tuned on a single consumer GPU. You would need at least two high-end GPUs to fine-tune the 70B version, quantized.

Llama 3 has the same architecture as Llama 2 but a much larger vocabulary which contains 128k entries against 32k entries for Llama 2. According to Meta this vocabulary extension significantly improved the model. I also assume that it makes the models much better at dealing with other languages since it can compose words with fewer tokens (i.e., lower fertility). The pre-training data of Llama 3 contains 5% of high-quality non-English data. Note: Meta still mentioned on the model cards that Llama 3 is intended to be used for English tasks.

On the other hand, an extension of the vocabulary means that the token embeddings require more data to be accurately estimated. Meta trained Llama 3 on 15T tokens. In contrast, Llama 2 was only trained on 2T tokens. To the best of my knowledge, Llama 3 is the first model to be trained on so many tokens. Google Gemma for comparison was trained on 6T tokens which already seemed a lot at that time, a couple of months ago.

Fine-tune Llama 3 8B on Your GPU

Note: I wrote the following tutorial for Llama 3 8B but you can also apply it to the 70B version. It works the same.

Memory Requirements to Fine-tune Llama 3B

Fully Fine-tuning an LLM requires updating all its parameters to adapt it to a new task. This fine-tuning consumes a lot of memory.

The model needs to be entirely loaded into the GPU memory. Moreover, AdamW, the optimizer usually employed to fine-tune LLMs, creates and stores in the GPU memory 2 parameters for each parameter in the model. Tensors created during fine-tuning, i.e., the activations, must all be stored for all layers to compute the gradients that determine how to update the model’s parameters during backpropagation.

To sum up, fine-tuning Llama 3 8B with, for instance, a batch size of 8 and a sequence length of 512, would consume:

That’s a lot of memory! We are far from an affordable configuration. One 80 GB GPU wouldn’t be enough. Note: This memory consumption is an estimation without considering any optimizations, such as gradient checkpointing and tensor parallelism. With an optimized framework, we can significantly reduce it.

Fortunately, we can easily reduce the memory consumption of all these three components:

  • Optimizer states: By default, AdamW’s parameters are float32 which consumes 4 bytes each. AdamW-8bit is a good alternative that quantizes the parameters to 8 bits, i.e., it reduces the memory consumption from 59.8 GB to 15 GB. Note: If you use a framework that doesn’t copy the model’s parameters, it would be significantly less.

  • Model: We can quantize the model to 4-bit. It divides the memory consumption by nearly 4, i.e., from 15 GB to 4 GB. Note: In practice, not all the modules of the LLM are quantized to preserve its performance.

  • Activations: We need to store the activations to compute the gradients. However, with gradient checkpointing, we can recompute them on the fly during backpropagation instead of keeping all of them during the entire training step. It greatly reduces the memory consumption of the activations from 54 GB to 10 GB.

After applying all these optimizations, fine-tuning consumes 29 GB. This is still too much but at least now it is possible to fine-tune the model on two 24 GB GPUs.

We can further reduce it with LoRA. LoRA freezes the model and fine-tunes an adapter on top of it. It significantly reduces memory consumption since the optimizer only has to create new parameters for the adapter's parameters. As these adapter’s parameters are rarely over 100 million for 8B models, the memory consumption of the optimizer states for LoRA is often negligible.

With the LoRA configuration detailed in the next section, the adapter has only 42M parameters, i.e., the optimizer states for these parameters consume less than 200 MB of memory with AdamW-8bit.

With LoRA and quantized Llama 3 8B, we can fine-tune the Llama 3 8B with a 16 GB GPU.

QLoRA for Llama 3 8B

QLoRA is LoRA with quantized LLMs. To fine-tune Llama 3 8B with this method, we need to install:

pip install --upgrade bitsandbytes transformers peft accelerate datasets trl

Then, import:

import torch, os
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig
)
from trl import SFTTrainer, SFTConfig

If you have a recent GPU, I recommend using the bfloat16 data type for better training stability, and FlashAttention to reduce the memory consumption for processing long sequences. The following piece of code automatically detects if your GPU is compatible:

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

Then, we need to initialize and configure the tokenizer. Usually, LLMs are pre-trained without a pad token. However, for fine-tuning, since not all our training examples have the same length, we need to pad them to fill the batches. Llama 2 has a UNK token that I like to use for padding. Llama 3 doesn’t have this token.

We could create and add a PAD token to the vocabulary but a simpler alternative is just to assign the EOS token as a pad token. 

model_name = "meta-llama/Meta-Llama-3-8B"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_name, add_eos_token=True, use_fast=True)
tokenizer.pad_token = "<|eot_id|>"
tokenizer.pad_token_id = 128009
tokenizer.padding_side = 'left'

We need to add the EOS token at the end of each training example for the model to learn when to stop generating. Usually, this can be easily done by passing “add_eos_token=True” to “AutoTokenizer.from_pretrained”. However, as I’m writing this article, “add_eos_token” is ignored and doesn’t add the EOS token. We have to add the EOS token by hand, once the training data are loaded (see the next code block).

Note that we pad to the left of the examples. Padding to the right isn’t compatible with FlashAttention if you want to use it.

As for the fine-tuning dataset, I chose timdettmers/openassistant-guanaco which I find small and good enough for this tutorial.

ds = load_dataset("timdettmers/openassistant-guanaco")

#Add the EOS token
def process(row):
    row["text"] = row["text"]+"<|end_of_text|>"
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

Then, we create the quantization configuration and load the model:

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
)

This configuration defines that we load the model in 4-bit with the quantization of the quantization constants (i.e., double quantization) and that the forward pass will create bfloat16 tensors if your GPU supports this data type. Note: If your GPU doesn’t support bfloat16, the notebook will use float16. However, it might make the training unstable. If you observe that the training loss falls to 0 or NaN, change your compute_dtype to torch.float32.

To reduce the memory consumption of the activations, we also need to enable gradient checkpointing. This is done with:

model = prepare_model_for_kbit_training(model)

For the configuration of LoRA, I use:

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

You may want to increase the rank to obtain better results. Increasing the rank will also increase the memory consumption as it adds parameters to the adapter.

Next, we define the training arguments and hyperparameters:

training_arguments = SFTConfig(
        output_dir="./Llama3_8b_QLoRA",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=8,
        gradient_accumulation_steps=4,
        per_device_eval_batch_size=8,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=100,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=100,
        num_train_epochs=3,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
)

I use paged AdamW 8-bit which stores some of the optimizer states to the CPU RAM when needed to further decrease the GPU memory consumption. If you have enough memory, use adamw_8bit instead to speed up fine-tuning.

For the batch size, I arbitrarily chose a batch size of 32 with a batch size per device of 8 and gradient accumulation steps of 4 (8x4=32). This configuration consumes 16.6 GB of GPU RAM. If you only have a 16 GB GPU, reduce the batch size per device to 4.

I didn’t try other values for the learning rate than 1e-4.

Finally, to start fine-tuning, run:

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

It should take nearly 10 hours to complete the 3 epochs with a recent GPU. I used the L4 instance of Google Colab.

Merging a QLoRA Adapter for Llama 3

To avoid loading the adapter every time you want to use it, you can merge it into Llama 3. This merging, when the adapter has been fine-tuned with QLoRA, has to be carefully done to preserve most of the accuracy of the adapter. We have to follow this procedure:

  • Load and quantize Llama 3

  • Dequantize Llama 3 to the compute dtype used during QLoRA fine-tuning

  • Merge the adapter into the dequantized model

  • Save the resulting model

Quantize Llama 3 with AWQ

AWQ is a quantization scheme that preserves the important weights of the model. It is accurate and is also supported by efficient kernels for inference. I explain AWQ in this article:

To use it, first install AutoAWQ:

pip install autoawq

Then, the quantization is performed with a few lines of code. For instance, to quantize the model obtained after the merge in the previous section:

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

tokenizer_path = "meta-llama/Meta-Llama-3-8B"
model_path = './dqz_merge/'
quant_path = 'llama-3-oasstguanaco3e-awq-4bit'
quant_config = { "zero_point": True, "q_group_size": 128, "w_bit": 4, "version": "GEMM" }

# Load model and tokenizer
model = AutoAWQForCausalLM.from_pretrained(model_path, safetensors=True)
tokenizer = AutoTokenizer.from_pretrained(tokenizer_path, use_fast=True)

# Quantize
model.quantize(tokenizer, quant_config=quant_config)

# Save quantized model with safetensors
model.save_quantized("./"+quant_path, safetensors=True)
tokenizer.save_pretrained("./"+quant_path)

This will save the quantized model in a directory named “llama-3-oasstguanaco3e-awq-4bit”.

Conclusion

We saw how to fine-tune Llama 3 with QLoRA. It only requires a GPU with 16 GB of memory. If you have a 24 GB GPU, you don’t even need to quantize Llama 3. In other words, a standard LoRA would work. The absence of quantization would also double the speed of fine-tuning.

QLoRA and LoRA only fine-tune adapters. If you really want to fine-tune the full model, you might want to try GaLore. GaLore projects the gradients to low-rank subspaces to significantly reduce their dimensions and, consequently, their memory consumption. While GaLore greatly reduces the memory requirements of the optimizer states, you will still need 48 GB of GPU RAM (or 30 GB when layerwise GaLore will be fixed).
