When a new large language model (LLM) is released, it is now common to see it submitted and ranked on the OpenLLM leaderboard. An LLM ranked first on this leaderboard has a good chance to be “trending” on the Hugging Face hub for the following weeks.

In the background, the OpenLLM leaderboard is simply running the Evaluation Harness, an open-source framework by EleutherAI that can evaluate LLMs on many public benchmarks. MMLU, HellaSwag, TruthfulQA, Lambada, etc. are all available in the Evaluation Harness.

In this article, I present the Evaluation Harness. In particular, we will see how we can use it to check locally the performance of quantized LLMs and LoRA adapters. If the LLM you want to evaluate runs on your computer, you can also easily evaluate it with the Evaluation Harness.

The Evaluation Harness Framework

The Evaluation Harness is a framework designed to evaluate the performance of generative language models across a diverse range of tasks. It encompasses over 60 standard academic benchmarks specifically tailored for LLMs and features multiple subtasks.

The framework is built to be versatile, supporting models loaded through various platforms such as the Hugging Face Hub.

This framework is also particularly efficient. There are several features to exploit as well as possible the computational resources available. For instance, there is an option “auto” for the batch size that automatically detects and uses the maximum batch size your GPU can support without running out of memory.

For even faster inference, the framework can also use vLLM.

Additionally, it accommodates the evaluation of quantized LLMs and adapters like LoRA, as supported in HuggingFace's PEFT library. As we will see in the next section, their evaluation is very straightforward.

I won’t discuss it in this article, but it also supports custom prompts and evaluation metrics. It’s also possible to design your own evaluation configuration.

Run the Evaluation Harness on Your Computer

In this article, since we are doing AI on a budget, we will focus on evaluating quantized LLMs and LoRA adapters using consumer hardware.

Requirements

I have only tested evaluation with a GPU. If you can load the model you want to evaluate on your GPU, the framework can also run on the same GPU.

If some of the VRAM of your GPU remains available, you can use it to increase the batch size and speed up the evaluation.

Note that, while the framework can automatically detect the maximum batch size given the VRAM available, I found that this functionality is not 100% accurate. It can trigger out-of-memory errors. This is particularly frustrating when it happens after several hours of evaluation.

To prevent this, I recommend running it first with the “--batch_size auto”, then checking in the output of the framework what is the maximum batch_size detected. It should be indicated as “Determined largest batch size:”.

Slightly reduce the value reported and then rerun the framework with this value for batch_size. For instance, if the maximum batch_size detected is 8, rerun the framework with “--batch_size 6”.

As for the software requirements, we can install the framework and all its dependencies with:

pip install git+https://github.com/EleutherAI/lm-evaluation-harness.git

This will install the command “lm-eval”.

Then, if you want to evaluate LLMs quantized with bitsandbytes, install:

pip install bitsandbytes

If you want to evaluate models quantized with GPTQ, install:

pip install auto-gptq optimum

Evaluate Quantized LLMs

Before evaluating quantized LLMs, let’s see how to evaluate a standard LLM available on the Hugging Face hub, for instance, Llama 2 7B. Note: You must have registered your Hugging Face access token since Llama 2 is not accessible without it. If you don’t know how to do it, check the notebook (#33).

lm_eval --model hf \
    --model_args pretrained=meta-llama/Llama-2-7b-hf,dtype="float16" \
    --tasks truthfulqa,hellaswag,winogrande \
    --device cuda:0 \
    --batch_size 6\
    --output_path ./eval_llama2_7b \
    --log_samples

The arguments are the following:

  • model: “hf” means that the model to evaluate is from the Hugging Face Hub. It can be in a public or private repository.

  • model_args: Several arguments specifying how to load the model

    • pretrained: The name of the model repository on the HF hub

    • dtype: Which data type to use for loading the model. The default is float32, which is too high for most consumer GPUs to load Llama 2 7B. float16 makes the model twice smaller without degradation.

  • tasks: Benchmarks to use for the evaluation. The complete list of the available benchmarks and their identifier is given by the command “lm-eval --tasks list”.

  • device: The device on which the model should be placed. “cuda:0” means the GPU with the id 0.

  • batch_size: The batch size used for inference. Set it to “auto” if you trust that the framework can automatically detect the right maximum batch size.

  • output_path: It will create a directory with this name storing the evaluation results.

  • log_samples: Save the samples generated for the evaluation in the output directory. This is particularly useful for analysis but keep in mind that it may create huge files.

This evaluation will consume up to 15.5 GB of VRAM. I used the V100 GPU of Google Colab to run it. It can run on any recent GPU with 16 GB of VRAM (e.g., an NVIDIA RTX 4060 or 4080). It took 42 minutes (inference + running the evaluation metrics).

The command produces a JSON file containing all the results. It also prints it in the output:

For the number of “shots”, I used the default value which is 0. That’s why the numbers you see in the screenshot above are slightly lower than the ones displayed on the OpenLLM Leaderboard for Llama 2 7B. You can change the number of shots with the argument “--num_fewshot”.

Now, if we want to evaluate the model quantized to 4-bit using bitsandbytes NF4, we simply have to add “,load_in_4bit=True" in the “model_args”, as follows:

lm_eval --model hf \
    --model_args pretrained=meta-llama/Llama-2-7b-hf,load_in_4bit=True,dtype="float16" \
    --tasks truthfulqa,hellaswag,winogrande \
    --device cuda:0 \
    --batch_size 14\
    --output_path ./eval_llama2_7b_nf4 \
    --log_samples

It yields the following results:

Since quantized models are slower for inference, this takes around 2 hours to complete.

Despite the quantization, the performance is only slightly worse for hellaswag and winogrande while it’s slightly better for truthfulqa. It’s good proof of how accurate is the NF4 quantization.

If you want to evaluate a model quantized with GPTQ, there are no particular arguments. The GPTQ quantization will be automatically detected:

lm_eval --model hf \
    --model_args pretrained=kaitchup/Llama-2-7b-gptq-4bit\
    --tasks truthfulqa,hellaswag,winogrande \
    --device cuda:0 \
    --batch_size 16 \
    --output_path ./eval_llama2_7b_gptq \
    --log_samples

Inference with GPTQ models is much faster than with bitsandbytes NF4 models.

Evaluate LoRA Adapters

Evaluating a model with a loaded LoRA adapter is also very simple. We only need to add “,peft=” to “model_args” followed by the name of the adapter on the Hugging Face hub. The adapter can be hosted in a private repository.

With the following command, I evaluate Llama 2 7B with an adapter quickly fine-tuned on the oasst-guanaco dataset. The model is also quantized with bitsandbytes NF4.

lm_eval --model hf \
    --model_args pretrained=meta-llama/Llama-2-7b-hf,load_in_4bit=True,peft=kaitchup/Llama-2-7B-oasstguanaco-adapter,dtype="float16" \
    --tasks truthfulqa,hellaswag,winogrande \
    --device cuda:0 \
    --batch_size 14\
    --output_path ./eval_llama2_7b_nf4_lora \
    --log_samples

Recommendations and Conclusion

Evaluating LLMs with the Evaluation Harness is very simple and inexpensive. Note that it also supports vLLM for faster evaluation.

New tasks are regularly added. If you don’t know which tasks to use for evaluation, have a look at how recent LLMs are evaluated. You may also select some of the tasks used by the OpenLLM Leaderboard.

Many tasks are available but some of them should not be used anymore. For instance, the tasks proposed for machine translation evaluation are outdated (e.g., WMT16). Much more recent better-constructed benchmarks are available online (e.g., WMT23) but not yet implemented in the Evaluation Harness.

A good rule of thumb is to use tasks that are less than 1 year old. With older benchmarks, there is a greater risk of data contamination, i.e., the model might have been trained on the benchmarks and consequently might yield artificially high metric scores.
