Most of the chat models that you can find online only went through SFT. The performance of these models is already quite good but to fully achieve the training of an instruct LLM we have two more steps to perform:

  • Training a reward model

  • Reinforcement learning (with PPO)

If SFT is already good, why do we need to continue with two more steps?

In short, these two steps help a lot to align the model with the users. Instruct LLMs are especially less biased and less toxic than models trained only with SFT.

As we saw, SFT can be costly if you train the model for several epochs. Training the reward model is much more straightforward.

In this article, we will see what is the reward model and how to train it with DeepSpeed Chat.

This is achievable with a much smaller base model than SFT. The main cost of training the reward model is in the training data which typically requires humans’ input.

What is the reward model for instruct LLMs?

First, let’s check how OpenAI trained their own instruct LLM (GPT-3.5). It’s documented in this paper (Section 3.5).

Training language models to follow instructions with human feedback (Ouyang et al., 2022)

They trained a model taking a prompt and a response as input and returning a scalar reward. In other words, it returns a value that signals how good, or bad, is the pair prompt-response. A correct response to the prompt will have a higher scalar reward than an incorrect response.

It’s very important to note that this scalar reward is not designed to be interpretable by humans. You won’t be able to judge the correctness of a response by just reading the scalar reward. However, for the same prompt, you can rank the responses using the scalar reward.

The purpose of the reward model is to give feedback to the model trained at step 3. I’ll detail how this scalar reward is used at step 3 in the next article of this series.

To train the reward model, OpenAI refers to the previous work by Stiennon et al. (2020).

They used the SFT model trained at step 1 to generate from 4 to 9 responses for each prompt. Then, they asked humans to rank the responses for each prompt.

Once the ranking is done, the training data is created. Technically, the training data consists of pairs of responses, given a prompt.

Let’s see one way to format the training data.

We denote three responses R1, R2, and R3, for the same prompt. The human ranking is as follows:

  1. R3

  2. R1

  3. R2

Among the 3 responses, R3 is the best response while R2 is the worst. We form all the possible pairs (u,v):

  • (R3,R1)

  • (R2,R1)

  • (R3,R2)

Then, we can label each pair. If u is better than v, we label 1, if not, we label 0. Note: 0 and 1 are arbitrary values.

We get:

  • (R3,R1) → 1 (i.e., R3 is better than R1, according to humans)

  • (R2,R1) → 0

  • (R3,R2) → 1

This could be one simple way to design our training data for the reward model, but, in practice, the format can be much more complicated than that to optimize training efficiency.

OpenAI notes that shuffling all training examples leads to overfitting. It’s also inefficient. Instead, putting only responses for the same prompt in a training batch prevents overfitting and is much more efficient. We just have to compute the forward pass for the prompt, once.

One last thing to note is that the problem learned by this reward model is somewhat simpler than learning to generate as we did for SFT.

We can use a much smaller model than the one we used for SFT. OpenAI used a 6 billion parameter model for the reward model, while they used a 175B (supposedly…) for SFT.

Simplifications for cheaper training of the reward model

Training the reward model can be quite cheap, but creating the training data is a very expensive task. We have to:

  1. Generate responses with the SFT model: I would suggest nucleus sampling at inference time to generate k-diverse responses.

  2. Rank the k responses with human annotators

The first item has a computational cost. Generating the responses with SFT can take several hours if you have a small VRAM.

The second item requires human resources. This task performed over thousands of responses would cost thousands of dollars, especially if you want reliable rankings, e.g., you would assign each ranking task to more than one annotator to compute inter-annotator agreements and discard the rankings with a poor agreement.

Tackling these two items is ideal but far too expensive since our goal is only to train an instruct LLM on consumer hardware.

Actually, most instruct LLMs that you will find online didn’t follow these steps. Instead, the reward model is often trained on an existing dataset of responses already ranked by humans, i.e., not generated by the SFT model trained at step 1.

We will do the same here. I will use datasets of responses generated by another LLM and ranked by humans.

Train a reward model with DeepSpeed Chat

The notebook for training the reward model is available here.

For SFT, I used LoRA for a cheaper fine-tuning.

For training the reward model, we don’t need LoRA.

Since we don’t need a big model, I use OPT-350M. It only requires 9 GB of VRAM.

I use the same dataset I used for SFT, but remember that for SFT I used only 20% of the data, and I have already indicated to DeepSpeed Chat during SFT that I’ll use 40% of the data for training the reward model.

All the datasets have a column “rejected” that we didn’t use for SFT (we used the column “chosen”). For each prompt, we have only two choices. This is less than what OpenAI did (they used between 4 and 9 responses) but it can be enough if you have enough training data.

For each prompt, we have two responses ranked:

  1. chosen

  2. rejected

There’s only one pair (chosen, rejected) for each prompt. Since we have only one training example per prompt, in contrast to what OpenAI did, we can mix in the same batch pairs obtained from different prompts.

With this ranking, the reward model has to learn that the chosen response is better than the rejected response.

As you can see in this extract of Dahoas/synthetic-instruct-gptj-pairwise, “rejected” doesn’t necessarily mean that the answer is bad, but it’s not as good as the “chosen” one.

With DeepSpeed Chat, training the reward model on this kind of dataset is as simple as one command line. I left most hyperparameters to default. Note: If you need an explanation for some hyperparameters, leave a comment.

cd training/step2_reward_model_finetuning/
deepspeed --num_gpus 1 main.py --model_name_or_path facebook/opt-350m \
   --data_path Dahoas/rm-static Dahoas/full-hh-rlhf Dahoas/synthetic-instruct-gptj-pairwise yitingxie/rlhf-reward-datasets \
   --data_split 2,4,4 \
   --gradient_checkpointing \
   --num_train_epochs 1 \
   --per_device_eval_batch_size 4 \
   --per_device_train_batch_size 4 \
   --seed 1234 \
   --num_padding_at_beginning 1 --weight_decay 0.1 --disable_dropout --gradient_accumulation_steps 16 --zero_stage 0 \
   --deepspeed --output_dir ./rm_ds/

Note that you only need a GPU with 9 or 10 GB of VRAM to train this model, but you need a lot of CPU RAM. In my experiments, the CPU RAM consumption peaked at 41 GB (before the training itself started). If you don’t have that much CPU RAM, Google Colab Pro would work, or consider upgrading your computer configuration. CPU RAM kits of 2*32 GB are quite affordable now (less than $150).

For the following step for training the instruct LLM, we will use the reward model. If you don’t train your own, you can use the one I have uploaded on the HF hub:

  • kaitchup/OPT-350M-RM-DSChat

DeepSpeed Chat also proposes a script to test your model. You can run it like this (in the same directory as the training script):

python rw_eval.py --model_name_or_path kaitchup/OPT-350M-RM-DSChat

Note: Replace “kaitchup/OPT-350M-RM-DSChat” with the path to your model.

You should get an output similar to this:

It seems to work well: The good answers got better scores than the bad answers.

As we discussed, the absolute scores themselves are meaningless. In the second example, the good answer has a negative score while the answer is acceptable (but truncated). The model didn’t learn to score, it learned to rank.

We are ready for the last step of training an instruct LLM: reinforcement learning with PPO. We will use both models that we have trained at step 1 and step 2.

Note: The following addresses a bug I got with rw_eval.py.

The rw_eval.py script should run without any errors… But in my case, using a V100 GPU, it didn’t.

I got this error:

Default process group has not been initialized, please make sure to call init_process_group.

The issue comes from this method:

torch.distributed.get_rank()

I didn’t take the time to understand what was going on exactly here. However, since torch.distributed.get_rank() is only used to print logs for this evaluation, I simply removed all the calls to torch.distributed.get_rank().

Fortunately, they are all in this file: DeepSpeedExamples/applications/DeepSpeed-Chat/training/utils/model/model_utils.py

You will find several times in this file these two lines:

if torch.distributed.get_rank() == 0:
            print ....

Remove them and then rw_eval.py should run as expected.
