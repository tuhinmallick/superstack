However, several open-source frameworks now enable fine-tuning LLMs without coding.

Axolotl is one of them. It’s a tool designed to simplify the fine-tuning process, with extensive support for various configurations and architectures. With Axolotl, users only need to create a fine-tuning configuration file and provide it to the framework. Axolotl takes care of all other aspects, including managing package dependencies and dataset preprocessing.

Setting up Axolotl

If you fine-tune LLMs with HF Transformers, then fine-tuning with Axolotl is straightforward. You can see Axolotl as a layer on top of HF Transformers in which you only have to define a configuration for the fine-tuning.

It is very simple and doesn’t require any coding.

To streamline fine-tuning, Axolotl does a very good job. However, the first use of Axolotl can be quite intimidating. There are a huge number of options that can be defined in an Axolotl configuration file and, in my opinion, they are not very well documented.

You can find all the configuration options in this file. Note: I couldn’t confirm whether this file is up to date but at least the most important settings are defined.

We will see in the next section the options that matter for QLoRA and LoRA fine-tuning.

But first, we have to install Axolotl:

git clone https://github.com/OpenAccess-AI-Collective/axolotl
cd axolotl && pip3 install packaging ninja && pip3 install -e '.[flash-attn,deepspeed]'

Moreover, if you use a gated model, such as Llama 3 and Mistral, you will need to enter your Hugging Face access token after running this command:

huggingface-cli login

Understanding the Configuration File of Axolotl

Let’s see now a concrete example of an Axolotl configuration file. In this example, I aim to fine-tune Llama 3 8B to transform it into a chat model. For a memory-efficient fine-tuning feasible on consumer hardware, I use a QLoRA configuration very close to the configuration I used to fine-tune Llama 3 with Transformers in this article:

For Axolotl, this configuration looks like this:

(I commented directly in the configuration all the important settings)

#Enter here the id of the model you want to fine-tune. It can be an HF repository or a local path.
base_model: meta-llama/Meta-Llama-3-8B

#The configuration settings for the bitsandbytes configuration used for QLoRA. If you don't use QLoRA, you don't need to write these settings.
bnb_config_kwargs:
  llm_int8_has_fp16_weight: false
  bnb_4bit_quant_type: nf4
  bnb_4bit_use_double_quant: true

#Set the loading of the model in 4-bit. It can't be set in the bnb_config_kwargs (I tried...) but I suspect it will be in the future since HF Transformers is deprecating "load_in_4bit"
load_in_4bit: true

strict: false

#Define here your training dataset. For a chat dataset, be aware that Axolotl doesn't support many format. The original format of Hugging Face's Ultrachat 200k is not supported. I had to use a version converted to the ShareGPT format.
datasets:
  - path: PhilipMay/UltraChat-200k-ShareGPT-clean
    split: train
    type: sharegpt
    conversation: chatml
#For the chat template, I use chatml.
chat_template: chatml
#Where to save the fine-tuned adapter
output_dir: ./ultrachat-200k-sharegpt

#We must set that we are fine-tuning a QLoRA adapter with this line
adapter: qlora

#The maximum sequence length. By default, Axolotl will remove all the sequences longer than this value from the training and evaluation data.
sequence_len: 512

#Set packing to true for faster training
sample_packing: false
eval_sample_packing: false

#Since not all the examples have the same length, we must pad them.
pad_to_sequence_len: true

#The LoRA configuration
lora_r: 16
lora_alpha: 16
lora_dropout: 0.05
lora_target_modules:
  - q_proj
  - v_proj
  - k_proj
  - o_proj
  - gate_proj
  - down_proj
  - up_proj

#To directly target all the modules targeted above we could just set the following instead:
#lora_target_linear: true

#Training arguments
gradient_accumulation_steps: 4
micro_batch_size: 8
optimizer: paged_adamw_8bit
lr_scheduler: linear
learning_rate: 0.0001

#For the evaluation dataset, I use the test split from the same dataset used for training
test_datasets:
  - path: PhilipMay/UltraChat-200k-ShareGPT-clean
    split: test
    type: sharegpt
    conversation: chatml

#Use bfloat16 if your GPU is compatible
bf16: auto

#Use gradient checkpointing to save up to 70% of memory. 
gradient_checkpointing: true
gradient_checkpointing_kwargs:
   use_reentrant: true

#Activate FlashAttention for faster and memory-efficient processing of long training examples
flash_attention: true

warmup_ratio: 0.1
num_epochs: 4
logging_steps: 10
eval_steps: 100
saves_per_epoch: 1
weight_decay: 0.0

#The special token used for padding Llama 3
special_tokens:
  pad_token: <|eot_id|>

If some training arguments are unclear, remember that I wrote an article explaining them:

We can save this configuration into a YAML file. I named it “test_axolotl.yaml“.

Fine-tuning Llama 3 with Axolotl

To start fine-tuning Llama 3 with the Axolotl configuration defined above, run:

accelerate launch -m axolotl.cli.train test_axolotl.yaml

How to read the logs?

Before the training starts, Axolotl preprocesses the datasets and provides useful information about the training settings.

[2024-06-14 09:14:51,566] [DEBUG] [axolotl.load_tokenizer:281] [PID:6970] [RANK:0] BOS: 128000 / <|begin_of_text|>
[2024-06-14 09:14:51,566] [DEBUG] [axolotl.load_tokenizer:282] [PID:6970] [RANK:0] PAD: 128009 / <|eot_id|>
[2024-06-14 09:14:51,567] [DEBUG] [axolotl.load_tokenizer:283] [PID:6970] [RANK:0] UNK: None / None

The logs confirm that the special tokens are properly set.

Saving the dataset (1/1 shards): 100% 13067/13067 [00:00<00:00, 45046.52 examples/s]

The dataset I used for training has 200k rows. Axolotl uses only 13,067 rows. What happened?

By default, Axolotl removes the training examples longer than the sequence_len. Since most examples are longer than 512 tokens, most of them are removed. Note that this behavior is different from the default behavior of Hugging Face Transformers which will truncate the long examples instead of removing them.

It also shows the memory consumption before and after creating the adapter. For this configuration, we can see that the adapter only consumes 89 MB. The logs also print the number of parameters, similar to what Hugging Face’s Trainer does before starting training.

Then, once the training starts, it prints regular information on how it is going:

{'loss': 1.1045, 'grad_norm': 0.4453125, 'learning_rate': 6.134969325153374e-06, 'epoch': 0.02}
{'loss': 1.0903, 'grad_norm': 0.546875, 'learning_rate': 1.2269938650306748e-05, 'epoch': 0.05}
{'loss': 1.0635, 'grad_norm': 0.498046875, 'learning_rate': 1.8404907975460123e-05, 'epoch': 0.07}
{'loss': 0.9691, 'grad_norm': 0.2353515625, 'learning_rate': 2.4539877300613496e-05, 'epoch': 0.1}
{'loss': 0.9132, 'grad_norm': 0.224609375, 'learning_rate': 3.067484662576687e-05, 'epoch': 0.12}
{'loss': 0.9193, 'grad_norm': 0.2119140625, 'learning_rate': 3.6809815950920246e-05, 'epoch': 0.15}
{'loss': 0.8425, 'grad_norm': 0.244140625, 'learning_rate': 4.2944785276073626e-05, 'epoch': 0.17}
{'loss': 0.8571, 'grad_norm': 0.28125, 'learning_rate': 4.907975460122699e-05, 'epoch': 0.2}
{'loss': 0.8289, 'grad_norm': 0.25390625, 'learning_rate': 5.521472392638037e-05, 'epoch': 0.22}
{'loss': 0.8388, 'grad_norm': 0.314453125, 'learning_rate': 6.134969325153375e-05, 'epoch': 0.24}

  • loss: We want this value to decrease. It means that the model is learning from the data. For every 200 steps, the trainer will also compute the evaluation loss. We want the evaluation loss to also decrease. If it doesn’t, the trainer is probably overfitting the training data.

  • grad_norm: This is the norm of the weight update. A higher norm means a more aggressive update. For language generation tasks, a norm higher than 5 is unusual and may indicate the training is going to diverge with exploding gradients. It’s more susceptible to happen when training with mixed-precision, i.e., with float16, which might not be precise enough.

  • learning_rate: The learning rate used by the current training step. In the configuration, we chose a warmup with a linear scheduler. The learning rate will increase up to its maximum value, which we defined in the configuration, and then will linearly decrease.

  • epoch: How much of the training data has been seen. For instance, 0.24 means that the trainer has seen 24% of the data. 2.0 means that it saw the data twice.

Once the training is finished, it tells you how long it took and where the adapter was saved:

100% 1632/1632 [12:55:21<00:00, 28.51s/it]
[2024-06-14 22:10:51,961] [INFO] [axolotl.train.train:173] [PID:6970] [RANK:0] Training Completed!!! Saving pre-trained model to ./drive/MyDrive/axo/ultrachat-200k-sharegpt

This is a standard adapter that you can use for inference with any library that supports adapters.

Axolotl also proposes an inference engine that you can run as follows:

accelerate launch -m axolotl.cli.inference test_axolotl.yaml --qlora_model_dir="./qlora-out" --gradio

Conclusion

Axolotl is a framework that is easy to use for training LLMs. If you don’t want to code, this is a very good option.

A few downsides:

  • How to configure the datasets (custom chat template, prompt, splits, …) isn’t clear. There are many settings that are not easy to understand.

  • The framework is under-documented. Finding what a particular option does often requires looking at the code or launching a training to see what happens.
