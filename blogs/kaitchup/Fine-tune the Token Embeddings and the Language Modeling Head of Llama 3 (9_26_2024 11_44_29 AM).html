We can easily adapt a pre-trained large language model (LLM) to new tasks thanks to low-rank adaptation (LoRA). LoRA freezes the entire model and adds a small amount of trainable parameters on top of it. By only training these new parameters instead of the entire model, LoRA, and its quantized variant QLoRA, save a lot of GPU memory and make it possible to fine-tune LLMs on consumer hardware.

LoRA is usually only applied to attention and MLP modules. The token embeddings and the language modeling head remain the same after LoRA fine-tuning. This is often not ideal since the token embeddings learned during pre-training are very general without any domain or task specialization. Some of them may have even been left untrained. This is the case for instance for some special tokens of Llama 3 8B.

Ideally, we should retrain the token embeddings and the language modeling head to better adapt the model to a new task or domain.

But what is the cost of this retraining? Is it really worth it? When should we do it?

In this article, I investigate the impact of retraining the token embeddings and language modeling head of Llama 3 during (Q)LoRA fine-tuning. We will see that, due to the large vocabulary of Llama 3, it is indeed very costly in GPU memory but still feasible on consumer hardware. More importantly, we will see that retraining the token embeddings and language modeling head of Llama 3 can significantly improve the fine-tuning.

What Are the Token Embeddings and the Language Modeling Head of Llama 3?

The token embeddings are the representations of all the vocabulary entries of Llama 3. When Llama 3 encodes an input, one of the very first steps is to assign each token in the input with its token embedding. Each token embedding has 4,096 dimensions for Llama 3 8B and Llama 3 has a vocabulary containing 128,256 tokens. In total, the token embeddings contain 128,256 * 4,096 = 525M parameters.

On the other hand, the language modeling head is the last linear layer of the transformer. It resizes a tensor with the size of the hidden states to the size of the vocabulary. That’s 4,096 * 128,256 = 525M parameters again.

If we print the model once loaded with Hugging Face Transformers, we can see the size of the token embeddings and language modeling head:

LlamaForCausalLM(
  (model): LlamaModel(
    (embed_tokens): Embedding(128256, 4096)
    (layers): ModuleList(
      (0-31): 32 x LlamaDecoderLayer(
        (self_attn): LlamaFlashAttention2(
          (q_proj): Linear(in_features=4096, out_features=4096, bias=False)
          (k_proj): Linear(in_features=4096, out_features=1024, bias=False)
          (v_proj): Linear(in_features=4096, out_features=1024, bias=False)
          (o_proj): Linear(in_features=4096, out_features=4096, bias=False)
          (rotary_emb): LlamaRotaryEmbedding()
        )
        (mlp): LlamaMLP(
          (gate_proj): Linear(in_features=4096, out_features=14336, bias=False)
          (up_proj): Linear(in_features=4096, out_features=14336, bias=False)
          (down_proj): Linear(in_features=14336, out_features=4096, bias=False)
          (act_fn): SiLU()
        )
        (input_layernorm): LlamaRMSNorm()
        (post_attention_layernorm): LlamaRMSNorm()
      )
    )
    (norm): LlamaRMSNorm()
  )
  (lm_head): Linear(in_features=4096, out_features=128256, bias=False)
)

These two modules of Llama 3 8B combined represent 1.05B parameters, i.e., 12.5% of its total size. These parameters are among the most important of the transformer. Most quantization methods don’t quantize them to preserve accuracy. If we have a model with a large vocabulary like Llama 3, a significant portion of the model will not be quantized. These 1.05B parameters occupy 2.1 GB of memory even after quantization with GPTQ or AWQ.

Fine-tuning the Token Embeddings and Language Modeling Head of Llama 3

A typical LoRA configuration for fine-tuning LLMs using the Llama architecture looks like this:

peft_config = LoraConfig(
        lora_alpha=8,
        lora_dropout=0.05,
        r=8,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ["k_proj", "q_proj", "v_proj", "o_proj","gate_proj", "up_proj", "down_proj"]
)

The target modules are all the linear modules from the self-attention and MLP. The token embeddings and the language modeling head remain frozen.

This is good enough for most use cases. However, for models with a large vocabulary, e.g., Llama 3, Qwen, or Gemma, it means that a large number of parameters in the model won’t be impacted by the fine-tuning.

Let’s take an extreme case where we want the model to be adapted to a new language. If the LLM has been pre-trained mainly on English tokens, the token embeddings were trained to model English tokens. Frequent English tokens will be better represented and will have more importance in the entire model. If we switch to a new language very different from English and with characters rarely used in English, e.g., Polish, tokens rarely seen in the pre-training data will be under-trained. For instance, “Dzień dobry” (“good morning”) is a very unlikely sequence of tokens in English. However, in Polish text, this is a frequent sequence and we need it to be well encoded.

In this situation, if we want to fine-tune our LLM on Polish text, we shouldn’t freeze the language modeling head and token embeddings. We should retrain them to better model this “domain shift”.

Pirate speak was probably very rare in the pre-training data of Llama 3. For instance:

Ahoy there, matey!

is a sequence of tokens that is very unlikely in English and thus difficult to model with the original Llama 3’s token embeddings.

Moreover, in this same article, we saw that the special tokens used by our chat template were not trained. Our model couldn’t know when to generate the correct special tokens since all the special tokens had the same embeddings.

Let’s correct this by retraining the token embeddings and the language modeling head. This should help the model tackle the domain shift, i.e., from regular English to pirate speak and learn the special token embeddings.

We can use the same fine-tuning code and simply modify the LoraConfig, as follows:

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        modules_to_save=["lm_head","embed_tokens"],
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

With “modules_to_save”, we specify which modules whose parameters we want to fully fine-tune. lm_head is the language modeling head. embed_tokens are the token embeddings.

This configuration yields 1.05B trainable parameters. If we weren’t fully fine-tuning lm_head and embed_tokens, we would only have 41M trainable parameters.

1.05B trainable parameters would require more than 14.3 GB of GPU RAM just for the AdamW optimizer’s states. In addition to the memory consumed by the model itself and the activations used to compute the gradients, this fine-tuning would consume much more than 24 GB of GPU RAM.

Fortunately, using the 8-bit version of AdamW can reduce the memory consumption of the optimizer’s states to 3.59 GB, or even slightly less with the paged version of AdamW 8-bit.

If we use a training batch size of 8 with a sequence length of 512, this fine-tuning is possible with 22.5 GB of GPU RAM.

Here is the full fine-tuning code:

import torch, os, multiprocessing
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig,
    TrainingArguments,
)
from trl import SFTTrainer

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

model_name = "meta-llama/Meta-Llama-3-8B"

tokenizer = AutoTokenizer.from_pretrained(model_name)
assistant_name = "Barbe Noire"
system_prompt = "You are the pirate Barbe Noire. You answer only as Barbe Noire and using pirate speak."
format_system_prompt = "<|start_header_id|>system<|end_header_id|>\n\n"+system_prompt+"<|eot_id|>"
tokenizer.chat_template = "<|begin_of_text|>"+format_system_prompt+"{% set loop_messages = messages %}{% for message in loop_messages %}{% set content = '<|start_header_id|>' + message['role'] + '<|end_header_id|>\n\n'+ message['content'] | trim + '<|eot_id|>' %}{% if loop.index0 == 0 %}{% set content = bos_token + content %}{% endif %}{{ content }}{% endfor %}{% if add_generation_prompt %}{{ '<|start_header_id|>"+assistant_name+"<|end_header_id|>\n\n' }}{% endif %}"

ds = load_dataset("Peyton3995/dolly-15k-mistral-pirate")
def process(row):
    messages = [
      {"role": "user", "content": row['instruction']},
      {"role": assistant_name, "content": row['response']},
    ]
    row["text"] = tokenizer.apply_chat_template(messages, tokenize=False)+"<|end_of_text|>"
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

print(ds['train'][0]['text'])

tokenizer.pad_token = tokenizer.eos_token
tokenizer.padding_side = 'left'

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
)

model = prepare_model_for_kbit_training(model)

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        modules_to_save=["lm_head","embed_tokens"],
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

training_arguments = TrainingArguments(
        output_dir="./Llama3_8b_Pirate_QLoRA",
        optim="paged_adamw_8bit",
        per_device_train_batch_size=8,
        gradient_accumulation_steps=4,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=100,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        num_train_epochs=3,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

If we compare the training loss with the fine-tuning I’ve done in my previous article that was using the same code, we can see that the new fine-tuning is learning much faster (and probably overfit…). It successfully learns the new domain. Moreover, it also learns embeddings for the special tokens of our chat template. Now, the model should be able to generate “<|eot_id|>” and stop generating after that, instead of randomly generating special tokens. Let’s check:

import torch, os
from peft import PeftModel
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig
)

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

base_model =  "meta-llama/Meta-Llama-3-8B"
adapter_name = "./Llama3_8b_Pirate_QLoRA/checkpoint-1407/"

tokenizer = AutoTokenizer.from_pretrained(adapter_name)

terminators = [
    tokenizer.eos_token_id,
    tokenizer.convert_tokens_to_ids("<|eot_id|>")
]

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          base_model, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
)

model = PeftModel.from_pretrained(model, adapter_name)

messages = [
  {"role": "user", "content": "Where are you from?"},
]

seq = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
print(seq)

encoded_input = tokenizer(seq, return_tensors="pt").to(model.device)
with torch.cuda.amp.autocast():
  outputs = model.generate(
      encoded_input['input_ids'],
      max_new_tokens=150,
      eos_token_id=terminators,
      do_sample=True,
      temperature=0.8,
      top_p=0.9,
  )


response = outputs[0][encoded_input['input_ids'].shape[-1]:]
print(tokenizer.decode(response))

It generates:

Arr, matey, I be a pirate from the Seven Seas! I be roamin' the Caribbean waters and raidin' ships from the Spanish Main to the Bahamas. We be livin' the life of freedom and adventure on the open ocean! Yarr!<|eot_id|>

It works! The model only generated the assistant’s answer instead of a full dialogue and didn’t generate any other special tokens than the expected ones.

The downsides of Fully Fine-tuning the Token Embeddings and the Language Modeling Head

This fine-tuning consumes much more memory than the standard (Q)LoRA fine-tuning. Moreover, since we are training many more parameters, they will tend to overfit the training dataset, especially if we are using a high learning rate and a small dataset.

Using timdettmers/openassistant-guanaco, which is a small instruction dataset (<10k samples), I compared the training loss and validation loss of 4 different LoraConfig:

  • Config 1: Standard QLoRA

  • Config 2: Standard QLoRA + full fine-tuning of embed_tokens and lm_head

  • Config 3: Standard QLoRA + full fine-tuning of lm_head

  • Config 4: Standard QLoRA + full fine-tuning of embed_tokens

I run the fine-tuning for 3 epochs. The code is in the notebook.

The results:

As we can see, when we introduce more trainable parameters, the training loss quickly decreases. However, the validation loss increases after the first epoch. The lowest validation loss is obtained with Config 1.

Three epochs are probably too many for this dataset. Nonetheless, Config 2 should be able to reach a validation loss at least as low as Config 1. This is achievable by optimizing the hyperparameters, especially the learning rate, which should be much lower, and maybe with a slightly longer warmup. Increasing LoRA’s alpha may also help to reduce overfitting.

Conclusion

Fully fine-tuning the token embeddings is critical when we want to add special tokens to the vocabulary. However, this is not free. It increases memory consumption and requires a more careful selection of hyperparameter values.

If you are not adding any special tokens, or don’t need to fine-tune existing ones, retraining the token embeddings might not be necessary. I would recommend it only if you think there is a significant domain shift, i.e., fine-tuning for another language than English or a text style/genre that is rare on the Internet.
