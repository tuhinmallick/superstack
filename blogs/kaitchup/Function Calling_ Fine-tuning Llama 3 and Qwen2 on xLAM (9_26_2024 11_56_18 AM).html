Recent large language models (LLMs) are highly capable in most language generation tasks. However, since they operate based on next-token prediction, they often struggle with accurately performing mathematical operations. Additionally, due to their knowledge cut-off, they may lack the information needed to answer some queries accurately.

One way to alleviate these issues is through function calling. Function calling allows LLMs to reliably connect to external tools. It enables effective tool usage and interaction with external APIs. For example, retrieving information from the Internet and performing mathematical operations can be accomplished through function calling by interfacing the LLM with a web search engine and a calculator.

In this article, we will see how to fine-tune LLMs for function calling. I use xLAM, a dataset of 60k entries of function calling released by Salesforce for fine-tuning Llama 3 and Qwen2. We will see how to format the dataset and how we can exploit the fine-tuned adapters for function calling.

Function Calling for LLMs: How Does it Work?

For instance, if you prompt a standard LLM with “Give me the square root of 3342398”, it will generate the answer, one digit at a time, which will probably be very inaccurate. Let’s try it with Llama 3 Instruct:

import transformers
import torch

model_id = "meta-llama/Meta-Llama-3-8B-Instruct"

pipeline = transformers.pipeline(
    "text-generation",
    model=model_id,
    model_kwargs={"torch_dtype": torch.bfloat16},
    device_map="auto",
)

messages = [
    {"role": "system", "content": "You are a calculator."},
    {"role": "user", "content": "Give me the square root of 3342398"},
]

terminators = [
    pipeline.tokenizer.eos_token_id,
    pipeline.tokenizer.convert_tokens_to_ids("<|eot_id|>")
]

outputs = pipeline(
    messages,
    max_new_tokens=256,
    eos_token_id=terminators,
    do_sample=True,
    temperature=0.6,
    top_p=0.9,
)
print(outputs[0]["generated_text"][-1])

The results:

The square root of 3342398 is 1833.13

It’s close but wrong. The right answer is “1828.222634144977” LLMs can’t do math accurately. They can reason on math problems and approximate results but they won’t be as accurate as a calculator. They are not designed for it. However, they can call a calculator. We simply need the model to “understand” that we are requesting the square root of the number 3342398.

For this, we can fine-tune the LLMs on a dataset for function calling. Usually, datasets for function calling have at least two columns:

  • Query (or prompt): This is a standard prompt, e.g., “Give me the square root of 3342398”

  • Tool call: This is usually in the JSON format, e.g.:

{
    "name": "math.sqrt"    
    "description": "Python square root function"
    "tool_id":20
    "arguments": 
    {
        "1": 3342398
    }
}

At inference time, we want the model to generate the “tool call” given the “query”. Using our example above, the mode would return a JSON object with a tool id “20”. After parsing the LLM’s output, we can retrieve the IDs of the tools and their arguments.

Then, we could pass the “arguments” to the selected tools. For the example above, it would return the exact square root of the requested number.

Nonetheless, note that function calling is a very difficult task for LLMs. Many things can go wrong:

  • The model might not generate a valid JSON for the tool call.

  • The mode might not properly “understand” the query, e.g., calling the tool with the id 19 instead of 20 or not setting the right arguments.

Function calling should always be supported by control code, for instance, making sure that the JSON object is valid before parsing, and checking that the arguments can be cast to the format expected by the tool. For instance, if the model generates the following tool call:

{
    "name": "math.sqrt"    
    "description": "Python square root function"
    "tool_id":20
    "arguments": 
    {
        "1": "sqrt of 3342398"
    }
}

the code should deny the call because the argument “sqrt of 3342398" can’t be cast to the float type expected by the square root function.

xLAM: A Large Dataset for Function Calling

xLAM is a dataset for fine-tuning LLMs for function calling. It is available on the Hugging Face Hub:

  • Salesforce/xlam-function-calling-60k (CC-BY license)

It’s gated. You need a Hugging Face account, an access token, and also to agree with Salesforce’s terms (form on the model card).

The dataset has only one split (train) and is organized into 4 columns:

  • id: An identifier for the example, e.g., “5,633”

  • query: A human prompt requiring function calling capacities, e.g.,

What is the smallest number that both 12 and 15 can divide into without leaving a remainder?

  • tools: A list of tools or functions that will be called to answer the query. Each tool is a JSON object with multiple fields describing the expected parameters, e.g.,

[{"name": "least_common_multiple", "description": "Computes the least common multiple (LCM) of two positive integers.", "parameters": {"a": {"description": "The first positive integer.", "type": "int"}, "b": {"description": "The second positive integer.", "type": "int"}}}]

  • answers: The list of tool and function calls with arguments retrieved from the query. For instance, 

[{"name": "least_common_multiple", "arguments": {"a": 12, "b": 15}}]

The LLM fine-tuned for function calling should generate “answers”. Then, for the example above, we would have to parse the answer of the model to retrieve the name of the tool “least_common_multiple” and its arguments " {"a": 12, "b": 15}”.

We could imagine that we have a Python program that simply checks whether we have this tool and that the arguments are correct (number, type, etc.). A simple example could be:

if tool == "least_common_multiple":
    check = check_args_least_common_multiple(args)
    if check:
        return least_common_multiple(args["a"], args["b"])

Then we would return to the user the value returned by the code above.

xLAM contains 60k examples for training. That’s more than enough but I found that some of them are of poor quality. Some have empty answers or seem to call for tools unrelated to the query:

Preprocessing xLAM

Fine-tuning an LLM for function calling is not very different from standard fine-tuning. The code is mostly the same. The main difference is in the data preprocessing. But we have to answer first: what do we want the LLM to generate given a user query?

For instance, you might only want your LLM to generate the name of the tool to call and its arguments. This would be simple to learn for the LLM.

For this tutorial, I trained the model to learn to generate the answers and the tools columns. I made this template:

<user>{query}</user>

<tools>{tools}</tools>

<calls>{answers}</calls><|end_of_text|>

For instance, it yields:

<user>Where can I find live giveaways for beta access and games?</user>

<tools>{'name': 'live_giveaways_by_type', 'description': 'Retrieve live giveaways from the GamerPower API based on the specified type.', 'parameters': {'type': {'description': 'The type of giveaways to retrieve (e.g., game, loot, beta).', 'type': 'str', 'default': 'game'}}}</tools>

<calls>{'name': 'live_giveaways_by_type', 'arguments': {'type': 'beta'}}
{'name': 'live_giveaways_by_type', 'arguments': {'type': 'game'}}</calls><|end_of_text|>

As there is a lot of JSON syntax to generate, this task is quite difficult for an LLM. To preprocess xLAM with this template, I used this code:

ds = load_dataset("Salesforce/xlam-function-calling-60k", split="train")

#Add the EOS token
def process(row):
    row["query"] = "<user>"+row["query"]+"</user>\n\n"

    tools = []
    for t in json.loads(row["tools"]):
      tools.append(str(t))

    answers = []
    for a in json.loads(row["answers"]):
      answers.append(str(a))

    row["tools"] = "<tools>"+"\n".join(tools)+"</tools>\n\n"
    row["answers"] = "<calls>"+"\n".join(answers)+"</calls>"
    row["text"] = row["query"]+row["tools"]+row["answers"]+tokenizer.eos_token
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

The training examples are now in a new column named “text”.

Install the following libraries:

pip install --upgrade bitsandbytes transformers peft accelerate datasets trl

Then, import:

import torch, os, multiprocessing, json
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig
)
from trl import SFTTrainer, SFTConfig

Next, we check whether FlashAttention and bfloat16 are compatible with the GPU. If they are, we will use them. We also set the padding token (more details in this article):

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

model_name = "meta-llama/Meta-Llama-3-8B"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_name, use_fast=True)
tokenizer.pad_token = "<|eot_id|>"
tokenizer.pad_token_id = 128009
tokenizer.padding_side = 'left'

I made a QLoRA function to which we will only have to pass our preprocessed xLAM:

def QLoRA(ds):
  bnb_config = BitsAndBytesConfig(
          load_in_4bit=True,
          bnb_4bit_quant_type="nf4",
          bnb_4bit_compute_dtype=compute_dtype,
          bnb_4bit_use_double_quant=True,
  )
  model = AutoModelForCausalLM.from_pretrained(
            model_name, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
  )

  model = prepare_model_for_kbit_training(model, gradient_checkpointing_kwargs={'use_reentrant':True})

  #Configure the pad token in the model
  model.config.pad_token_id = tokenizer.pad_token_id
  model.config.use_cache = False # Gradient checkpointing is used by default but not compatible with caching

  peft_config = LoraConfig(
          lora_alpha=16,
          lora_dropout=0.05,
          r=16,
          bias="none",
          task_type="CAUSAL_LM",
          target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
  )

  training_arguments = SFTConfig(
          output_dir="./Llama3_8b_xLAM",
          optim="adamw_8bit",
          per_device_train_batch_size=8,
          gradient_accumulation_steps=4,
          log_level="debug",
          save_steps=250,
          logging_steps=10,
          learning_rate=1e-4,
          fp16 = not torch.cuda.is_bf16_supported(),
          bf16 = torch.cuda.is_bf16_supported(),
          max_steps=1000,
          warmup_ratio=0.1,
          lr_scheduler_type="linear",
          dataset_text_field="text",
          max_seq_length=512,
  )

  trainer = SFTTrainer(
          model=model,
          train_dataset=ds,
          peft_config=peft_config,
          tokenizer=tokenizer,
          args=training_arguments,
  )

  trainer.train()
QLoRA(ds)

I only trained for 1000 steps and didn’t search for better hyperparameters. I would recommend training at least for 1 epoch and setting “max_seq_length” to 1024 to make sure that the training examples are not truncated too often. These changes would improve a lot the adapter but would also increase the memory consumption and the training would take more than one day on consumer hardware. Setting a lower learning rate might also help.

Once the training is finished, you can try the fine-tuned adapter with the following code. First, we load the model and the adapter, using the same quantization configuration used during fine-tuning:

import torch,os
from peft import PeftModel
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig
)

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'
quantization_config=BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_compute_dtype=compute_dtype,
    bnb_4bit_use_double_quant=True,
    bnb_4bit_quant_type="nf4",
)

adapter= "./Llama3_8b_xLAM/checkpoint-1000"
model_name = "meta-llama/Meta-Llama-3-8B"
tokenizer = AutoTokenizer.from_pretrained(model_name, use_fast=True)
print(f"Starting to load the model {model_name} into memory")

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    quantization_config=quantization_config,
    torch_dtype=compute_dtype,
    device_map={"": 0},
    attn_implementation=attn_implementation,
)

print(model)
model = PeftModel.from_pretrained(model, adapter)

Then, we can try some queries. Note: We don’t want the model to be creative for function calling but rather output the most probable answer. In other words, we should deactivate the sampling and set the temperature to 0.0.

prompt = "<user>Check if the numbers 8 and 1233 are powers of two.</user>\n\n<tools>"
inputs = tokenizer(prompt, return_tensors="pt").to("cuda")
outputs = model.generate(**inputs, do_sample=False, temperature=0.0, max_new_tokens=150)
result = tokenizer.decode(outputs[0], skip_special_tokens=True)
print(result)

It generates:

<user>Check if the numbers 8 and 1233 are powers of two.</user>

<tools>is_power_of_two</tools>

<calls>{'name': 'is_power_of_two', 'arguments': {'num': 8}}
{'name': 'is_power_of_two', 'arguments': {'num': 1233}}</calls>

This is not too bad for only 1000 training steps. However, it didn’t generate a good tool description between <tools> and </tools>. This isn’t JSON and is very different from what we have in the training data. This would be fixed with longer training.

In the notebook, you will also find the code to fine-tune Qwen2 for function calling. This is the same as for Llama 3, except for the configuration of the pad token. I use the same fine-tuning hyperparameters which is surely suboptimal.

For the same prompt as in the example above, Qwen2 yields:

<user>Check if the numbers 8 and 1233 are powers of two.</user>

<tools>Tools:</tools>

<calls>is_power_of_two(8)</calls>
<calls>is_power_of_two(1233)</calls>

This is worse than with Llama 3 8B. It didn’t produce any JSON. The hyperparameters must be adjusted.

Conclusion

Fine-tuning LLMs for function calling gives them the ability to perform accurate mathematical operations, retrieve up-to-date information, and call APIs to perform a wide range of tasks.

Using the xLAM dataset, which contains 60,000 function calling entries, models like Llama 3 and Qwen2 can be trained to generate tool calls. However, as we saw, we might need to train the model for a long time before they properly learn to generate in the expected format. Simplifying the format, for instance, changing from JSON to YAML, and dropping the tool description from the training data, would significantly help the model to learn how to do function calling.
