Mixtral-8x7B is one of the best open LLMs. It is also very challenging to fine-tune it on consumer hardware. The model occupies 96.8 GB of memory when fully loaded. Fine-tuning would require even more memory to store the optimizer states and training batches. For instance, an H100 GPU with 80 GB of RAM wouldn’t be enough.

In this situation, QLoRA with 4-bit quantization is an appealing solution. It divides the model size by 4 while reducing the size of the optimizer states by fine-tuning only a LoRA adapter on top of the model.

Yet, even with QLoRA, we still need 32 GB of GPU memory to fine-tune Mixtral-8x7B. I tried it in this article:

But what if we could fine-tune Mixtral-8x7B quantized to a lower precision?

For instance, we can quantize Mixtral-8x7B with AQLM to 2-bit with minimal degradation of the model’s performance. But are AQLM models easy to fine-tune?

In this article, I show how to fine-tune Mixtral-8x7B quantized with AQLM using only 16 GB of GPU RAM. In other words, we only need a $500 GPU to fine-tune Mixtral. I also discuss how to optimize the fine-tuning hyperparameters to further reduce memory consumption while maintaining a good performance. To my surprise, fine-tuning a 2-bit Mixtral is fast and may yield an even better model than with QLoRA while using twice less memory.

I focus on Mixtral in this article but you can apply the same steps to any models quantized with AQLM. If you want to know how AQLM works and how to quantize a model with it, I wrote an article about it here:

Fine-tuning a 2-bit LLM Quantized with AQLM

The authors of AQLM already propose a quantized version of Mixtral-8x7B on the Hugging Face hub:

  • ISTA-DASLab/Mixtral-8x7b-AQLM-2Bit-1x16-hf (previously at BlackSamorez/Mixtral-8x7b-AQLM-2Bit-1x16-hf)

We will fine-tune this model. It occupies only 12.6 GB of memory. We can fine-tune it with a 16 GB GPU but I rather recommend a 24 GB GPU for faster fine-tuning and to be able to fine-tune a larger LoRA adapter.

First, we need to install AQLM and the latest versions of PEFT, Transformers, Accelerate, and TRL. I also installed bitsandbytes to use paged AdamW 8-bit as this will further reduce memory consumption.

pip install transformers peft trl accelerate bitsandbytes
pip install aqlm[gpu,cpu]

We will need all the following imports:

from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    TrainingArguments,
)
from trl import SFTTrainer

Then, we load the AQLM model:

model = AutoModelForCausalLM.from_pretrained(
    "ISTA-DASLab/Mixtral-8x7b-AQLM-2Bit-1x16-hf",
    trust_remote_code=True, torch_dtype="auto", device_map="cuda", low_cpu_mem_usage=True
)

And the tokenizer:

tokenizer = AutoTokenizer.from_pretrained("ISTA-DASLab/Mixtral-8x7b-AQLM-2Bit-1x16-hf")
tokenizer.pad_token = tokenizer.eos_token

An important step is to prepare the model for this fine-tuning. I used prepare_model_for_kbit_training which seems to work well with AQLM.

model = prepare_model_for_kbit_training(model)

prepare_model_for_kbit_training also enables gradient checkpointing. Without it, we wouldn’t be able to do this fine-tuning on consumer hardware.

With this fine-tuning, we will turn Mixtral-8x7B into an instruct/chat model. I used timdettmers/openassistant-guanaco since it is small and good enough for this tutorial. It is also already in the right format to be used for fine-tuning with TRL since it has a single column named “text”.

dataset = load_dataset("timdettmers/openassistant-guanaco")

As for the LoraConfig, I defined it as follows:

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate", "w1", "w2", "w3"]
)

Be aware that it yields a huge adapter with 242M parameters. This is because I target the modules w1, w2, and w3. These modules are inside the 8 experts of Mixtral, so 24 modules in total. On the other hand, the other modules (self-attention and the gate) are shared by the 8 experts. Their impact on the total number of trainable parameters is negligible.

Nonetheless, even with 242M parameters, this configuration still fits on a 24 GB GPU.

For the training hyperparameters, I chose the following:

training_arguments = TrainingArguments(
        output_dir="./fine-tuned_MixtralAQLM_2bit/",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=4,
        gradient_accumulation_steps=4,
        per_device_eval_batch_size=4,
        log_level="debug",
        logging_steps=25,
        learning_rate=1e-4,
        eval_steps=25,
        save_strategy='steps',
        max_steps=100,
        warmup_steps=25,
        lr_scheduler_type="linear",
)

The most important hyperparameters are:

  • per_device_train_batch_size and gradient_accumulation_steps: I set them to 4 and 4, which yields a total batch size of 16 (4*4).

  • learning_rate: I set it to 1e-4 because it seems to work well. This is definitely not the best value.

  • lr_scheduler_type: I set it to “linear” as it has been shown to always perform on par, if not better, than other schedulers.

  • optim: paged_adamw_8bit performs well while consuming significantly less memory than the original AdamW implementation. The downside is that it slows down fine-tuning, especially if you have an old CPU. If you have enough memory, replace it with “adamw_torch“ or “adamw_8bit”.

I have also set max_steps to 100 for this tutorial. It’s enough to confirm that the model is learning but it’s not enough to train a good model. Train for at least one epoch to get a reasonably good adapter.

Then, we can start training with TRL’s SFTTrainer:

trainer = SFTTrainer(
        model=model,
        train_dataset=dataset['train'],
        eval_dataset=dataset['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=256,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

Depending on your task, you may want to increase “max_seq_length” which I set to 256. This is a low value but it helps to consume less memory.

The fine-tuning went well according to the training and validation losses:

It took a total of 2 hours and 41 minutes. I used the A100 of Google Colab. If you use an RTX GPU, expect a similar training time. If you use an older GPU, e.g., a T4 or an RTX 20xx, it could be between 2 and 4 times slower.

Fine-tuning an AQLM model works surprisingly well. When I tried fine-tuning Mixtral with standard QLoRA, on the same dataset, it consumed 32 GB of VRAM, without targeting w1, w2, and w3 for LoRA, and the perplexity didn’t decrease so well.

Reducing the Memory Consumption

If you only have a GPU with 16 GB of VRAM, fine-tuning Mixtral is still possible but it will be slower and won’t perform as well.

Try the following changes of configuration (from best to worst to maintain the model performance):

  • Decrease per_device_train_batch_size and proportionally increase gradient_accumulation_steps. The minimal value for the training batch size is 1. If you decrease it from 4 to 1 (4x smaller), then you should increase gradient_accumulation_steps from 4 to 16 (4x larger).

  • Target only the self-attention modules by removing w1, w2, w3, and maybe gate from the LoraConfig.

  • Reduce the LoRA rank, the r parameter in the LoraConfig, to 8 or 4.

Conclusion

AQLM is already very well supported by Hugging Face PEFT and Transformers. As we saw in this article, fine-tuning an AQLM model is both fast and memory-efficient. Since I have only fine-tuned for a few training steps, I didn’t evaluate the fine-tuned adapter with benchmarks but the perplexity (or validation loss) reached after 100 fine-tuning steps is promising.

One of the downsides of this approach is that, since the model is already quantized, we can’t merge the fine-tuned adapter. Moreover, as quantizing LLMs with AQLM is extremely costly, AQLM models are still very rare.
