Llama 3.1 70B Instruct is currently one of the best large language models (LLMs) for chat applications. However, running a 70B model on consumer GPUs for fast inference is challenging.

Llama 3.1 70B has 70.6 billion 16-bit parameters. Since one 16-bit parameter occupies 2 bytes of memory, Llama 3.1 70B requires 141.2 GB of GPU memory to be loaded.

With 4-bit quantization, one parameter would occupy only 0.5 bytes. We could run the model on 2x24 GB consumer GPUs such as 2 RTX 4090.

To run on a single GPU, we would need to quantize the model with a precision lower than 2.5-bit. However, if naively performed, such a quantization of Llama 3.1 70B would make the model worse than Llama 3.1 8B, while still being a few GBs larger.

We saw several 2-bit quantization methods that work well for 70B models but have significant drawbacks. They are either too costly (AQLM) or require subsequent fine-tuning (HQQ+).

ExLlamaV2 can quantize a 70B model on a consumer GPU. It applies mixed-precision quantization by aggressively quantizing the less important parameters while preserving the most useful ones.

Is ExLlamaV2 accurate enough to quantize Llama 3.1 70 Instruct to a low precision?

I also explained in this article how ExLlamaV2 works.

Since I published this article, the quantization method has not been updated but the framework now supports Llama 3.1. You can use the notebook (link in the introduction) to make your own quantization of Llama 3.1.

I recommend having around 300 GB free on your hard drive.

It’s very challenging to do it on Google Colab. I did it for Llama 3 70B using Google Drive for storage but got restricted by Google because of the numerous data transfers. This time, I used RunPod (referral link).

First, install ExLlamaV2:

pip install --upgrade transformers exllamav2 datasets 
git clone https://github.com/turboderp/exllamav2.git

Then, download Llama 3.1 70B Instruct to a local directory, e.g., “./Llama-3.1-70B-Instruct/“ (the directory must exist):

from huggingface_hub import snapshot_download
from huggingface_hub import login
login(token='') #Your HF token here

snapshot_download(repo_id="meta-llama/Meta-Llama-3.1-70B-Instruct", ignore_patterns=["*.bin", "*.pth"], local_dir="./Llama-3.1-70B-Instruct/", local_dir_use_symlinks=False)

The next step is to produce a measurement file containing the quantization accuracy for various bit-per-weight (bpw) for each component of the models.

mkdir ./Llama-3.1-70B-Instruct/temp/
python exllamav2/convert.py \
    -i ./Llama-3.1-70B-Instruct/ \
    -o ./Llama-3.1-70B-Instruct/temp/ \
    -nr \
    -om ./Llama-3.1-70B-Instruct/measurement.json \
    -rs 8.0

This step is the most time-consuming. You can find my measurement.json here.

“-rs” is the RoPE scaling factor of Llama 3.1 70B. If you forget it, quantization might be significantly worse.

Then, you can now quantize with the quantization precision of your choice:

python exllamav2/convert.py \
    -i ./Llama-3.1-70B-Instruct/ \
    -o ./Llama-3.1-70B-Instruct/temp/ \
    -nr \
    -m  ./Llama-3.1-70B-Instruct/measurement.json\
    -cf ./Llama-3.1-70B-Instruct/4.0bpw/ \
    -b 4.0 \
    -rs 8.0

This command quantizes Llama 3.1 70B Instruct with an average precision of 4 bits (“-b 4.0“). I did it also with 2.2, 2.5, and 3.0 bits.

You can find the 4-bit model here:

  • kaitchup/Meta-Llama-3.1-70B-Instruct-ExLlamaV2-4bit

The 2.2, 2.5, and 3.0-bit versions are also available on The Kaitchup’s HF space but I don’t recommend using them. As we will see in the next section, they don’t work well.

ExLlamaV2 vs. AWQ vs. GPTQ

I compared the perplexity of the ExLlamaV2 models with the perplexity of AWQ and GPTQ models made by Hugging Face:

Note: Here, we can compare the perplexities of these models because they are different versions of the same model, Llama 3.1 70B Instruct. They all use the same vocabulary and have the same maximum context length. If the models had been trained using different vocabularies, we would have needed to compare them using a different metric.

For reference, I also computed the perplexity of Llama 3.1 8B Instruct on the same datasets. You can see it as a baseline. If a quantized Llama 3.1 70B performs lower than Llama 3.1 8B while consuming more memory, it means that quantization is not worth it and that we should use a smaller model instead.

To compute the perplexity, I use the script provided by ExLlamaV2:

  • GitHub: tests/compare_quants.py

It computes the perplexity using samples from the Wikitext, C4, and FineWeb datasets. It also prints the maximum GPU VRAM consumed by the model for inference and the time needed to compute the perplexity.

The results:

Average speed (iteration per second, lower is better) for computing the perplexity:

  • Llama 3.1 8B Instruct FP16: 2.84

  • Llama 3.1 70B Instruct AWQ: 3.33

  • Llama 3.1 70B Instruct GPTQ: 6.19

  • Llama 3.1 70B Instruct EXL2 4.00 bpw: 3.9

  • Llama 3.1 70B Instruct EXL2 3.00 bpw: 3.86

  • Llama 3.1 70B Instruct EXL2 2.5 bpw: 3.82

  • Llama 3.1 70B Instruct EXL2 2.2 bpw: 3.81

Note: Some GPTQ kernels were not properly installed and I couldn’t fix it. It results in a slower inference with the GPTQ models.

AWQ and GPTQ models are significantly better (lower perplexity) than Llama 3.1 8B Instruct but they consume nearly 40 GB of GPU RAM. That’s 24 GB more than Llama 3.1 8B.

By leveraging mixed-precision quantization, ExLlamaV2 achieves a slightly worse perplexity than AWQ and GPTQ while consuming 4 GB less.

However, with lower precision, ExLlamaV2 fails to remain accurate. The model reaches perplexity far higher than Llama 3.1 8B while consuming much more memory.

The models are nearly useless. They are also slower than the AWQ model.

Conclusion

Should you use ExLlamaV2 for Llama 3.1 70B Instruct?

If you have a GPU with 40 GB of VRAM, quantizing the model with ExLlamaV2 to 4-bit seems a good option. It is almost as accurate as AWQ while occupying less memory. However, it is 1.16x slower.

For lower precision quantization, ExLlamaV2 doesn’t appear useful. We need a better quantization algorithm to run Llama 3.1 70B on one consumer GPU.

AQLM could be an option but the quantization is very costly. As far as I know, there isn’t an AQLM version of Llama 3.1 70B, yet. HQQ is also possible but will struggle to reduce the size of the model below 24 GB, even with 2-bit quantization, since it doesn’t quantize the embedding parameters. HQQ models also often require post fine-tuning.

Quantization with AutoRound is another option but again no one quantized the model with it, yet. I’ll try to do it but it might take some time.
