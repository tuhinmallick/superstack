With Command-R+, Mixtral-8x22b, and Llama 3 70B that were all released within a few weeks, we have now LLMs that perform more and more closely to the best GPT-4 models. However, these models are huge. They all have more than 70 billion parameters:

  • Command-R+: A 104B parameter model

  • Mixtral-8x22b: A mixture-of-expert (MoE) model with 141B parameters

  • Llama 3 70B: A model with 70.6B parameters

Can you fine-tune and run these models on your computer?

In this article, I explain and analyze their memory consumption for inference and fine-tuning. The method I present applies to any transformer LLMs to estimate their memory consumption without downloading them. We will see that, while the memory consumption of Command-R+, Mixtral-8x22b, and Llama 3 70B is huge, there are several techniques to significantly reduce it, such as quantization and memory-efficient optimizers.

Memory Required for Inference with Command-R+, Mixtral-8x22B, and Llama 3 70B

Loading the model

These three models are all distributed with 16-bit weights: float16 for Command-R+ and bfloat16 for Mixtral and Llama 3 70B. In other words, one parameter consumes 16 bits, or 2 bytes, of memory.

1 billion parameters occupy 2 billion bytes. 1 billion bytes is 1 gigabyte, meaning that 1B parameters occupy 2 GB of memory. 100B parameters occupy 200 GB of memory. Note: This is an approximation since 1 KB is not equal to 1,000 bytes but to 1,024 bytes.

To know how many parameters a model has without downloading it, you can check the model card:

If you want to do fast inference on GPUs, you will need to load the model entirely on the GPU RAM. It means that you will need:

  • For Command-R+: 193.72 GB of GPU RAM

  • For Mixtral-8x22B: 262.63 GB of GPU RAM

  • For Llama 3 70B: 131.5 GB of GPU RAM

In other words, you will need 2x80 GB GPUs, e.g., two H100s, to load Llama 3 70B, one more GPU for Command-R+, and another one for Mixtral.

Memory Consumption of Activations

Once the model is loaded, we need more memory to store the model’s activations, i.e., the tensors created during inference. These tensors are passed from one layer to the next one. Their size in memory is not straightforward to estimate.

To estimate the memory consumption of these activations, I use the method proposed in this paper:

Reducing Activation Recomputation in Large Transformer Models

We need to know the following to estimate the memory consumption:

  • s: The maximum sequence length (the number of tokens in the input)

  • b: The batch size

  • h: The hidden dimensions of the model

  • a: The number of attention heads

A standard transformer layer is composed of a self-attention block and an MLP block, each connected by two layer-norms. The memory consumption for each component is estimated as follows.

For the self-attention

The attention block consists of self-attention mechanisms and a linear projection. The memory requirements include:

Note: When I write “sbh”, read it as a multiplication “s*b*h”.

  • The linear projection retains its input activations, which have a size of 2sbh.

  • The linear projection and self-attention input activations, requiring 2sbh each.

  • Self-attention's Query (Q) and Key (K) matrices needing 4sbh combined.

  • Softmax requiring 2as²b.

  • Storage for the attention applied over Self-attention’s Values (V) adding up to 2as²b + 2sbh.

The total memory required for the attention block amounts to 10sbh + 4as²b.

For the MLP block

The MLP block involves two linear layers:

  • The linear layers store inputs which cost 2sbh and 8sbh, with the GeLU non-linearity also needing 8sbh. Note: We probably don’t need to store GeLU’s activations for inference but I’ll still count them to give an estimation of the memory consumption at worst, in case the framework used for inference stores these activations.

Overall, the MLP block demands 18sbh of storage.

For the layer-norms

Each layer-norm stores its input, needing 2sbh each, summing to 4sbh for the two norms.

For an entire layer

The total memory requirement to store activations for one transformer layer totals 10sbh + 4as²b (from the attention block) + 18sbh (from the MLP block) + 4sbh (from the layer norms).

Activation memory consumption per layer = 32 sbh + 4as²b

If we use a 16-bit data type, which is common for activations, we have to multiply this number by 2 since each activation parameter would cost 2 bytes.

This equation roughly approximates the real memory consumption of the activations. Most inference frameworks are optimized to significantly reduce it by removing tensors as soon as they are useless rather than keeping them until the layer is fully processed. On the other hand, various buffers consuming memory are also created during inference. Nonetheless, in my experiments, I found it yields a good approximation with frameworks such as Hugging Face Transformers. With other frameworks that are much more optimized, such as vLLM and TGI, the memory consumption is greatly reduced.

Reducing the Memory Consumption for Inference

Most of the memory consumption for inference comes from the model’s parameters. Recent quantization algorithms can significantly reduce this memory consumption. They compress the model by reducing the bit width of most parameters while trying to preserve its accuracy.

8-bit quantization is almost lossless while 4-bit quantization only slightly degrades the performance.

AQLM for instance shows good performance for 2-bit quantization applied to Mixtral-8x7B. The problem with AQLM is that quantizing the model is very costly. It can take several weeks in the cloud for very large models.

Another alternative is to move the model to another memory device, for instance, the CPU RAM. The downside is that it slows down inference, especially with batch decoding. Frameworks optimized for CPU inference help to keep inference reasonably fast. Neural Speed for instance is one of the fastest frameworks for inference on CPU with quantized models.

Memory Required for Fine-tuning Command-R+, Mixtral-8x22B, and Llama 3 70B

For fine-tuning LLMs, estimating the memory consumption is slightly more complicated. In addition to storing the model weights and activations, for all layers, we also need to store the optimizer states.

Memory Consumption of the Optimizer States

The AdamW optimizer, which is the most popular for fine-tuning LLMs, creates and stores 2 new parameters for each parameter of the model. If we have a 100B model, the optimizer will create 200B new parameters!

Moreover, for better training stability, the optimizers’ parameters are float32, i.e., each parameter occupies 4 bytes of memory.

Moreover, it creates a copy of the model’s parameters as float32 and the gradients. Gradients are usually 16-bit parameters when training with mixed-precision.

The optimizer alone is the main reason why fine-tuning consumes more memory than inference.

For instance, for Mixtral-8x22B, the optimizer creates 2*141B = 282B float32 parameters in addition to the copy of the model’s parameter, 141B float32, and the gradients, 141B float16. It consumes 1,053.53 GB of memory to which we have to add the memory occupied by the model itself, i.e., 262.63 GB. In total, we need 1,315.63 GB of GPU memory. That’s roughly seventeen 80 GB GPUs!

And then, that’s not nearly enough to fine-tune the model. We also need memory to store the activations.

Memory Required to Compute the Gradients

In contrast with inference for which we only need to store activations for a single layer before passing them to the next one, fine-tuning requires storing all the activations created during the forward pass. This is necessary to compute the gradients which are then used to backpropagate the errors and updates the model’s weights.

To estimate the memory necessary to compute the gradients, we can use the same formula used for inference and then multiply the result by the number of layers. We also need to take into account the dropout masks for the self-attention and MLP modules, which add 2 sbh.

If L is the number of layers, then the memory consumed to compute the gradients is

L(34sbh + 5as²b)

Reducing the Memory Consumption for Fine-tuning

Since the optimizer states consume a huge amount of memory, significant research has been done to reduce their memory footprint, e.g.:

  • LoRA: Freezes the entire model and adds a trainable adapter with a few million parameters. With LoRA, we only store the optimizer states for the adapter’s parameters.

  • QLoRA: LoRA but with the model quantized to 4-bit or lower precision.

Conclusion

In this article, we saw how to estimate the memory consumption of transformer models. This estimation is not trivial and the method I described can only approximate it while disregarding any optimizations of the frameworks used.

This method doesn’t work for models using other architectures than the transformer. For instance, the memory consumption of the activations for Mamba and RWKV is significantly lower since they don’t use the attention mechanism of the transformer.
