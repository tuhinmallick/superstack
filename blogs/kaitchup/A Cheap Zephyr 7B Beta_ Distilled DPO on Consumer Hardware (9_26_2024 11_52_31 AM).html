While Zephyr 7B Beta was a relatively cheap model to make thanks to DPO and distillation, Hugging Face still needed 16 A100 80 GB GPUs for a few hours to train it. Note: In the cloud, the cost of training Zephyr would be a few hundred dollars.

In this article, we will see how to turn Mistral 7B into a Zephyr 7B Beta on consumer hardware using a parameter-efficient training method and quantization (QLoRA). I also adapt the training data made and used by Hugging Face, along with the training hyperparameters, to speed up training and reduce memory consumption.

Hardware Requirements and Settings

I use quantization and LoRA to reduce the hardware requirements. Mistral 7B quantized can be loaded on a GPU with 6 GB of VRAM. Since DPO exploits two models simultaneously, the SFT model and the model trained with DPO, I recommend at least 12 GB of VRAM to load the models. Then, we need more VRAM for the training batches. We will use small batches for SFT and DPO that will require an additional 2 GB (more or less) of VRAM.

We need a GPU with at least 15 GB of VRAM. I used the V100 of Google Colab to run the notebook. Using an NVIDIA RTX 4060 with 16 GB of VRAM is a good alternative for a consumer GPU under $500. An even better alternative would be the RTX 4090 which has 24 GB of VRAM.

Loading and quantizing Mistral 7B

Nothing special here, we have done it many times in previous articles:

model_name = "mistralai/Mistral-7B-v0.1"
tokenizer = AutoTokenizer.from_pretrained(model_name, add_eos_token=True, use_fast=True)
tokenizer.pad_token = tokenizer.unk_token
tokenizer.pad_token_id =  tokenizer.unk_token_id

compute_dtype = getattr(torch, "float16")

Training and Validation Datasets

Hugging Face used its own filtered version of Ultrachat to fine-tune Mistral 7B:

  • HuggingFaceH4/ultrachat_200k (use the “sft” splits)

They have also fixed grammatical errors and casing as they observed that it can have a significant impact on the results.

According to the technical report, this is how they fully fine-tuned Mistral 7B on this dataset:

We use a cosine learning rate scheduler with a peak learning rate of 2e-5 and 10% warmup steps. We train all models with a global batch size of 512 and use packing with a sequence length of 2048 tokens.

This configuration consumes a huge amount of VRAM. Since they fully fine-tuned Mistral 7B, it would require around 15 GB just to load the model. But this is nothing compared to the VRAM consumed by the training batches.

Batch sizes

The training batches contain 512 sequences of 2048 tokens. They are huge. Ultrachat contains very long sequences of tokens. A sequence length of 2048 is thus justified, but we can’t afford it on consumer hardware.

We can use LoRA for parameter-efficient fine-tuning instead of fully fine-tuning Mistral 7B. We won’t need a large batch size for LoRA. I set it to 16 which is rather standard for LoRA fine-tuning.

Sequence length

Moreover, each example in Ultrachat is a multi-turn dialogue. One way to reduce the size of the training examples is to use only the first dialogue turns.

def format_ultrachat(ds):
  text = []
  for row in ds:
    if len(row['messages']) > 2:
      text.append("### Human: "+row['messages'][0]['content']+"### Assistant: "+row['messages'][1]['content']+"### Human: "+row['messages'][2]['content']+"### Assistant: "+row['messages'][3]['content'])
    else: #not all dialogues have more than one turn
      text.append("### Human: "+row['messages'][0]['content']+"### Assistant: "+row['messages'][1]['content'])
  ds = ds.add_column(name="text", column=text)
  return ds
dataset_train_sft = load_dataset("HuggingFaceH4/ultrachat_200k", split="train_sft")
dataset_test_sft = load_dataset("HuggingFaceH4/ultrachat_200k", split="test_sft[:5%]")

dataset_test_sft = format_ultrachat(dataset_test_sft)
dataset_train_sft = format_ultrachat(dataset_train_sft)

The sequences are saved into a new column “text” in the dataset object. We will only use this column during training.

I use the “test” split to compute the validation loss during training. I only use 5% of this split to speed up validation.

LoRA

For the configuration of LoRA, I use lora_alpha=rank=16 as usual while targeting the most important attention and MLP modules.

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', 'gate_proj', 'down_proj', 'up_proj']
)

If you want to reduce the VRAM consumption, you may only target the MLP modules instead:

target_modules= ["gate_proj", "down_proj", "up_proj"]

Training steps

Then we have to decide how long we want to train Mistral 7B. Here it only depends on your budget… Hugging Face trained for one epoch. They tried to train for 2 and 3 epochs but didn’t see any positive difference in the final DPO results.

However, one epoch takes a long time and could be too costly.

If you use the same batch size configuration that I used, I recommend training for at least 3000 steps.

Learning rate, warm-up, scheduler, and optimizer

For the learning rate, Hugging Face used 2e-5 which I found to be good enough. I tried higher learning rates (>1e-4) but the training didn’t seem to converge.

Hugging Face used a cosine scheduler for SFT while they used a linear scheduler for DPO. They don’t explain this difference.

Given that the linear scheduler performs almost always better than cosine, I used a linear one for SFT, with 10% warm-up steps (the same as Hugging Face).

As for the optimizer, for memory efficiency, I used the paged AdamW 8-bit. If you have enough VRAM, use the standard AdamW instead which is much faster.

Distilled DPO

Loading and quantizing the models

DPO uses two models: the model trained with SFT (often called the “reference model”) and the model to be trained with DPO.

With Hugging Face’s TRL, we can train DPO with both models quantized and with LoRA adapters. It saves a lot of memory.

First, load the model to be trained with DPO, Mistral 7B:

model_name = "mistralai/Mistral-7B-v0.1"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_name, use_fast=True)
tokenizer.pad_token = tokenizer.unk_token
tokenizer.pad_token_id =  tokenizer.unk_token_id
tokenizer.padding_side = 'left'

compute_dtype = getattr(torch, "float16")
bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, quantization_config=bnb_config, device_map={"": 0}
)
model = prepare_model_for_kbit_training(model)
#Configure the pad token in the model
model.config.pad_token_id = tokenizer.pad_token_id
model.config.use_cache = False

Then load the reference model trained with SFT:

model_ref_name = "mistralai/Mistral-7B-v0.1"
bnb_config_ref = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model_ref = AutoModelForCausalLM.from_pretrained(
          model_ref_name, quantization_config=bnb_config, device_map={"": 0}
)
model_ref = PeftModel.from_pretrained(model_ref, "./results/checkpoint-3000/")

Replace “./results/checkpoint-3000/” with the directory containing the adapter that you fine-tuned with SFT.

Training and Validation Datasets

For DPO, Hugging Face used the UltraFeedback that they preprocessed to select a “chosen” output and a “rejected” output:

  • HuggingFaceH4/ultrafeedback_binarized (use the “prefs” splits)

Similarly to Ultrachat, Ultrafeedback’s examples are very long dialogues formatted with JSON. Hugging Face didn’t reveal yet how they dealt with JSON.

DPO expects three columns in the dataset:

  • prompt

  • chosen

  • rejected

I took the prompt column of the dataset and wrapped it with “### Human:” and “### Assistant:” tags to obtain a format similar to what I used for SFT. Note: In my previous article on DPO, I didn’t do this wrapping for the prompt but observed later that wrapping the tags improves the results.

Then for “chosen” and “rejected”, I only used the first answer given by the assistant.

Here is the processing code:

def format_ultrafeedback(ds):
  text = dict()
  text['chosen'] = []
  text['rejected'] = []
  prompt = []
  for row in ds:
    prompt.append("### Human: "+row['prompt']+"### Assistant: ")
    for col in ['chosen','rejected']:
      text[col].append(row[col][1]['content'])
  ds = ds.rename_column("chosen", "chosen_json")
  ds = ds.rename_column("rejected", "rejected_json")
  ds = ds.rename_column("prompt", "prompt_text")

  ds = ds.add_column(name="chosen", column=text['chosen'])
  ds = ds.add_column(name="rejected", column=text['rejected'])
  ds = ds.add_column(name="prompt", column=prompt)
  return ds
dataset_train_dpo = load_dataset("HuggingFaceH4/ultrafeedback_binarized", split="train_prefs")
dataset_test_dpo = load_dataset("HuggingFaceH4/ultrafeedback_binarized", split="test_prefs[:20%]")

dataset_test_dpo = format_ultrafeedback(dataset_test_dpo)
dataset_train_dpo = format_ultrafeedback(dataset_train_dpo)

Training Hyperparameters

For the training hyperparameters, Hugging Face used the same that they used for Zephyr 7B Alpha:

We use a linear learning rate scheduler with a peak learning rate of 5e-7 and 10% warmup steps. We train all models with a global batch size of 32 and use β = 0.1

As I discussed in my previous article using DPO, I recommend using the same. Note that if you have a GPU with 16 GB VRAM, using a batch size of 2 and gradient_accumulation_steps=16 (total batch size=32) might trigger an out-of-memory error. To avoid it, I use a batch size of 1 and gradient_accumulation_steps=32 which is equivalent but uses less memory.

LoRA

I use the same one that I used for SFT. Again, you can remove some target modules if you need to save some VRAM.

Training steps

One of the main findings reported by Hugging Face in the technical report is that overfitting with DPO helps.

They observed that after one epoch DPO starts overfitting the training data. It almost reaches perfect accuracy as shown in the figure below:

Despite this overfitting, the model trained for 3 epochs is the one that performs the best on MT Bench and AlpacaEval.

We can’t afford to train DPO for 3 epochs on consumer hardware. It would take many days.

In the technical report, Hugging Face also shows that training SFT and DPO for one epoch still achieves excellent results. See the first cell in this figure:

I think skipping 2 epochs of DPO training for only a 0.34 points drop on MT Bench is a good trade-off.

But again, training DPO for 1 epoch is still very long on consumer hardware.

If you can’t afford it, just train DPO for as long as possible. I recommend a minimum of at least 5000 training steps. It would take around 3 days with an RTX 4060 GPU.

Conclusion

Training a cheap Zephyr is possible on consumer hardware but expect to train your models for several days before getting reasonably good results.
