FlashAttention is a popular method to optimize the attention computation in the Transformer. It can significantly accelerate inference and fine-tuning for large language models (LLM).

FlashAttention is now implemented in many frameworks, including Hugging Face Transformers and Unsloth, and supports most of the recent LLMs.

FlashAttention in a nutshell

FlashAttention is described in the following arXiv paper:

FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness

It can be summarized by this figure from the paper:

The main idea behind FlashAttention is to perform as much as we can of the attention computation on the SRAM of the GPU (the summit of the pyramid in the figure above). The SRAM is an on-chip memory that is much faster than the GPU HBM/DDR RAM (the one that we usually call the “VRAM”) but also much more expensive and thus available only in very small quantities (usually less than 100 MB).

If you remove the fan from your graphic card, you will see the memory modules which are usually black chips surrounding the GPU chip. On the other hand, you won’t be able to locate the SRAM since it is physically integrated into the GPU chip. This is because it is physically in the GPU chip that the SRAM is much faster than the HBM memory.

FlashAttention decomposes the attention computation into small blocks that can be loaded on the SRAM. In other words, it avoids writing the large attention matrix on the HBM/DDR. This is illustrated by the middle part of the figure above.

The author observed a 7x speed up in the attention computation.

FlashAttention was proposed in May 2022. It’s now well supported by most deep-learning frameworks and has been improved with FlashAttention-2.

FlashAttention-2

You can find a description of all the optimization brought by FlashAttention-2 in this arXiv paper:

FlashAttention-2: Faster Attention with Better Parallelism and Work Partitioning

Reducing the number of non-matmul operations

FlashAttention-2 further speeds up attention computation by reducing the number non-matmul operations.

But what’s a matmul operation?

When we train and run LLMs, the GPUs perform a huge number of matrix multiplications, a.k.a, matmul operations.

Recent GPUs have cores dedicated to speed up their computation. For instance, the Tensor Cores of NVIDIA GPUs are specialized in computing matmul operations. Tensor cores have been introduced with the RTX 20xx generation but more recent GPUs, such as the RTX 40xx, have more Tensor cores and are faster. Nonetheless, note that FlashAttention is only supported by Ampere GPUs (RTX 30xx) or newer.

Making matmul operations instead of non-matmul operations can make a huge speed difference. For the A100 GPU, FlashAttention’s author describes it as follows:

As an example, the A100 GPU has a max theoretical throughput of 312 TFLOPs/s of FP16/BF16 matmul, but only 19.5 TFLOPs/s of non-matmul FP32. Another way to think about this is that each non-matmul FLOP is 16x more expensive than a matmul FLOP. To maintain high throughput, we want to spend as much time on matmul FLOPs as possible.

Improved parallelization for long sequences

The first version of FlashAttention is optimized for parallel computing over batch size and attention heads. It works especially well for large batches. However, there are various scenarios in which we can’t make large batches, for instance, when we deal with long sequences of tokens.

FlashAttention-2 now also parallelizes over the sequence length. Consequently, even when using small batch sizes with long sequences, we can still benefit from FlashAttention.

Improved partitioning and support for more LLMs

FlashAttention-2 does also a better job at partitioning work between computing threads. It reduces the amount of communication and synchronization between the threads (or more precisely, between “warps”, which are groups of 32 threads).

FlashAttention-2 is also now supported by models with up to 256 heads. Models using multi-query attention (MQA) as well as grouped-query attention (GQA) can also benefit from FalshAttention-2.

How to Use FlashAttention-2 with Mistral 7B and Llama 2

The simplest way to use FlashAttention-2 is through Hugging Face Transformers.

But first, let’s check that your GPU is compatible by running this code:

import torch

major_version, minor_version = torch.cuda.get_device_capability()
if major_version >= 8:
  print("GPU is compatible with FlashAttention")
else:
  print("GPU is not compatible with FlashAttention")

FlashAttention only works with NVIDIA Ampere GPUs and more recent GPUs. For instance, if you have GPUs from the RTX 30xx or 40xx generation, it will work. On Google Colab, it works only with the A100.

If your GPU is compatible, you can run the code described in the remainder of this article.

First, you need to install flash-attn:

pip install -q -U flash-attn

Then, when you load the model with Transformers, you only need to set “attn_implementation='flash_attention_2'”:

model = AutoModelForCausalLM.from_pretrained(
          "meta-llama/Llama-2-7b-hf", torch_dtype=torch.bfloat16,  attn_implementation='flash_attention_2'
)

Or, with Mistral 7B:

model = AutoModelForCausalLM.from_pretrained(
          "mistralai/Mistral-7B-v0.1", torch_dtype=torch.bfloat16, attn_implementation='flash_attention_2'
)

FlashAttention-2 will be automatically used for the loaded model. There is nothing else to do.

Note that the model must be loaded with bf16 (or fp16) or quantized when loaded (with load_in_8bit=True or load_in_4bit=True).

Faster QLoRA Fine-tuning with FlashAttention-2

I tried QLoRA fine-tuning for Llama 2 with and without FlashAttention-2. Note: I chose Llama 2 because the acceleration obtained with FlashAttention is very significant with this model. However, if you use Mistral 7B, unless you deal with very long sequences of tokens, e.g., more than 4096 tokens, you won’t notice a significant difference.

I loaded the model as follows (full code in the notebook):

compute_dtype = getattr(torch, "bfloat16")
bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, quantization_config=bnb_config, device_map={"": 0}, attn_implementation='flash_attention_2'
)

Moreover, when setting up your tokenizer, set the padding side to “left”:

tokenizer.padding_side = 'left'

FlashAttention-2 is not compatible with right-padding.

Remember that FlashAttention-2 parallelizes attention computation over:

  • Batch size

  • Sequence length

  • Attention heads

The number of attention heads is fixed by the model architecture but we can adjust the batch size and the sequence length to better exploit FlashAttention.

With Hugging Face’s TRL, I think the best way to exploit FlashAttention is to use the “packing” option:

SFTTrainer supports example packing, where multiple short examples are packed in the same input sequence to increase training efficiency.

In other words, “packing” extends the length of the training examples by concatenating several examples until the maximum sequence length is reached. It suits particularly well FlashAttention-2 thanks to its optimization for the computation of attention over long sequences.

Then, adjust the training batch size to have batches as large as your GPU can handle.

For the A100 in Google Colab, I use these settings (FlashAttention is mainly impacted by the arguments in bold):

training_arguments = TrainingArguments(
        output_dir="./llama2_with_FA2",
        evaluation_strategy="steps",
        do_eval=True,
        per_device_train_batch_size=4,
        per_device_eval_batch_size=4,
        log_level="debug",
        save_steps=20,
        logging_steps=20,
        learning_rate=4e-4,
        eval_steps=20,
        bf16=True,
        max_steps=200,
        warmup_steps=10,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=dataset['train'],
        eval_dataset=dataset['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=2048,
        tokenizer=tokenizer,
        args=training_arguments,
        packing=True
)

I ran two QLoRA fine-tuning with this configuration: one with FlashAttention activated and the other one without FlashAttention (i.e., not setting “use_flash_attention_2=True”).

Here are the interesting numbers showing how much faster QLoRA is with FlashAttention-2:

Pytorch’s SDPA is the default implementation used by Hugging Face Transformers if your GPU is compatible. “Eager” is the fallback strategy if the GPU is not compatible with SDPA.

Using this configuration, QLoRA fine-tuning with FlashAttention-2 is 1.13x faster than SDPA. It looks like a modest improvement but on long fine-tuning runs, it can shorten fine-tuning by several days.

Conclusion

Unless your GPU doesn’t support it, I can’t see any reason for not activating FlashAttention-2. However, expect only a small impact if you use small batch sizes and short sequences.

In Hugging Face Transformers, it is as simple as setting attn_implementation='flash_attention_2'.
