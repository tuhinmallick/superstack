Meta only released two versions of Llama 3: 8B and 70B. Another version of Llama 3, called Llama 3 120B, has also appeared and has drawn a lot of discussions on social networks for its surprising performance in specific tasks such as creative writing.

This “Llama 3 120B” is not a product of Meta but a version made by Maxime Labonne with Mergekit. In a previous article, I presented Mergekit and showed how to use it to create powerful LLMs by simply merging several LLMs into one:

·

Read full story

However, Llama 3 120B is not a merge of several LLMs. It has been made from a single LLM, Llama 3 70B, by simply duplicating 60 of its 80 layers. Yet, despite its simplicity, this process seems to have worked relatively well on Llama 3 70B.

In this article, I explain how to reproduce Llama 3 120B. I show how to use the same process for Llama 3 8B and evaluate the resulting models. Going further than just duplication, I also show how to reorder and remove layers, e.g., to reduce the size of an LLM with mergekit. We will see that while most of these operations damage the model’s performance, it is possible to recover some of it through fine-tuning.

Making Llama 3 120B

Llama 3 120B was made with mergekit. The configuration used to make the model is also provided in the Llama 3 120B’s repository:

  • mergekit_config.yml

slices:
- sources:
  - layer_range: [0, 20]
    model: meta-llama/Meta-Llama-3-70B-Instruct
- sources:
  - layer_range: [10, 30]
    model: meta-llama/Meta-Llama-3-70B-Instruct
- sources:
  - layer_range: [20, 40]
    model: meta-llama/Meta-Llama-3-70B-Instruct
- sources:
  - layer_range: [30, 50]
    model: meta-llama/Meta-Llama-3-70B-Instruct
- sources:
  - layer_range: [40, 60]
    model: meta-llama/Meta-Llama-3-70B-Instruct
- sources:
  - layer_range: [50, 70]
    model: meta-llama/Meta-Llama-3-70B-Instruct
- sources:
  - layer_range: [60, 80]
    model: meta-llama/Meta-Llama-3-70B-Instruct
merge_method: passthrough
dtype: float16

This configuration takes different ranges of layers from the “model”, Llama 3 70B instruct, and then stacks them in the same order as they are written in the configuration file. This is the “passthrough” method. The layers are naively copied from the model and not modified.

For instance, the first layer range is [0,20]. It means that the model will first have the layers from id 0 to 19. Then, we have the layer range [10, 30] meaning that the following layers will be the layers 10 to 29 from Llama 3 70B.

Since we took [0,20] and then [10, 30], the layers from 10 to 19 will be duplicated. To make Llama 3 120B, all the layers are duplicated except the first 10 layers and the last 10 layers.

In total, the model has 140 layers. To make it with mergekit, we run:

mergekit-yaml mergekit_config.yml ./llama-3-120B --cuda --lazy-unpickle --allow-crimes

Since layers are only copied, creating the model is very fast. It can also be done using only the CPU if you remove “- - cuda“.

Does this make a good model?

Usually, the resulting model is much worse than the original one. Remember that we only duplicate layers with this method. In this new model, layer 19 from Llama 3 70B is now followed by layer 10. Layer 10 was not trained to receive the output of layer 19. This could be bad.

Somehow Llama 3 120B is better than Llama 3 70B at some tasks, especially in creative writing. In my opinion, this is this particular skill of Llama 3 120B that made this model very popular. See examples here and here.

The model has not been fine-tuned but I assume that a fine-tuning would significantly improve it since fine-tuning would teach the layers how to deal with the input/output of their new surrounding layers.

Duplicate, Remove, and Reorder Layers of Llama 3 8B

With mergekit and its passthrough method, we are not limited to just duplicate layers. We can prune layers or even reorder layers from the same model. We can also use layers from different models.

For this article, out of curiosity, I tried different configurations for Llama 3 8B:

  • Duplicating only the first layers, the last layers, or all the layers

  • Removing the layers from the middle

  • Reordering layers arbitrarily

Then, I fine-tuned the resulting model to see how difficult it was to recover the lost accuracy. I did the fine-tuning with QLoRA 8-bit.

The fine-tuning method:

import torch, os
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    TrainingArguments,
)
from trl import SFTTrainer

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'


base_model = "meta-llama/Meta-Llama-3-8B"
tokenizer = AutoTokenizer.from_pretrained(base_model,add_eos_token=True)
tokenizer.pad_token = tokenizer.eos_token
tokenizer.padding_side = 'left'

def sft(model_dir, model_output=''):
  if model_output=='':
    model_output = model_dir+'/sft/'
  model = AutoModelForCausalLM.from_pretrained(
            model_dir, load_in_8bit=True, device_map={"": 0}, attn_implementation=attn_implementation
  )

  model = prepare_model_for_kbit_training(model)
  ds = load_dataset("timdettmers/openassistant-guanaco")


  peft_config = LoraConfig(
          lora_alpha=16,
          lora_dropout=0.05,
          r=16,
          bias="none",
          task_type="CAUSAL_LM",
          target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
  )

  training_arguments = TrainingArguments(
          output_dir=model_output,
          evaluation_strategy="steps",
          do_eval=True,
          optim="paged_adamw_8bit",
          per_device_train_batch_size=4,
          gradient_accumulation_steps=4,
          per_device_eval_batch_size=4,
          log_level="debug",
          save_steps=50,
          logging_steps=25,
          learning_rate=1e-4,
          fp16 = not torch.cuda.is_bf16_supported(),
          bf16 = torch.cuda.is_bf16_supported(),
          eval_steps=25,
          max_steps=100,
          warmup_ratio=0.1,
          lr_scheduler_type="linear",
  )

  trainer = SFTTrainer(
          model=model,
          train_dataset=ds['train'],
          eval_dataset=ds['test'],
          peft_config=peft_config,
          dataset_text_field="text",
          max_seq_length=512,
          tokenizer=tokenizer,
          args=training_arguments,
  )

  trainer.train()

The resulting model has 37 layers.

It significantly degraded the model with a higher training loss and a higher validation loss. In one of my experiments, I forgot to include the last layer of Llama 3 8B and used a layer_range of [27, 31] instead of [27, 32]. In this case, the results were much worse, as expected.

It is better than the previous combination but still far from the original model.

Overall, it seems that duplicating fewer layers results in better models… Which is not surprising given how naive this method is.

Layer Removal

I also tried to remove layers from Llama 3 8B using this method. You can see it as a naive layer-pruning method that we could use to aggressively and quickly reduce the size of the model. I nearly removed one-third of the model:

slices:
- sources:
  - layer_range: [0, 10]
    model: meta-llama/Meta-Llama-3-8B
- sources:
  - layer_range: [21,32]
    model: meta-llama/Meta-Llama-3-8B
merge_method: passthrough
dtype: float16

I removed the layers in the middle rather than the first and last layers since these layers in the middle are usually less important to transformer models.

As expected, the results are bad. Even though we might still think a 2.5 validation loss is OK, the results on the Evaluation Harness say that the model is useless with very low accuracy on the benchmarks I tried:

Layer Reordering

In this last experiment, I reorder the layers by moving the first 4 layers after the 9th layer:

slices:
- sources:
  - layer_range: [5, 9]
    model: meta-llama/Meta-Llama-3-8B
- sources:
  - layer_range: [0,4]
    model: meta-llama/Meta-Llama-3-8B
- sources:
  - layer_range: [10,32]
    model: meta-llama/Meta-Llama-3-8B
merge_method: passthrough
dtype: float16

It yielded the worst results of all my experiments:

The text generated by the model is gibberish, even after fine-tuning.

Conclusion

The passthrough method doesn’t work well with small models but I also suspect it doesn’t work well either with large models such as Llama 3 70B. Llama 3 120B is a really lucky attempt to extend Llama 3 70B.

Even with fine-tuning, it doesn’t seem possible to easily recover the lost accuracy.
