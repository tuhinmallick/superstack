Along with Llama 3 405B, Meta also released new versions of Llama 3 8B and 70B (“Llama 3.1”). You can find them here:

  • Llama 3.1 Collection

The main differences with Llama 3 include the official support of German, French, Italian, Portuguese, Hindi, Spanish, and Thai, along with function calling. These new versions have been post-trained on very long sequences. They can handle contexts of up to 128k tokens without a noticeable accuracy drop.

Llama 3 vs. Llama 3.1: How Are They Different?

Llama 3.1 is presented in this report by Meta:

The Llama 3 Herd of Models

In this report, they renamed “Llama 3.1” to “Llama 3” for brevity. It makes the differences between Llama 3.1 and Llama 3 difficult to identify.

New capabilities of Llama 3.1

Llama 3.1 has been further trained on more multilingual data, to officially support more languages, and on function calling datasets.

Llama 3.1 has also been fine-tuned (last steps of pre-training) on sequences of 128k tokens to handle long contexts. This kind of fine-tuning is very complex.

Naive fine-tuning on long sequences degrades the model performance on short sequences. Unfortunately, Meta does not provide much detail on the technique they used for this fine-tuning. However, they mention that the long context data contained 800B tokens. Since Llama 3.1 has been pre-trained on 15.6T, it means that 5% of the pre-training was dedicated to learning how to model long contexts.

As we can see, the “content” of some of the special tokens is different. Llama 3.1 introduced “<|finetune_right_pad_id|>”, “<|eom_id|>”, and “<|python_tag|>”. There are no other modifications.

“<|eom_id|>” and “<|python_tag|>” can be used for function calling. <|finetune_right_pad_id|> could be a padding token. Llama 3 didn’t have one so it’s good news that facilitates fine-tuning. However, why there is “right” in the “content” of the token is a mystery to me. Right or left padding shouldn’t make a difference for fine-tuning unless right and left padding are not implemented the same in the model or the fine-tuning pipeline.

In my fine-tuning experiments below, I tried both padding directions.

The main differences are in how we will handle the padding token. In my fine-tuning code of Llama 3, I used the “eot_id” token for padding while padding left. For Llama 3.1, since Meta named a special token for padding, we will use it.

I ran LoRA and QLoRA fine-tuning twice, only changing the padding direction between each fine-tuning run. I did the same with the previous version of Llama 3. Note: All these experiments used the same seed and the same machine to guarantee that the model sees the training examples in the same order and that the training is also initialized the same.

The entire training code is in the notebook.

Here are the learning curves drawn from the validation runs for LoRA and QLoRA:

Main observations:

  • Llama 3.1 learns faster and better than Llama 3.

  • Padding on the right side consistently gave better results than padding on the left side, with Llama 3 and Llama 3.1.

  • QLoRA fine-tuning yields a lower loss than LoRA. It’s not the first time I observed this. This is counter-intuitive as we would expect QLoRA’s quantization to lead to worse results. I assume this is due to the language modeling head and token embeddings not being cast to float32 in my LoRA fine-tuning code, i.e., they remain in bfloat16 which is less precise.

The impact of the padding side is surprising to me. Padding shouldn’t have an effect. Right or left, the model should ignore padding tokens the same and not be affected by the padding direction. The difference we observe between left and right might be due to different implementations and optimizations, in FlashAttention, TRL, or Llama 3 itself. I suspect it comes from Meta's implementation of Llama 3 in Transformers. By inserting “right” in “<|finetune_right_pad_id|>”, Meta gave a hint that Llama 3 is meant to be padded right.

Conclusion

Llama 3.1 learns better than Llama 3. There is probably no more reason to use this previous version of Llama 3. Given the results of my experiments, I also recommend padding on the right side for fine-tuning.
