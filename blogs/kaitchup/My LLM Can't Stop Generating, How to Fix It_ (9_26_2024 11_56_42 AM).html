My LLM Can't Stop Generating, How to Fix It?

Detect and fix issues with the EOS token

After fine-tuning a large language model (LLM) on your data, you might observe that the model doesn’t know when to stop generating tokens. Although the first tokens answer your prompt appropriately, the model continues to produce irrelevant tokens until it reaches the maximum sequence length.

This is a very common issue.

This issue arises when the end-of-sequence (EOS) token is not properly configured. This can occur with models like Llama 3, Qwen2, and many other LLMs.

Properly configuring the EOS token for fine-tuning can be challenging and may require several iterations to find an effective solution.

In this article, I present and implement three simple tests to diagnose issues with the EOS token. Using Llama 3 as an example, we will see how to address each issue effectively. These tests and solutions are applicable to any generative LLM, ensuring you can resolve similar problems regardless of the model you're working with.

What Is the EOS Token?

The EOS token is a special token that triggers the end of the generation. Generative (auto-regressive) LLMs generate the tokens one by one. If the LLM generates the EOS token, the inference framework will stop the generation.

Recent LLMs have an EOS special token in their vocabulary. Some examples of EOS tokens:

  • Llama 3: <|end_of_text|>

  • Phi-3: <|endoftext|>

  • Llama 2: </s>

For instruct fine-tuning, i.e., turning a base LLM into a chat model, the EOS token must be inserted by the tokenizer of the LLM at the end of the user answers in the training data.

There are many ways to do this but not a standard method implemented by all LLMs and frameworks. That’s why correctly setting the EOS token can be time-consuming. A fine-tuning code of Llama 2 might not work well for Llama 3 just because of differences in the EOS token implementation.

I identified three common issues, related to the EOS token, that often occur when fine-tuning a new LLM with LoRA/QLoRA, leading to an endless generation:

  • The tokenizer doesn’t add the EOS token to the training data and thus will never be seen during fine-tuning.

  • The EOS token wasn’t pre-trained and thus will have the same embeddings as all the other special tokens that were not pre-trained.

  • The EOS token, if also set as the padding token, will be ignored during fine-tuning.

Understanding How an LLM is Pre-trained

Before debugging the EOS token, we should try to understand why pre-trained LLMs don’t have the EOS token ready to use for fine-tuning.

State-of-the-art LLMs are pre-trained on trillions of tokens from the Web. These tokens don’t include the EOS token since this is a special token manually added to the tokenizer.

When these pre-training data are prepared, they are not segmented into self-contained training examples. The pre-training data are fed to the LLM in batches. These batches contain consecutive tokens as they are found on the Web. It’s different from the batches used for instance for instruct fine-tuning.

For instruct fine-tuning, each training example is a sequence of tokens containing an instruction and its answers. These examples are “self-contained”. We want the LLM to learn where is the end of the answer. For instance, a batch can contain 32 training examples finished by an EOS token and padded to make sure that all the examples have the same length in this batch. The EOS token here means the end of a training example.

For pre-training, we only want the model to learn the probability of a token given all the previous n tokens in the batch. LLMs are pre-trained on hundreds if not thousands of GPUs. That’s enough memory to make batches of millions of tokens. We don’t need the LLMs to learn when to generate an EOS token at this stage. We only want the model to learn from as many tokens as possible in one batch.

Nonetheless, there is often an EOS token in the vocabulary, how is it pre-trained?

We don’t know. LLM makers usually don’t publish their pre-training recipe, even when they claim that their LLM is open-source. The EOS token may have been in the training data by chance, used only for a few training batches easy to annotate with an EOS token, or just inserted by default at the end of each training batch.

Now that we know why the EOS token might not be pre-trained, let’s see how to handle it for fine-tuning.

Test 1: Is add_eos_token supported by the tokenizer?

With Hugging Face Transformers, the simplest way to add the EOS token to the fine-tuning data is to pass add_eos_token=True when instantiating the tokenizer:

tokenizer = AutoTokenizer.from_pretrained(model_id, add_eos_token=True)

When the Trainer will prepare the data, the tokenizer will automatically add the EOS token.

Most LLMs implemented in Hugging Face Transformers support add_eos_token. This is not the case for Llama 3 and Qwen2.

Once an LLM is released, before fine-tuning it, I recommend checking that add_eos_token is supported. For instance, using this code:

from transformers import AutoTokenizer, set_seed
set_seed(1234)  # For reproducibility

prompt = "### Human: Hello!### Assistant: Hello!"
model_id = "meta-llama/Meta-Llama-3-8B"

tokenizer = AutoTokenizer.from_pretrained(model_id)
print("\nEOS token: "+tokenizer.eos_token)
print("Example of a tokenized sentence without add_eos_token:")
print(tokenizer.decode(tokenizer(prompt)['input_ids']))

tokenizer = AutoTokenizer.from_pretrained(model_id, add_eos_token=True)
print("\nExample of a tokenized sentence with add_eos_token:")
print(tokenizer.decode(tokenizer(prompt)['input_ids']))

if tokenizer.eos_token_id not in tokenizer(prompt)['input_ids']:
  print("\n\nThe EOS token hasn't been added. add_eos_token is not supported. Consider adding the EOS token manually.")
else:
  print("\n\nadd_eos_token is supported.")

It tokenizes a sentence with a tokenizer instantiated with add_eos_token. If after the tokenization there is not an EOS token in the sequence, it means that add_eos_token is not supported by the tokenizer.

For instance, with Llama 3 this code will print:

The EOS token hasn't been added. add_eos_token is not supported. Consider adding the EOS token manually.

While with Llama 2, it will print:

add_eos_token is supported.

If add_eos_token is not supported, we have to add the EOS token manually to all the training examples, as follows:

import multiprocessing
from transformers import AutoTokenizer, set_seed
from datasets import load_dataset
set_seed(1234)  # For reproducibility

model_id = "meta-llama/Meta-Llama-3-8B"

tokenizer = AutoTokenizer.from_pretrained(model_id)

ds = load_dataset("timdettmers/openassistant-guanaco", split='train')

#Add the EOS token
def process(row):
    row["text"] = row["text"]+tokenizer.eos_token
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

#print one tokenizer example to check that the EOS token has been added
print('\nExample of tokenized sequence:')
print(tokenizer.decode(tokenizer(ds['text'][0])['input_ids']))

Test 2: Has the EOS token been pre-trained?

LoRA and QLoRA fine-tuning doesn’t update the token embeddings. If the EOS token has not been pre-trained, i.e., its embeddings are 0 or extremely close to 0, the model won’t be able to use it. Worse, if the model has other tokens that were not pre-trained, the model might confuse the EOS token with other special tokens whose embeddings are almost identical.

To check whether this will be an issue, we can compute the norm of the token embeddings of the EOS token and compare it with other tokens. For instance, for Llama 3:

import torch
from transformers import AutoTokenizer, AutoModelForCausalLM, set_seed, BitsAndBytesConfig
from numpy import linalg as LA

set_seed(1234)  # For reproducibility

prompt = "The best tomato sauce is"

checkpoint = "meta-llama/Meta-Llama-3-8B"

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=torch.float16,
        bnb_4bit_use_double_quant=True,
)

tokenizer = AutoTokenizer.from_pretrained(checkpoint)
model = AutoModelForCausalLM.from_pretrained(checkpoint, quantization_config=bnb_config, torch_dtype=torch.float16, device_map="cuda")

print("\nCompute and print the norm for the BOS token, a random token, a reserved untrained token, and the EOS token:")

embeddings = model.model.embed_tokens.weight[128000]
print("BOS token")
print("Norm: "+str(torch.norm(embeddings)))

embeddings = model.model.embed_tokens.weight[12]
print("Some other token:")
print("Norm: "+str(torch.norm(embeddings)))

embeddings = model.model.embed_tokens.weight[128002]
print("Token <|reserved_special_token_0|>")
print("Norm: "+str(torch.norm(embeddings)))

embeddings = model.model.embed_tokens.weight[128001]
print("EOS token")
print("Norm: "+str(torch.norm(embeddings)))

will print:

BOS token
Norm: tensor(0.4592, device='cuda:0', dtype=torch.float16,
       grad_fn=<LinalgVectorNormBackward0>)
Some other token:
Norm: tensor(0.3625, device='cuda:0', dtype=torch.float16,
       grad_fn=<LinalgVectorNormBackward0>)
Token <|reserved_special_token_0|>
Norm: tensor(0., device='cuda:0', dtype=torch.float16,
       grad_fn=<LinalgVectorNormBackward0>)
EOS token
Norm: tensor(0.1903, device='cuda:0', dtype=torch.float16,
       grad_fn=<LinalgVectorNormBackward0>)

The EOS token of Llama 3 is pre-trained. There won’t be an issue with fine-tuning. If it weren’t pre-trained, its embeddings would have been close, or identical, to the embeddings of the reserved special tokens whose norm is 0.

In that case, with LoRA/QLoRA fine-tuning, we would have to fully fine-tune the token embeddings and language modeling head of the LLM to make sure that the model doesn’t confuse the EOS token with other not pre-trained tokens.

We saw how to do this in this article:

We need to add in the LoraConfig:

modules_to_save=["lm_head","embed_tokens"],

Note that the language modeling head, “lm_head”, and the token embeddings, “embed_tokens”, may have different names depending on the model architecture. You can find the names of these two modules by printing the model, print(model), with HuggingFace Transformers.

Since the EOS token of Llama 3 is pre-trained, we don’t need to do this for Llama 3. However, if you have enough GPU memory, I think it’s better to fully fine-tune the language modeling head and the token embeddings. This also considerably helps the model adapt to the fine-tuning data's domain and genre.

Test 3: Does padding interfere with the EOS token generation?

I explained “padding” in detail in this article:

Padding is necessary for fine-tuning but most recent LLMs don’t have a default pad token. We have to set a padding token before fine-tuning.

Most fine-tuning tutorials that you will find online use the EOS token as a padding token. It will work most of the time and is convenient since the EOS token is already a special token available in the vocabulary. Using the UNK token, when available, is also a good alternative.

Llama 3 doesn’t have a UNK token but has an EOS token. Following most tutorials, we could decide to use it as an EOS token. The first version of my notebook for fine-tuning Llama 3 also used the EOS token for padding:

tokenizer = AutoTokenizer.from_pretrained(model_name, use_fast=True)
#Don't do that!
tokenizer.pad_token = tokenizer.eos_token
tokenizer.pad_token_id = tokenizer.eos_token_id

This works with Llama 2 but won’t work well with Llama 3. Llama 3, once fine-tuned, will never generate the EOS token. You can see an example in the notebook.

The EOS token and the pad token are identical. The attention mask is properly created by the tokenizer (I checked), the EOS token is not masked and the pad tokens are masked. However, during training, it seems that the mask is not properly applied and the EOS token is always interpreted as a padding token. Consequently, fine-tuning ignores the EOS token.

We have to choose another token for padding.

I don’t have a good test to check whether using the EOS token for padding works for a given LLM. My recommendation is to never use the EOS token for padding, or any token that shouldn’t be ignored by the LLM.

Ideally, it should be a special token that was not pre-trained, e.g., a reserved special token or the eot_id token for Llama 3. The eot_id token of Llama 3 is used by Llama 3 Instruct but not by the base Llama 3. We can use it for padding.

tokenizer.pad_token = "<|eot_id|>"
tokenizer.pad_token_id = 128009

Fine-tuning Llama 3 to Generate the EOS Token

Finally, we have a fine-tuning code that teaches Llama 3 when to generate the EOS token:

import torch, os, multiprocessing
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig,
)
from trl import SFTTrainer, SFTConfig

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

model_name = "meta-llama/Meta-Llama-3-8B"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_name, use_fast=True)
tokenizer.pad_token = "<|eot_id|>"
tokenizer.pad_token_id = 128009
tokenizer.padding_side = 'left'

ds = load_dataset("timdettmers/openassistant-guanaco")

#Add the EOS token
def process(row):
    row["text"] = row["text"]+tokenizer.eos_token
    return row

ds = ds.map(
    process,
    num_proc= multiprocessing.cpu_count(),
    load_from_cache_file=False,
)

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
)

model = prepare_model_for_kbit_training(model, gradient_checkpointing_kwargs={'use_reentrant':True})


peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        modules_to_save=["lm_head","embed_tokens"],
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

training_arguments = SFTConfig(
        output_dir="./Llama3_8b_QLoRA_pad_embedlmhead/",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=8,
        gradient_accumulation_steps=4,
        per_device_eval_batch_size=8,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=100,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=100,
        num_train_epochs=3,
        warmup_ratio=0.1,
        dataset_text_field="text",
        max_seq_length=512,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

When using the fine-tuned adapter, it will generate the EOS token.

For instance, when prompted with:

### Human: Give me a list of 5 European countries.### Assistant:

The LLM generates:

 Here is a list of 5 European countries:

1. Germany
2. France
3. Italy
4. Spain
5. United Kingdom### Human: Can you sort them by area?### Assistant: Sure! Here is a sorted list of 5 European countries by area:

1. Russia
2. Ukraine
3. France
4. Spain
5. Italy<|end_of_text|>

The EOS token, “<|end_of_text|>”, is generated.

Conclusion

These three simple tests will be enough, for most LLMs, to identify and fix issues with the EOS token. You can systematically apply them for new LLMs that will be released. For instance, I used them when Qwen2 was released. I could check that it wasn’t supporting “add_eos_token” and that the EOS token was pre-trained.

If you have other issues with the EOS token, or if these tests are not enough to solve the endless generation of your LLM, let me know in the comments. I’ll try to help!
