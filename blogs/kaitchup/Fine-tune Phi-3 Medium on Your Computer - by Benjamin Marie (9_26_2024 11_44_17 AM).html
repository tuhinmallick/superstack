Microsoft’s Phi models, from Phi-1 to Phi-3 mini, had all been small models with less than 4 billion parameters. Microsoft even used to call Phi “SLMs” for “small language models”.

With the release of Phi-3 medium and its 14 billion parameters, Phi now also includes large models. According to Microsoft's evaluation of Phi-3, this medium version achieves a performance close to much larger and state-of-the-art large language models (LLM) such as Llama 3 70B.

Nonetheless, with its larger size, can we fine-tune Phi-3 medium on consumer hardware? Can quantization preserve its accuracy while sufficiently reducing its size?

In this article, we will answer these questions. I first briefly review Phi-3 medium and its architecture. Then, we will see how to fine-tune the model with QLoRA. Fine-tuning Phi-3 medium is possible on your computer if you have a GPU with 16 GB of RAM. It might even be possible on a 12 GB GPU with a small batch size.

Phi-3 Medium: What’s the Difference with the Mini?

Like the mini version, Phi-3 medium is an instruct LLM available in two versions optimized for different context lengths:

  • 4k: microsoft/Phi-3-medium-4k-instruct

  • 128k: microsoft/Phi-3-medium-128k-instruct

The 128k version has been created by applying LongRoPE to the 4k version.

The 4k version tends to be better on shorter sequences than the 128k version.

Phi-3 are instruct LLMs. They work much better when prompted with the chat format used during their instruct fine-tuning. The chat template is provided in the tokenizer. Phi-3 medium is very similar to the mini version. It’s a larger model with 40 attention heads, 40 layers, and an embedding dimension of 5120, yielding 14B parameters. The tokenizer, and thus the vocabulary, is the same. It has also been pre-trained on more tokens than the mini: 4.8T vs. 3.3T tokens.

On public benchmarks, Phi-3 medium appears better than larger LLMs:

It outperforms Command-R+, which has 90B more parameters, on most benchmarks. It’s also much better than GPT-3.5-turbo and slightly (but clearly) underperforms Llama 3 70B instruct.

These results make no sense to me. Microsoft has either found a recipe that everyone should use now to pre-train LLMs or the pre-training data were contaminated by the benchmark data. The training data were not released and the word “contamination” doesn’t appear in the technical report or the model card. So it’s impossible to verify what they did to guarantee that this evaluation is fair and correct.

Phi-3 are definitely very good models but probably not as good as the benchmarks tell us (and that’s fine since this is the case for most LLMs).

Fine-tuning Phi-3 medium with QLoRA

The fine-tuning code is very standard since Phi-3 uses the Llama architecture. We can use the same code as for fine-tuning Llama 3 except that we don’t need to manually add the EOS token to each training example since the tokenizer of Phi-3 supports “add_eos_token” :

Fine-tune Llama 3 on Your Computer

import torch, os
from datasets import load_dataset
from peft import LoraConfig, prepare_model_for_kbit_training
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig
)
from trl import SFTTrainer, SFTConfig

#use bf16 and FlashAttention if supported
if torch.cuda.is_bf16_supported():
  os.system('pip install flash_attn')
  compute_dtype = torch.bfloat16
  attn_implementation = 'flash_attention_2'
else:
  compute_dtype = torch.float16
  attn_implementation = 'sdpa'

model_name = "microsoft/Phi-3-medium-4k-instruct"
#Tokenizer
tokenizer = AutoTokenizer.from_pretrained(model_name, add_eos_token=True, use_fast=True)
tokenizer.pad_token = tokenizer.eos_token
tokenizer.pad_token_id =  tokenizer.eos_token_id
tokenizer.padding_side = 'left'

ds = load_dataset("timdettmers/openassistant-guanaco")

bnb_config = BitsAndBytesConfig(
        load_in_4bit=True,
        bnb_4bit_quant_type="nf4",
        bnb_4bit_compute_dtype=compute_dtype,
        bnb_4bit_use_double_quant=True,
)
model = AutoModelForCausalLM.from_pretrained(
          model_name, quantization_config=bnb_config, device_map={"": 0}, attn_implementation=attn_implementation
)

model = prepare_model_for_kbit_training(model)

#Configure the pad token in the model
model.config.pad_token_id = tokenizer.pad_token_id
model.config.use_cache = False # Gradient checkpointing is used by default but not compatible with caching

peft_config = LoraConfig(
        lora_alpha=16,
        lora_dropout=0.05,
        r=16,
        bias="none",
        task_type="CAUSAL_LM",
        target_modules= ['k_proj', 'q_proj', 'v_proj', 'o_proj', "gate_proj", "down_proj", "up_proj"]
)

training_arguments = SFTConfig(
        output_dir="./Phi-3-medium_QLoRA",
        evaluation_strategy="steps",
        do_eval=True,
        optim="paged_adamw_8bit",
        per_device_train_batch_size=8,
        gradient_accumulation_steps=4,
        per_device_eval_batch_size=8,
        log_level="debug",
        save_strategy="epoch",
        logging_steps=100,
        learning_rate=1e-4,
        fp16 = not torch.cuda.is_bf16_supported(),
        bf16 = torch.cuda.is_bf16_supported(),
        eval_steps=100,
        num_train_epochs=3,
        warmup_ratio=0.1,
        lr_scheduler_type="linear",
)

trainer = SFTTrainer(
        model=model,
        train_dataset=ds['train'],
        eval_dataset=ds['test'],
        peft_config=peft_config,
        dataset_text_field="text",
        max_seq_length=512,
        tokenizer=tokenizer,
        args=training_arguments,
)

trainer.train()

The hyperparameters are identical to the ones I used in my article fine-tuning Llama 3 8B. Despite Phi-3 medium having 6B parameters more than Llama 3 8B, fine-tuning Phi-3 medium with this configuration only consumed 15 GB of GPU RAM, i.e., 1.6 GB less than Llama 3 8B. The main reason is that the vocabulary of Phi-3 has nearly 100k fewer entries than Llama 3 8B (32k vs. 128k). The token embeddings are not quantized by QLoRA. Consequently, quantization is less effective for Llama 3 8B.

The learning curve of Phi-3 for this instruct fine-tuning is also very interesting when compared with the learning curve of Llama 3:

Phi-3 medium achieves a much lower validation loss than Llama 3 8B. I experimented a lot with this dataset (timdettmers/openassistant-guanaco), and you can find my results for other models in my articles, but, as far as I remember, I didn’t see any model achieving a validation loss below 1.1 on this dataset with QLoRA fine-tuning. Moreover, the training loss doesn’t decrease significantly after the first 200 steps while Llama 3 8B’s training loss almost linearly decreases.

My assumption to explain these differences is that Phi-3 medium has seen this dataset before, i.e., during pre-training. Of course, I might be wrong, but I don’t have any other explanation.

The remainder of the notebook shows how to merge the adapter in the base model. The code is the same as for Llama 3. However, if you want to quantize the resulting model, remember that bitsandbytes will perform poorly. Since AWQ doesn’t support Phi-3(as of June 3rd, 2024), I recommend to quantize it with GPTQ.

Conclusion

Phi-3 medium is a 14B LLM using the Llama architecture. As such, it is very easy to fine-tune and quantize the model.

Due to a much smaller vocabulary than Llama 3 8B, Phi-3 medium can be fine-tuned with a smaller GPU. Inference also consumes less memory.

The performance of Phi-3 medium on public benchmarks is very curious. Moreover, there is a high probability that it has been pre-trained on datasets’ validation and test splits available on the Hugging Face Hub. We will never know. Nonetheless, Phi-3 medium is a good LLM that should be a good alternative to Llama 3 8B despite being larger.
